# titanX 数据主链与业务 ID 总表

记录时间：2026-03-12
定位：为 titanX Manual-first Baseline 冻结业务数据主链，供主控架构与节点实现直接使用
范围：
- businessId allocator
- edge 驱动 upstreamResolver
- BaseNodeShell / runtime / result 分层
- writeback 时机与目标
- providerFamily 归属层
- manual-first 最小数据闭环
- TS 接口草案

---

## 1. 先给主控的直接结论

## 1.1 可直接做
- 按类型独立编号的 `businessId allocator`
- 基于 `nodes + edges` 的 `upstreamResolver`
- `BaseNodeShell / operator / compiled / runtime / result` 五层分离
- 执行后“先写节点，后落 runtime_records” 的双写回流程
- `providerFamily` 放在 compiled config 层，并把执行快照写进 runtime
- manual-first 的最小手动闭环：编辑 -> 编译 -> 执行 -> 写回 -> 下游继续引用

## 1.2 需要包装后做
- `upstreamSnapshot`：只作为本次执行快照保留，不能替代 edge 真相
- `runtime_records`：只做持久化与排障副本，不能做节点 UI 主真相
- `subtype`：只保留产品层语义，如 `digital-human` / `video-enhance`，不能直接替代 provider routing
- `primary result` 切换：需要 result 层统一承接，不能让各节点自己各存一份主图/主视频字段

## 1.3 不建议放进 v1
- 让下游节点长期持有上游最终结果副本作为持续真相
- 把 `providerFamily` 暴露成 UI 可编辑字段
- 维护一个脱离当前 nodes 图状态的 allocator 计数器
- 先讨论 Zustand/Redux 等大状态库，而不先冻结数据分层与写回规则
- 让 runtime_records 反向驱动 UI 主状态

## 1.4 推荐最小实现方案
- `allocateBusinessId(nodes, nodeType)`
- `resolveUpstream({ nodeId, nodes, edges })`
- `compileNodeExecutionPlan(node, upstream)`
- `dispatchExecution(compiled.providerFamily, compiled)`
- `applyExecutionWriteback(nodeId, execution)`
- `persistRuntimeRecord(kind, execution)`

## 1.5 风险点 / 踩坑点
- 旧 contract 里还残留 titanLX 路径，titanX 必须覆盖为自己的文档路径
- `status`、`runtime`、`result` 容易多处重复定义，必须单一职责
- 若跳过 compiled config，后续 provider family / prompt compiler / runtime record 都会散
- `upstreamResolver` 如果把 edge 真相变成节点缓存真相，断边、切主结果、撤销后会出脏数据
- `providerFamily` 如果放到 shell/UI 层，前端会退化成 provider 控制台

## 2. allocator 推荐算法

## 2.1 冻结规则
每个新建节点必须使用按类型独立编号业务 ID：
- `text-01`
- `image-01`
- `video-01`
- `audio-01`

删除后复用最小缺失编号。

## 2.2 推荐算法
采用**基于当前节点图状态的纯函数 allocator**：
- 输入：当前全部 `nodes` + 目标 `nodeType`
- 输出：新的 `businessId`
- 不依赖全局自增计数器
- 撤销/重做后天然一致
- 模板实例化、复制、上传建节点、输出点建节点都可复用

### 算法步骤
1. 过滤出同类型节点
2. 提取 `businessId` 中的数字部分
3. 升序排序
4. 从 `1` 开始找最小缺失编号
5. 返回 `prefix-${n.padStart(2, '0')}`

### 推荐前缀映射
| nodeType | prefix |
| --- | --- |
| `text` | `text` |
| `image` | `image` |
| `video` | `video` |
| `audio` | `audio` |
| `analysis` | `analysis` |
| `result` | `result` |
| `storyboard` | `storyboard` |
| `url` | `url` |

### TS 草案
```ts
export type TitanNodeType =
  | 'text'
  | 'image'
  | 'video'
  | 'audio'
  | 'analysis'
  | 'result'
  | 'storyboard'
  | 'url'

export function allocateBusinessId(
  nodes: Array<{ nodeType: TitanNodeType; businessId: string }>,
  nodeType: TitanNodeType,
): string {
  const prefix = nodeType
  const used = nodes
    .filter((n) => n.nodeType === nodeType)
    .map((n) => Number(String(n.businessId).replace(`${prefix}-`, '')))
    .filter((n) => Number.isFinite(n) && n > 0)
    .sort((a, b) => a - b)

  let next = 1
  for (const id of used) {
    if (id === next) next += 1
    else if (id > next) break
  }

  return `${prefix}-${String(next).padStart(2, '0')}`
}
```

## 3. upstreamResolver 推荐流程

## 3.1 基本原则
- edge 才是上游关系真相
- 节点 data 中允许有 `upstreamSnapshot`，但它只能是执行快照
- 执行前临时解析，执行后按需记录，不长期把上游值复制为持续真相

## 3.2 推荐流程
1. 读取当前节点 `nodeId`
2. 从 `edges` 中找所有 `target = nodeId` 的入边
3. 按 source 节点类型提取可消费内容：
   - text 节点：`content` / `result.text`
   - image 节点：当前主图 URL + prompt summary
   - video 节点：当前主视频 URL + cover + analysis summary
   - audio 节点：当前主音频 URL + transcript/summary（若有）
4. 形成统一 `resolvedUpstream`
5. 根据当前节点类型做白名单过滤和字段映射
6. 交给 `compileNodeExecutionPlan()`

## 3.3 推荐 resolver 输出
```ts
export interface ResolvedUpstream {
  sourceNodeIds: string[]
  texts: string[]
  imageUrls: string[]
  videoUrls: string[]
  audioUrls: string[]
  resultRefs: Array<{
    sourceNodeId: string
    sourceBusinessId: string
    sourceNodeType: TitanNodeType
    resultId?: string
    role: 'text' | 'image' | 'video' | 'audio'
    url?: string
    text?: string
    isPrimary: boolean
  }>
}
```

## 3.4 节点侧映射建议
| current nodeType | consume from resolvedUpstream |
| --- | --- |
| `text` | `texts[]`, `imageUrls[]`, `videoUrls[]`, `audioUrls[]` |
| `image` | `texts[]`, `imageUrls[]` |
| `video` | `texts[]`, `imageUrls[]`, `videoUrls[]`, `audioUrls[]` |
| `audio` | `texts[]` |

## 3.5 为什么必须 edge 驱动
因为 titanX 当前产品主链要求：
- 主结果可切换
- edge 可删除
- 结果可写回
- 下游默认跟随当前主结果

如果不用 edge 真相而改用节点本地缓存真相，会在这些场景下失真。

## 4. BaseNodeShell / runtime / result 的分层建议

## 4.1 推荐总分层
建议把节点数据固定拆成 5 层：
1. `BaseNodeShell`：身份与画布态
2. `operator`：用户可编辑输入
3. `compiled`：执行前标准化配置
4. `runtime`：执行过程态
5. `result`：结果态与主次选择

## 4.2 推荐职责

### BaseNodeShell
负责：
- 节点身份
- 业务 ID
- 节点类型
- 画布选择/折叠/位置等 UI 共性

### operator
负责：
- 用户当前可编辑输入
- prompt / model / ratio / duration / voice 等 operator-facing 字段
- 不包含 provider raw payload

### compiled
负责：
- 从 operator + upstream + registry 编译出的标准执行计划
- `providerFamily`
- `providerModel`
- `adapterKey`
- `compiledPrompt`
- provider-ready params

### runtime
负责：
- 当前执行状态
- taskId / state / failCode / failMsg
- queuedAt / startedAt / finishedAt
- writeback 时机追踪

### result
负责：
- 多结果列表
- 当前主结果
- 预览摘要
- 结果来源快照

## 4.3 推荐接口草案

## 5. writeback 时机与目标

## 5.1 推荐写回时机

### T0: 提交前编译完成
写：
- `compiled`
- `runtime.submitStatus = 'pending'`

### T1: 提交成功
写：
- `runtime.taskId`
- `runtime.providerState = 'waiting' | 'queuing'`
- `runtime.queuedAt`
- `canvasStatus = 'running'`

### T2: 轮询中
写：
- `runtime.providerState`
- `runtime.rawResultJson?`
- `runtime.failCode/failMsg?`

### T3: 成功返回
写：
- `result.results[]`
- `result.primaryResultId`
- `runtime.finishedAt`
- `canvasStatus = 'success'`

### T4: 持久化副本
写：
- `runtime_records/{kind}/...json`
- `runtime.lastRecordPath`

## 5.2 推荐写回目标

### 第一写回目标：当前节点 self
这是 UI 主真相。
必须写回：
- `canvasStatus`
- `compiled`
- `runtime`
- `result`

### 第二写回目标：runtime_records
这是持久化与排障副本。
至少保存：
- `nodeId`
- `businessId`
- `taskId`
- `providerFamily`
- `providerModel`
- `status`
- `resultUrls[]`
- `createTime`

### 第三写回目标：下游可见性
不直接写下游节点结果值。
只通过 edge + 上游当前主结果让下游在下次执行时解析。

## 5.3 为什么不能只写 runtime_records
- 节点要立即更新预览与主结果
- manual-first 场景下 UI 交互不能依赖文件回读
- runtime_records 主要服务排障、恢复、离线索引，而不是前端主状态

## 6. providerFamily 推荐归属层

## 6.1 推荐结论
`providerFamily` 的主归属层应是：**compiled-config-level**。

## 6.2 原因
- 它不是用户直接编辑字段
- 它比 runtime 更早决定 adapter 路由
- 它应由 `modelRegistry + operator.mode/subtype/model` 共同推导
- 它不应直接污染节点 UI

## 6.3 推荐规则

### shell 层
不放 `providerFamily` 作为用户编辑字段。

### operator 层
只保留：
- `model`
- `mode`
- `subtype`
- `qualityMode` 等用户语义字段

### compiled 层
推导：
- `providerFamily`
- `providerModel`
- `adapterKey`

### runtime 层
保留执行快照：
- `executedProviderFamily`
- `executedProviderModel`

## 6.4 典型推导例子
| nodeType | operator fields | compiled providerFamily |
| --- | --- | --- |
| `image` | `model=nanobanana-pro` | `kie-jobs-image` |
| `image` | `model=flux-kontext-pro` | `flux-kontext` |
| `video` | `mode=text-to-video`, `model=wan/2-6-text-to-video` | `kie-jobs-video` |
| `video` | `model=veo3_fast` | `veo-family` |
| `text` | `model=gemini-2.5-pro` | `gemini-generate-content` |
| `audio` | `mode=tts`, `model=eleven-v3` | `elevenlabs-tts` |
| `audio` | `mode=music`, `model=music_v1` | `elevenlabs-music` |

## 7. 最小手动闭环数据流图

## 7.1 文字版流程图

```text
用户创建节点
  -> nodeFactory.createNode()
  -> allocateBusinessId()
  -> 写入 BaseNodeShell + operator 默认值

用户手动编辑节点
  -> 更新 operator state

用户点击执行
  -> resolveUpstream(nodeId, nodes, edges)
  -> compileNodeExecutionPlan(operator, upstream, modelRegistry)
  -> 写回 compiled + runtime.pending
  -> dispatchExecution(compiled.providerFamily)
  -> adapter 提交任务 / 轮询
  -> 写回 runtime
  -> 写回 result.primary + result.results[]
  -> 节点进入 success/error
  -> persistRuntimeRecord()

下游节点执行时
  -> 再次按 edge 解析上游当前主结果
```

## 7.2 最小闭环覆盖范围
这一版只要求覆盖：
- 手动创建节点
- 手动编辑输入
- 手动触发执行
- 真实 runtime 写回
- 结果 URL 持久化
- 下游继续解析主结果

不要求当前纳入：
- 自动整组执行
- agent 自动触发
- 自动批量跑全图
- 复杂重试策略

## 9. 推荐实施顺序

1. 先把 `allocateBusinessId()` 收口进 `nodeFactory`
2. 再把 `TitanNodeRecord = shell + operator + compiled + runtime + result` 固定下来
3. 接着实现 `resolveUpstream()`，只做 manual-first 必需分支
4. 再接 `compileNodeExecutionPlan()`，把 `providerFamily` 推导收口
5. 最后做 `applyExecutionWriteback()` + `persistRuntimeRecord()`

顺序不能反。
因为如果先写 adapter、不先冻结数据主链，后面所有节点 runtime 都会越做越散。

---

## 10. 本文冻结的最终主张

1. `businessId` 必须由 nodeFactory 统一分配，算法基于当前 nodes 纯函数求最小缺失编号
2. `upstreamResolver` 必须由 edge 真相驱动，节点内部只保留执行快照，不长期复制上游值当真相
3. 节点数据必须拆成 `shell / operator / compiled / runtime / result` 五层
4. writeback 必须先写节点 self，再落 runtime_records；持久化副本不能反向取代 UI 主状态
5. `providerFamily` 主归属层必须是 compiled config，而不是 shell/UI 层；runtime 只保留执行快照
6. manual-first 的最小闭环已经足够明确，不需要先把自动化、状态库大战、复杂 orchestrator 一起拉进 v1
