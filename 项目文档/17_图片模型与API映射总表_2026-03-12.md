# titanX 图片模型与 API 映射总表

记录时间：2026-03-12
状态：API 调研 round1 整合稿（先服务于 titanX Manual-first Baseline）

---

## 1. 总体结论

图片节点当前不适合继续只写成泛化 `prompt + ratio + size`。
API round1 调研后，至少可以先把图片模型分成两类：

### 1.1 KIE Jobs 通用图片家族
统一入口仍可优先按：
- `POST /api/v1/jobs/createTask`
- `GET /api/v1/jobs/recordInfo?taskId=...`

当前更适合接入 titanX v1 的原因：
- 与现有视频 jobs 家族的 runtime 形态相近
- 便于复用统一 taskId / recordInfo / runtime_records 结构
- 更适合作为 Image Node 第一版真实闭环主干

### 1.2 Flux Kontext 专用图片家族
已确认存在独立接口：
- `POST /api/v1/flux/kontext/generate`

当前结论：
- Flux Kontext 不是普通 `jobs/createTask` 的同构镜像
- 其输入字段、回调说明、编辑语义都更像独立 provider family
- titanX 不应把 Flux Kontext 和通用 jobs 图片模型硬塞进同一 payload builder 分支

## 2. Flux Kontext 专用家族

## 2.1 接口定位
接口：
- `POST /api/v1/flux/kontext/generate`

能力：
- 文生图
- 基于单张输入图的图片编辑

官方文档已确认的两种使用方式：
1. 文生图：传 `prompt + aspectRatio`
2. 图片编辑：传 `prompt + inputImage`，`aspectRatio` 可选

## 2.2 已确认输入字段
- `prompt`：必填
- `enableTranslation: boolean`
- `uploadCn: boolean`
- `inputImage: string`
- `aspectRatio: '21:9' | '16:9' | '4:3' | '1:1' | '3:4' | '9:16'`
- `outputFormat: 'jpeg' | 'png'`
- `promptUpsampling: boolean`
- `model: 'flux-kontext-pro' | 'flux-kontext-max'`
- `callBackUrl: string`
- `safetyTolerance: number`
- `watermark: string`

## 2.3 模型层结论
- `flux-kontext-pro`：标准款，适合默认主线路径
- `flux-kontext-max`：复杂场景/高质量增强款

对 titanX 的产品含义：
- Image Node 的“图片模型”至少要能表达 model family，而不只是单个 model id
- `flux-kontext-pro/max` 更像“编辑/生成通用图片模型家族”
- 其 `inputImage` 语义天然对应图片节点的参考图/编辑态，而不是视频节点的首尾帧语义

## 2.4 产品字段映射
- `图片模型` -> `model`
- `比例` -> `aspectRatio`
- `参考图/当前素材` -> `inputImage`
- `自动翻译` -> `enableTranslation`
- `输出格式` -> `outputFormat`
- `Prompt 增强` -> `promptUpsampling`
- `安全级别` -> `safetyTolerance`
- `水印` -> `watermark`

## 2.5 与 titanX 当前规则的关系
- 文档里已确认图片节点空态与素材态切换，这和 Flux Kontext 的 `prompt + inputImage` 模式天然兼容
- 但当前 titanX 第一阶段先做 `标注 / 裁剪 / 下载 / 全屏`，不建议同步把 Flux 全部编辑细节一次性接满
- 更稳妥的做法是：先把 Flux Kontext 当作独立 image provider family 接到统一 runtime 外壳下

## 3. 通用 jobs 图片家族

## 3.1 当前适合作为 titanX v1 主链
当前在用户现场语义里反复出现的图片模型口径包括：
- `全能图片模型 2`
- `全能图片模型 Pro`
- `全能图片模型`

再结合 KIE 当前公开图片市场页，round1 可先按以下家族理解：
- Google / Nano Banana 家族
- Google / Imagen 4 家族
- Flux 2 家族
- Seedream 4.5 家族
- Z-Image / 其他图像生成家族

当前不急着把所有 market 模型逐个写死在前端，而是先落一层 model registry 归类。

## 3.2 当前建议的 model family 分层
### A. 通用文本/参考图生成家族
适用：
- 文生图
- 图生图
- 产品图一致性变体
- 多角度/打光等后续高级入口

代表家族：
- `google/nano-banana`
- `google/nano-banana-pro`
- `google/nano-banana-edit`
- `google/imagen4`
- `google/imagen4-fast`
- `google/imagen4-ultra`
- `seedream4.5`

### B. 强编辑家族
适用：
- 精确编辑
- 保持主体不变的定向改图
- 产品参考图一致性任务

代表家族：
- `flux-2/pro-image-to-image`
- Flux Kontext 家族（独立 provider）

## 3.3 对产品口径的直接影响
`titanX` 里的“全能图片模型”不应继续直接等于某一个 API model id。
更合理的理解是：
- UI 层显示“全能图片模型 Pro”这类产品名
- registry 层再映射到真实 provider model
- payload builder 只吃标准化后的 compiled config

## 4. 图片节点第一版标准化 compiled config

```ts
interface CompiledImageConfig {
  providerFamily: 'kie-jobs-image' | 'flux-kontext'
  model: string
  prompt: string
  inputImageUrls?: string[]
  aspectRatio?: string
  resolution?: string
  outputFormat?: 'jpeg' | 'png'
  count?: number
  enableTranslation?: boolean
  promptUpsampling?: boolean
  safetyTolerance?: number
  watermark?: string
  cameraParams?: {
    cameraModel?: string
    lensModel?: string
    focalLength?: string
    aperture?: string
  }
}
```

关键点：
- `providerFamily` 必须显式存在
- `inputImageUrls` 是 titanX 统一字段，adapter 再映射成 `inputImage` 或其他 provider 字段
- `cameraParams` 目前仍属于 titanX 的隐式 prompt 注入层，不要假设官方每个图片模型都有独立 camera 字段

---

## 5. 图片节点 round1 最推荐的实现顺序

1. 先把 KIE jobs 图片家族收口成 titanX 的默认 provider family
2. 把 `全能图片模型 / Pro / 2` 改为 registry display label，不直接写死真实 model id
3. 在 `executionPayloadBuilder()` 内增加 `providerFamily='kie-jobs-image' | 'flux-kontext'`
4. 先让通用 jobs 图片家族真实跑通，再补 Flux Kontext 独立 adapter
5. `多角度 / 打光 / 扩图 / 重绘` 这 7 个顶部动作仍先保留入口，不要因为 API 已知就提前冲实现

---

## 6. 当前最重要的结论

1. 图片模型不止一套，至少要拆 `kie-jobs-image` 与 `flux-kontext` 两个 provider family
2. `全能图片模型 Pro` 更像产品名，不应直接充当 API model id
3. 图片节点的参考图/素材态，与 Flux Kontext 的 `inputImage` 编辑模式天然兼容
4. 相机参数当前更适合作为 titanX prompt compiler 注入项，而不是假设所有官方图片接口原生支持
5. round1 后，图片 API 映射已足够支撑 modelRegistry 第一版，不需要再停留在纯 UI 猜测
