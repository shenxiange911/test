## 2. 按文件拆的具体改造点

## 2.1 `app/src/App.jsx`

这是当前最核心的改造文件。

### 需要新增的 state
在 `FlowCanvas()` 顶层新增：

```ts
const [edges, setEdges] = useState(initialEdges)
const [pendingSource, setPendingSource] = useState(null)
const [creationMenu, setCreationMenu] = useState(null)
const laneRef = useRef(new Map())
```

其中建议语义：

```ts
pendingSource = {
  sourceNodeId: string,
  sourceHandleId?: string | null,
  mode: 'from-output-click' | 'from-output-drag',
}

creationMenu = {
  open: true,
  sourceNodeId: string,
  sourceHandleId?: string | null,
  mode: 'from-output-click' | 'from-output-drag',
  anchorClient: { x: number; y: number },
  anchorFlow: { x: number; y: number },
  allowedTypes: Array<string>,

```

### 需要替换的现有 state
- `App.jsx:38` 当前 `edges` 只有读，没有写，必须改成可写 state
- `App.jsx:40` 当前 `pendingNodeType` 只服务顶部工具条创建，不能继续拿来代表 creation menu

处理建议：
- 保留 `pendingNodeType` 只服务“顶部工具条 -> pane click 创建”
- 新增 `pendingSource + creationMenu` 专门服务“输出点 -> 菜单 -> 建下游节点”

### 需要新增的函数

#### A. `openCreationMenuFromOutput()`
职责：
- 接收 `sourceNodeId/sourceHandleId/clientX/clientY/mode`
- 在打开菜单那一刻同时生成 `anchorClient + anchorFlow`
- 写入 `pendingSource + creationMenu`

为什么放这里：
- `screenToFlowPosition()` 在这里已经可用
- overlay 菜单渲染也在这里

#### B. `closeCreationMenu()`
职责：
- 清空 `creationMenu`
- 清空 `pendingSource`

#### C. `createCanvasNode()`
职责：
- 接收 targetType 与最终 position
- 返回标准 node 对象
- 统一 type/data/position/draggable

#### D. `createCanvasEdge()`
职责：
- 接收 `source/sourceHandle/target`
- 返回标准 edge 对象

#### E. `handleCreationMenuSelect()`
职责：
- 菜单项确认后的唯一主入口
- 调 `resolveCreationPosition()` 算落点
- 调 `createCanvasNode()`
- 调 `createCanvasEdge()`
- 写入 `nodes/edges`
- 关闭菜单

#### F. `handleConnectStart()`
职责：
- 在用户从输出 handle 开始拖线时记录 source 信息

#### G. `handleConnectEnd()`
职责：
- 若拖到了 pane 空白且不是有效连接，则打开 creation menu
- 调 `screenToFlowPosition()` 生成 `anchorFlow`

### 需要调整的 ReactFlow props
当前 `ReactFlow` 只有：
- `nodes`
- `edges`
- `onNodesChange`
- `onNodeClick`
- `onPaneClick`

需要补：
- `nodeTypes`
- `onConnectStart`
- `onConnectEnd`

如果后续允许普通连线，也可补：
- `onConnect`

### 需要新增的 overlay 渲染位
最适合直接挂在：
- `App.jsx:72` 的 `.canvas-shell` 内部
- 与 `.canvas` 同级或后置同级

建议结构：
```tsx
<div className="canvas-shell">
  ...toolbar...
  <div className="canvas">
    <ReactFlow ... />
  </div>
  <CreationMenu ... />
</div>
```

这是本轮问题 1 的正式答案：
- 菜单渲染入口最适合先挂在 `app/src/App.jsx` 的 `FlowCanvas` 里
- 菜单点击触发入口则要挂到新建的 custom node 组件里

## 3.5 `app/src/flow/creationPosition.js`

### 目的
- 管理 `anchorFlow -> 最终新节点落点`

### 建议函数

#### `resolveCreationPosition({ anchorFlow, sourceNodeId, sourceHandleId, nodes, laneMap })`

## 4. 7 个必答问题的正式代码映射答案

## 4.1 现有代码里菜单入口最适合挂在哪个文件/组件
分成两个入口：

### 点击入口
最适合挂在新建的 `app/src/components/nodes/TitanNode.jsx`

原因：
- 输出点是节点内部交互
- 只有 custom node 才能准确表达“点 output，不是点节点本体”

### 渲染入口
最适合挂在 `app/src/App.jsx` 的 `FlowCanvas` 内 overlay 区

原因：
- `screenToFlowPosition()` 在这层可用
- `creationMenu` state 也最适合放这层
- 菜单不应跟 viewport 缩放

---

## 4.2 `pendingSource / creationMenu / anchorClient / anchorFlow` 各自适合放在哪层 state

### `pendingSource`
先放 `FlowCanvas` 顶层 state。

### `creationMenu`
先放 `FlowCanvas` 顶层 state。

### `anchorClient`
作为 `creationMenu` 的字段，不单独漂出去。

### `anchorFlow`
也作为 `creationMenu` 的字段。

### 附加说明
- 这些状态都不适合放进 node.data
- 也不适合放进 `Inspector`
- P0 阶段不必强上全局 store

---

## 4.3 `screenToFlowPosition()` 应该在哪个交互点调用
正式答案：只在“打开 creationMenu 的那一刻”调用。

### 两个具体调用点
#### A. `TitanNode` 输出点 click 事件回调里
流程：
- output click -> 回传 `clientX/clientY`
- `FlowCanvas` 里调用 `screenToFlowPosition()`
- 同时生成 `anchorClient + anchorFlow`

#### B. `onConnectEnd` 里
流程：
- 从输出 handle 拖线到 pane 空白
- `onConnectEnd` 判断无有效 target
- 立刻调用 `screenToFlowPosition()`
- 写入 `creationMenu`

### 不建议的调用点
- 菜单确认时
- node factory 内部

---

## 4.4 node 创建和 edge 创建分别由哪个函数负责

### node 创建
由 `createCanvasNode()` 负责。

P0 建议先定义在 `App.jsx`：
- 方便直接访问当前 `nodes`、`selected`、业务上下文

P1 再抽到：
- `app/src/flow/nodeFactory.js`

### edge 创建
由 `createCanvasEdge()` 负责。

P0 建议先定义在 `App.jsx`。

P1 再抽到：
- `app/src/flow/edgeFactory.js`

## 5. 最终执行建议

如果下一轮进入真正改代码阶段，最稳的落法不是直接猛改 `handlePaneClick()`，而是：

1. 先把 `App.jsx` 变成能管理 `pendingSource + creationMenu + setEdges` 的容器。
2. 再引入 `TitanNode.jsx`，把输出点点击和输出拖拽入口接出来。
3. 再用 `CreationMenu.jsx` 把“从 output 发起”的创建链闭起来。
4. 最后才补 `creationPosition.js` 的 lane / 避让策略。

这条顺序最适合当前 titanX/app 的空壳状态，也最符合前一份正式稿里已经冻结的产品语义。
