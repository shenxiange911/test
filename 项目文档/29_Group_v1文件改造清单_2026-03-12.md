## 1. App.jsx 需要承担哪些最小 flow shell 职责

`app/src/App.jsx` 在 P0 阶段必须承担的不是“全部业务”，而是“最小 flow shell”。

### 必须承担的职责
- 持有 `nodes`
- 持有 `edges`
- 持有 `selection`
- 持有 `toolbarMode`
- 持有 `activeInspectorTarget`
- 持有 `pendingNodeType`
- 接 `onNodesChange`
- 接 `onEdgesChange`
- 接 `onSelectionChange`
- 注册 `nodeTypes.group`
- 把 UI 触发转成命令调用：`groupSelection / ungroup / createTemplateFromGroup`
- 把当前 target 传给 `Inspector`

### 不应继续堆进 App.jsx 的内容
- 打组坐标换算
- 解组坐标换算
- 模板序列化细节
- GroupNode 内部视觉结构

### P0 最小新增 state 建议
```ts
const [nodes, setNodes] = useState(...)
const [edges, setEdges] = useState(...)
const [selection, setSelection] = useState({ nodes: [], edges: [] })
const [toolbarMode, setToolbarMode] = useState('default')
const [activeInspectorTarget, setActiveInspectorTarget] = useState({ type: 'node', nodeId: '1' })
const [pendingNodeType, setPendingNodeType] = useState(null)
const [lastTemplateDraft, setLastTemplateDraft] = useState(null)
```

其中 `lastTemplateDraft` 可选，但很适合作为 P0 的调试反馈。

## 2. GroupNode.jsx 需要暴露哪些 props / data 字段

## 文件

## React Flow 侧最小 props
P0 阶段至少消费：

```ts
{

  executable?: boolean
```

- 不自行计算 child 成员
- 不保存 selection / toolbar / inspector 状态

## 3. groupCommands.js 里至少需要哪些导出函数

## 文件
- `app/src/flow/groupCommands.js`

### 至少需要的导出函数

#### `groupSelection(params)`
用途：
- 读取当前 selection nodes
- 计算 group 范围
- 创建 group node
- 把 child 改写成 parent-relative
- 返回新 `nodes / edges / groupId`

#### `ungroup(params)`
用途：
- 找 group
- 找 child
- child 恢复绝对坐标
- 移除 parent relation / extent
- 删除 group node
- 返回新 `nodes / edges`

#### `collectGroupMembers(groupId, nodes, edges)`
用途：
- 统一收集 group child nodes
- 统一收集组内 edges
- 给解组和模板导出复用

#### `computeGroupBounds(nodes)`
用途：
- 基于选中节点计算 group 框
- 统一 padding 策略

#### `toRelativeNodePosition(node, groupBounds)`
用途：
- 打组时把画布绝对坐标转成 group 相对坐标

#### `toAbsoluteNodePosition(node, groupNode)`
用途：
- 解组时把 group 相对坐标转回画布绝对坐标

### 内部可继续拆的小函数
如果实现时发现太长，建议内部再拆：
- `buildGroupNode()`
- `getGroupChildNodes()`
- `getGroupInternalEdges()`
- `applyParentRelation()`

## 4. templateCommands.js 里至少需要哪些导出函数

## 文件
- `app/src/flow/templateCommands.js`

### 至少需要的导出函数

#### `createTemplateFromGroup(params)`
用途：
- 作为 App.jsx 的主入口
- 输入 `groupId / nodes / edges`
- 输出最小 template draft

#### `serializeGroupToTemplateDraft(groupNode, childNodes, childEdges)`
用途：
- 组装正式的模板结构对象

#### `stripRuntimeFields(node)`
用途：
- 去掉 React Flow 实例态字段
- 防止直接 dump 原 node

#### `buildRelativeNodePositions(childNodes)`
用途：
- 生成模板内的相对布局映射

### P0 可选但建议保留的函数
- `normalizeTemplateNode(node)`
- `normalizeTemplateEdge(edge)`
- `buildTemplateName(groupNode)`

这些函数即使第一版很简单，也应该有单独入口，后面不容易返工。

## 5. selection / toolbar / activeInspectorTarget 各自建议放哪层

### selection
建议放：`App.jsx` 顶层容器状态

### toolbar
建议放：`App.jsx` 顶层容器状态

更具体说，P0 不需要存完整 toolbar 对象，只需要：
- `toolbarMode: 'default' | 'multi-select' | 'group-selected'`

### activeInspectorTarget
建议放：`App.jsx` 顶层容器状态

推荐最小结构：
```ts
{
  type: 'node' | 'group' | 'selection' | 'none',
  nodeId?: string,
  groupId?: string
}
```

### 不建议放进哪层
以下都不建议承载这三类状态：
- React Flow node.data
- `GroupNode.jsx`
- `groupCommands.js`
- `templateCommands.js`

原因：
- 它们都是容器级 UI 状态，不是结构态，不是模板态

## 6. 哪些状态继续留在 React Flow nodes/edges，哪些不要塞进去

### 继续留在 React Flow nodes/edges 的状态
- node id
- edge id
- node type
- node position
- group style width / height
- child 的 parent relation
- child 的 `extent: 'parent'`
- group node.data 里的稳定业务字段：`groupId / title / color / layout / templateable / executable`

### 不要塞进 nodes/edges 的状态
- 当前 selection
- toolbarMode
- activeInspectorTarget
- 弹窗开关
- hover / focus 临时态
- 最近一次 template draft

### 也不要塞进 group node.data 的内容
- 当前模板导出结果
- 当前执行结果
- 当前运行错误列表
- 当前 Inspector 折叠展开状态
- 当前多选信息

结论很明确：
- 结构态留在 graph
- UI态留在 App.jsx
- 导出态留在 templateCommands

## 8. 推荐的文件创建顺序和落地顺序

### 第一步：先改 `app/src/App.jsx`
原因：
- 它必须先从 demo 升成 flow shell
- 没有它，后续文件没有接入口

### 第二步：新建 `app/src/flow/groupCommands.js`
原因：
- 先稳定结构层命令
- 避免 UI 先跑在结构前面

### 第三步：新建 `app/src/components/nodes/GroupNode.jsx`
原因：
- 当 group 结构能创建后，再补 visual shell 最合理

### 第四步：新建 `app/src/flow/templateCommands.js`
原因：
- Group 先跑通，再接 Template 导出最稳

### 第五步：改 `app/src/components/Inspector.jsx`
原因：
- 等 `activeInspectorTarget` 明确后再接三态

### 第六步：视需要再改 `app/src/components/StatusPanel.jsx`
原因：
- 它适合作为调试反馈位，但不是 P0 主链核心

## 9. 哪些点最容易因为 prototype 结构导致返工

### 9.1 把 App.jsx 当最终业务归宿
这是 prototype 最常见返工源。
如果把：
- 打组
- 解组
- 模板导出
- inspector 路由
都继续堆在 `App.jsx`，后面一定再拆一次。

### 9.2 把 GroupNode.jsx 写成“又会画 UI 又会改结构”
这会导致：
- UI 与命令耦合
- 后续换入口时逻辑难迁

### 9.3 把 selection / toolbar / inspector target 塞进 node.data
这是第二大返工源。
一旦这样做：
- 模板导出会带脏字段
- 节点复制会带 UI 污染
- Group / Template 边界会直接混掉

### 9.4 模板直接 dump React Flow node 对象
如果不通过 `templateCommands.js` 的净化层，后面模板结构一定被实例态污染。

### 9.5 parent relation 写法不统一
当前版本虽然兼容 `parentNode / parentId`，但如果项目里到处手写两种口径，后面升级版本时最容易出隐性兼容 bug。

建议：
- 在 `groupCommands.js` 内统一生成 parent relation 写法
- 不要把这个细节散落在多个组件

## 11. 最终冻结结论

Group v1 的文件级改造不该再停留在“加 Group 功能”这个层面，而应固定成下面这套最小骨架：
- `App.jsx` 负责 flow shell
- `GroupNode.jsx` 负责 group visual shell
- `groupCommands.js` 负责结构命令层
- `templateCommands.js` 负责导出命令层
- `Inspector.jsx` 负责 target 消费层

这套文件划分最符合 titanX/app 当前原型现实，也最不容易在 P0 后返工。
