# titanX React Flow 画布能力边界总表

记录时间：2026-03-12
定位：用于 titanX 画布层排期、交互分层、v1 边界冻结
状态：基于过程稿与 React Flow 官方/高可信资料完成的正式整合稿

---

## 1. 结论先行

React Flow **适合承载 titanX 的画布交互层**，但它提供的是“画布引擎 + 图结构交互基础能力”，不是现成的 titanX 产品交互。

更准确地说：
- **库没问题**：分组、选择、拖拽、缩放、平移、右键事件、坐标转换、节点工具条、viewport 事件，这些底层能力足够支撑 titanX。
- **缺的不是库，而是业务壳**：菜单系统、模式机、打组命令、下方编辑框、焦点拾取模式、多选工具条、撤消重做、节点创建工厂，这些都要 titanX 自己补。

因此 titanX v1 的正确策略不是“继续怀疑 React Flow 行不行”，而是：
- 冻结 React Flow 为画布底座
- 在其上补自己的 `flowStore + command layer + overlay layer + node factory`
- 把 v1 范围控制在“主链路可交付”，不要把白板类高级交互一起塞进第一阶段

---

## 2. React Flow 原生可直接承接的能力

| 能力 | 是否原生支持 | 证据 | titanX 结论 |
| --- | --- | --- | --- |
| group node / sub flow | 是 | `parentId` / `group` / `extent: 'parent'` | 可直接作为打组底层机制 |
| 节点右键菜单事件 | 是 | `onNodeContextMenu` | 事件层可直接用 |
| 画布右键菜单事件 | 是 | `onPaneContextMenu` | 事件层可直接用 |
| 点击空白事件 | 是 | `onPaneClick` | 可直接用于关闭菜单/编辑器 |
| 鼠标附近创建节点坐标换算 | 是 | `screenToFlowPosition` | 可直接用于双击空白创建、右键创建 |
| 节点屏幕定位换算 | 是 | `flowToScreenPosition` | 可直接用于编辑框/浮层定位 |
| 单节点工具条 | 是 | `NodeToolbar` | 可直接用于单节点顶部/侧边快捷条 |
| 选择事件 | 是 | `onSelectionChange` / `onSelectionStart` / `onSelectionEnd` | 可用于多选态驱动 |
| viewport 缩放/平移事件 | 是 | `onMove` / `onMoveStart` / `onMoveEnd` | 可用于浮层重定位或关闭 |
| 交互模式切换基础开关 | 是 | `panOnDrag` / `selectionOnDrag` / `elementsSelectable` / `nodesDraggable` 等 | 足够支持 titanX 模式切换 |
| 受控 nodes/edges | 是 | `nodes` / `edges` / `onNodesChange` / `onEdgesChange` | 可作为 titanX 统一状态入口 |

### 2.1 对 titanX 的直接意义

React Flow 已经能原生承接这些 titanX 画布层基础：
- 节点图结构
- 节点选中
- 画布空白点击
- 右键入口
- viewport 变化
- 分组的底层几何关系
- 坐标系转换

所以 titanX 当前画布问题的核心，不是“换库”，而是“业务层怎么组织”。

## 3. 需要 titanX 自己包装的能力

| 能力 | React Flow 是否直接给成品 | titanX 需要补什么 | 优先级 |
| --- | --- | --- | --- |
| 框选后工具条 | 否 | 监听 selection，计算多选 bounds，渲染自定义浮层 | P0 |
| 打组 / 解组命令 | 否 | 计算包围盒、创建 group node、重写 child `parentId`、解组恢复 | P0 |
| 右键菜单组件 | 否 | 菜单 UI、定位、防越界、关闭逻辑 | P0 |
| 鼠标附近创建节点菜单 | 否 | creation menu、节点工厂、类型过滤 | P0 |
| picking mode | 否 | 模式状态机、交互锁、拾取提示、退出逻辑 | P0 |
| 点击空白关闭编辑器 | 只给事件，不给状态壳 | 编辑器状态管理、关闭规则 | P0 |
| 节点局部展开编辑区 | 否 | 下方编辑框 overlay、节点-编辑器绑定、viewport 跟随 | P0 |
| viewport 下浮层稳定定位 | 否 | 自定义 overlay manager | P0 |
| undo / redo | 否 | 历史栈、命令层、快照/增量策略 | P1 |
| lasso / whiteboard 式选择 | 否（示例可参考） | 自定义 lasso 组件 | P1 |
| 组级工具条 | 否 | group selection overlay、组级 actions | P1 |

### 3.1 核心判断

React Flow 提供的是：
- `事件`
- `坐标`
- `viewport`
- `图结构`

而 titanX 需要自己补的是：
- `产品模式`
- `产品菜单`
- `产品命令`
- `产品状态层`
- `产品视觉浮层`

也就是说，**React Flow 是引擎，不是 titanX 的交互成品。**

## 4. 不建议放进 v1 的能力

| 能力 | 不建议原因 | 替代方案 |
| --- | --- | --- |
| 完整白板式 lasso 选择体系 | 实现成本高，且会和现有 selection / pane click / dragging 产生冲突 | v1 先用默认框选或 Shift 多选 |
| 复杂组内自动布局 | 会把“打组”问题升级成“布局引擎”问题 | v1 只做手动打组 + 整组移动 |
| 组内节点自动吸附 / 自动重排 | 容易破坏用户手动排布预期 | v1 不做自动重排 |
| selectionOnDrag 作为默认主交互 | 已知会影响 pane click / blur，且对编辑器关闭逻辑不友好 | v1 保守使用 Shift 框选或显式模式切换 |
| 全局 always-on picking mode | 会明显干扰正常拖拽、选中、编辑 | 只在焦点提取时短时进入 |
| 多层嵌套 group | 容易让解组、边层级、坐标处理复杂化 | v1 只允许一层 group |
| 复杂浮层动画体系 | 会把定位问题和动画问题绑定，增加抖动和维护成本 | v1 先用静态/轻动画浮层 |

### 4.1 v1 不做，不代表未来不能做

这些能力不是 React Flow 做不到，而是：
- 对 titanX 当前 Manual-first Baseline 不关键
- 会显著增加画布层不稳定性
- 会推迟主链路交付

v1 应优先保证：
- 节点能创建
- 节点能连接
- 节点能局部编辑
- 多选能打组
- 右键和空白交互稳定
- 编辑器关闭逻辑稳定

## 5. 画布层推荐事件流

以下事件流不是教程，而是 titanX 应冻结的画布总线。

### 5.1 空白双击创建节点

```text
用户双击 pane 空白
→ onPaneDoubleClick
→ screenToFlowPosition(clientX, clientY)
→ openCreationMenu({ source: 'pane', flowPosition, screenPosition })
→ 用户选择节点类型
→ nodeFactory.createNode(...)
→ flowStore.addNode(...)
→ 如有需要自动选中并打开编辑器
```

### 5.2 节点右键菜单

```text
用户右键 node
→ onNodeContextMenu(event, node)
→ preventDefault()
→ overlayStore.openContextMenu({ kind: 'node', nodeId, screenPosition })
→ 菜单点击 action
→ command layer 执行（duplicate / delete / group entry / create downstream / ...）
→ onPaneClick 或 outside click 关闭菜单
```

### 5.3 画布右键菜单

```text
用户右键 pane
→ onPaneContextMenu(event)
→ preventDefault()
→ overlayStore.openContextMenu({ kind: 'pane', flowPosition, screenPosition })
→ action: upload / add asset / add node / undo / redo / paste
```

### 5.4 框选后工具条

```text
用户框选/多选节点
→ onSelectionChange({ nodes, edges })
→ flowStore.updateSelection(...)
→ 若 selectedNodes.length >= 2
  → reactFlowInstance.getNodesBounds(selectedNodeIds)
  → flowToScreenPosition(bounds anchor)
  → overlayStore.openSelectionToolbar(...)
→ 执行 Group / Delete / Create Template / Run Group 等 action
```

### 5.5 打组 / 解组

```text
用户点击 Group
→ groupCommands.groupSelection(selectedNodeIds)
→ 计算 bounds
→ 创建 group node
→ 将 selected nodes 写入 parentId
→ 位置改为相对 group
→ 对 child 节点可选写 extent='parent'
→ 节点数组重排：parent 在前，children 在后
→ 选中 group

用户点击 Ungroup
→ groupCommands.ungroup(groupId)
→ 读取 child nodes
→ 将 child position 还原为绝对坐标
→ 移除 parentId / extent
→ 删除 group node
→ 恢复 child selection
```

### 5.6 Picking Mode

```text
用户在节点内点击“焦点提取/从其他节点提取”
→ canvasMode = 'picking'
→ overlayStore.showModeHint('点击其他节点以提取元素')
→ ReactFlow 进入受限交互：禁用正常编辑/拖拽/菜单
→ 用户点击目标节点
→ onNodeClick -> pickCommands.pickFromNode(targetNodeId)
→ 写回当前编辑节点的 reference/tags/focus data
→ 退出 picking mode
→ 恢复正常交互
```

### 5.7 点击空白关闭编辑器

```text
用户点击 pane 空白
→ onPaneClick
→ overlayStore.closeContextMenu()
→ overlayStore.closeSelectionToolbar()
→ if canvasMode === 'idle': editorStore.closeActiveEditor()
→ clear transient selection state if needed
```

### 5.8 节点局部展开编辑区

```text
用户点击节点缩略图/主内容区
→ onNodeClick(node)
→ editorStore.open(nodeId)
→ reactFlowInstance.getNode(nodeId)
→ flowToScreenPosition(node anchor)
→ overlayStore.positionEditor(nodeId, screenPositionBelowNode)
→ viewport move / zoom 时重新定位或降级关闭
```

## 6. 画布层推荐状态分层

titanX 当前最需要的不是再加更多组件，而是先把状态分层固定下来。

### 6.1 推荐状态树

```ts
interface FlowStore {
  nodes: RFNode[]
  edges: RFEdge[]
  viewport: { x: number; y: number; zoom: number }
  selection: {
    nodeIds: string[]
    edgeIds: string[]
    groupId?: string | null
  }
  canvasMode:
    | 'idle'
    | 'creating-from-pane'
    | 'creating-from-output'
    | 'picking'
    | 'grouping'
    | 'menu-open'
  pendingConnection?: {
    sourceNodeId: string
    sourceHandle?: string | null
    startScreen?: { x: number; y: number }
  } | null
}

interface OverlayStore {
  contextMenu?: {
    kind: 'pane' | 'node' | 'selection'
    screenX: number
    screenY: number
    nodeId?: string
  } | null
  creationMenu?: {
    source: 'pane' | 'output-handle'
    flowPosition: { x: number; y: number }
    screenPosition: { x: number; y: number }
    sourceNodeId?: string
  } | null
  selectionToolbar?: {
    selectedNodeIds: string[]
    screenPosition: { x: number; y: number }
    bounds: { x: number; y: number; width: number; height: number }
  } | null
  modeHint?: {
    kind: 'picking'
    text: string
  } | null
}

interface EditorStore {
  activeNodeId?: string | null
  activePanel: 'node-editor' | null
  anchorScreenPosition?: { x: number; y: number } | null
  isPinned?: boolean
}
```

### 6.2 状态分层原则

- `FlowStore` 只管图、选择、viewport、模式
- `OverlayStore` 只管菜单/工具条/提示层
- `EditorStore` 只管当前打开的下方编辑区
- 命令层不要直接散落在 node 组件里

### 6.3 为什么要这样拆

如果不拆：
- 节点组件自己开菜单
- 画布自己开菜单
- 编辑器自己记位置
- 右键和多选各有一套状态

最后一定会出现：
- pane click 不知道该关谁
- viewport move 不知道该更新谁
- picking mode 不知道该禁谁
- group selection 和 node selection 相互覆盖

所以 titanX v1 画布稳定性的关键不是“再写更多 UI”，而是**先冻结状态边界**。

## 7. 与 titanX 产品规则的逐条对照

### 7.1 group node
- **结论**：可直接做
- **库层能力**：`parentId`、`group`、`extent='parent'`
- **业务补壳**：打组命令、解组命令、节点顺序重排、组级菜单
- **v1 建议**：只做单层 group，不做嵌套 group

### 7.2 框选后工具条
- **结论**：需要包装后做
- **库层能力**：selection 事件、bounds 计算
- **业务补壳**：selection toolbar overlay
- **v1 建议**：先只给 `打组 / 删除 / 创建模板` 三个动作

### 7.3 右键菜单
- **结论**：可做，但菜单组件必须自建
- **库层能力**：`onNodeContextMenu`、`onPaneContextMenu`
- **业务补壳**：定位、防越界、关闭规则、action mapping
- **v1 建议**：节点菜单和 pane 菜单分开，不共用一套 actions

### 7.4 鼠标附近创建节点
- **结论**：可直接做
- **库层能力**：`screenToFlowPosition`
- **业务补壳**：creation menu、nodeFactory
- **v1 建议**：优先支持双击空白、输出口拖拽失败后创建、右键 add node

### 7.5 picking mode
- **结论**：必须自己包装
- **库层能力**：事件拦截和交互开关足够
- **业务补壳**：canvasMode、模式提示、退出逻辑、拾取写回
- **v1 建议**：短时模式，不做全局 persistent picking mode

### 7.6 点击空白关闭编辑器
- **结论**：可直接做，但有已知坑
- **库层能力**：`onPaneClick`
- **业务补壳**：editorStore.closeActiveEditor()
- **v1 建议**：先避免把 `selectionOnDrag` 当默认主模式

### 7.7 节点局部展开编辑区
- **结论**：React Flow 能承载，但不是原生现成功能
- **库层能力**：节点点击、节点坐标、viewport 事件
- **业务补壳**：自定义 editor overlay
- **v1 建议**：编辑区始终是单实例，只允许一个节点打开

### 7.8 viewport 缩放/平移下的菜单与定位
- **结论**：能做，但 overlay manager 必须独立
- **库层能力**：`onMove`、坐标转换
- **业务补壳**：菜单/编辑器重定位或自动关闭策略
- **v1 建议**：
  - 右键菜单：viewport move 时关闭
  - 下方编辑框：viewport move 时重定位

## 8. 已确认的风险点 / 踩坑点

### 8.1 `selectionOnDrag` 与 pane click 冲突
- 已有公开 issue：启用 `selectionOnDrag=true` 且 `panOnDrag=false` 时，pane mouse 事件可能不再触发
- 影响：
  - 点击空白关闭编辑器可能失效
  - input blur 可能不触发
- **v1 处理建议**：
  - 默认不用 `selectionOnDrag` 做主模式
  - 先保守使用 Shift 框选或显式框选模式

### 8.2 group 节点不是完整的“嵌套 DOM 容器”
- `parentId` 只是相对定位和移动跟随
- 不是完整布局容器
- **影响**：不要误以为组内有天然的完整隔离语义
- **处理建议**：组只作为画布逻辑容器，不作为复杂 UI 容器

### 8.3 父节点顺序要求
- parent 必须在 children 之前
- **影响**：打组/解组/导入导出时必须统一排序
- **处理建议**：所有节点写入前统一走 `sortNodesForParentOrder()`

### 8.4 自定义浮层不会自动跟随 viewport
- 右键菜单、编辑框、多选工具条都是 titanX 自己画
- **影响**：缩放/平移后容易错位
- **处理建议**：建立统一 `overlay manager`

### 8.5 坐标系混淆
- 屏幕坐标和 flow 坐标混用会直接导致菜单/编辑器错位
- **处理建议**：
  - 节点创建只认 flow 坐标
  - 菜单显示只认 screen 坐标
  - 节点下方编辑器：节点 anchor 先 flow，再转 screen

### 8.6 NodeToolbar 适合单节点，不适合多选
- 多选工具条如果硬套 NodeToolbar，会变得别扭
- **处理建议**：多选工具条自绘

### 8.7 prototype 容易误把“选中”和“编辑打开”耦合
- titanX 已明确要求：选中高亮与编辑器展开解耦
- **处理建议**：`selection` 和 `activeEditorNodeId` 必须拆开存

## 9. 最小可实现方案（可直接排期）

### 9.1 v1 目标

只做能稳定支撑 titanX Manual-first Baseline 的画布层：
- 空白双击创建节点
- 节点选中
- 单节点右键菜单
- pane 右键菜单
- 单实例节点下方编辑区
- 点击空白关闭编辑器
- 多选后工具条
- 打组 / 解组
- 基础 viewport 缩放平移

### 9.2 最小模块拆分

#### A. `flowStore`
负责：
- nodes / edges
- selection
- canvasMode
- viewport
- pendingConnection

#### B. `nodeFactory`
负责：
- 业务 ID 分配
- 节点默认数据
- 创建位置
- 可选自动连边

#### C. `overlayManager`
负责：
- node context menu
- pane context menu
- selection toolbar
- node editor overlay
- picking hint

#### D. `groupCommands`
负责：
- `groupSelection()`
- `ungroup(groupId)`

#### E. `editorStore`
负责：
- 当前打开的节点编辑器
- 下方编辑区定位
- pane click 关闭

### 9.3 v1 推荐实现顺序

1. `flowStore` + `nodeFactory`
2. pane 双击创建 + pane 右键菜单
3. node 右键菜单
4. 单节点下方编辑区 + pane click 关闭
5. 多选工具条
6. 打组 / 解组
7. picking mode
8. 视口变化下的 overlay 统一更新

### 9.4 v1 降级方案

如果时间不够，按下面降级：
- 多选工具条只保留 `打组`
- 组级操作只保留 `解组`
- picking mode 先不做 hover 高亮，只做“点击节点写回后退出”
- 编辑器在 viewport move 时先关闭，不做实时跟随
- 暂不做 undo/redo

## 10. 对 titanX/app 当前 prototype 的直接判断

### 10.1 库没问题的部分

React Flow 作为 titanX 的画布底座是成立的：
- 节点拖拽、缩放、平移、多选、分组的底层能力够用
- 坐标系转换足够支撑“鼠标附近创建节点”和“下方编辑区定位”
- 事件层足够支撑右键、pane click、selection、viewport change
- 不存在“必须换库才能做 titanX 交互”的证据

### 10.2 当前 prototype 真正缺失的部分

缺的不是 React Flow，而是 titanX 自己的业务壳：
- 缺 `flowStore` 统一状态层
- 缺 `nodeFactory` 节点工厂
- 缺 `overlay manager` 浮层体系
- 缺 `canvasMode` 模式机
- 缺 `groupCommands` / `editorStore` / `command layer`
- 缺“选中 vs 编辑打开”解耦
- 缺“pane click 关闭编辑器”的统一关闭逻辑
- 缺“viewport 变化下 overlay 更新”的统一策略

### 10.3 对 prototype 的直接判断

**当前 titanX/app prototype 的主要问题不是 React Flow 做不到，而是：业务交互外壳还没有真正建立。**

也就是说：
- **库能力是够的**
- **原型缺的是交互架构，不是底层画布能力**
- 如果继续只在节点组件里零散加状态，prototype 会越来越乱
- 如果先补 `store + command + overlay + factory` 四层，React Flow 完全可以继续作为 titanX v1 的稳定底座

- 打组/解组命令
- 节点下方编辑区
- picking mode
- viewport 下 overlay 管理

### 11.3 不建议放进 v1
- 完整白板 lasso 体系
- 多层嵌套 group
- 自动布局/自动吸附/自动重排
- 以 `selectionOnDrag` 作为默认主交互
- 复杂动画式 overlay 系统

### 11.4 推荐最小实现方案
- 固定 React Flow 为画布引擎
- 先补 `flowStore + nodeFactory + overlayManager + editorStore + groupCommands`
- 先交付：创建、右键、编辑、关闭、多选、打组
- 先不交付：undo/redo、lasso、复杂 group 白板能力

### 11.5 风险点 / 踩坑点
- `selectionOnDrag` 与 pane click 冲突
- 浮层定位随 viewport 变化错位
- parent/child 节点顺序错误
- 把选中和编辑打开耦合导致后续逻辑爆炸
- 在 node 组件内部散落状态，导致画布层不可维护
