## 2. 每个文件的职责边界

## 2.1 `app/src/App.jsx`
P0 唯一总调度器。

职责：
- 管理 `nodes / edges` 主 state
- 管理 `pendingNodeType`
- 管理 `pendingSource / creationMenu`
- 调用 `screenToFlowPosition()`
- 接收 `TitanNode` 抛出的 output 事件
- 接收 `onConnectStart / onConnectEnd`
- 在菜单确认时统一创建 node + edge
- 渲染 `CreationMenu`

不负责：
- 节点内部输出点 DOM
- 菜单内部 UI 结构细节

## 2.2 `app/src/App.css`
职责：
- 画布外层 overlay 定位上下文
- `CreationMenu` 基础样式
- `TitanNode` 最低可用样式

不负责：
- 业务逻辑
- 位置换算

## 2.3 `app/src/components/nodes/TitanNode.jsx`
职责：
- 节点可视化壳
- 输出点 click 入口
- 输出 handle / source handle 展现

不负责：
- 创建菜单状态
- 节点创建
- edge 创建

## 2.4 `app/src/components/CreationMenu.jsx`
职责：
- 根据 `anchorClient` 在鼠标附近显示菜单
- 渲染 `allowedTypes`
- 把选中的类型通过 `onSelect(type)` 回调给上层
- 处理关闭

不负责：
- `screenToFlowPosition()`
- node 创建
- edge 创建

## 3. 文件之间的依赖顺序

## 3.1 依赖图

```text
App.jsx
  ├─ 依赖 TitanNode.jsx 作为输出点入口
  ├─ 依赖 CreationMenu.jsx 作为选型 UI
  └─ 临时内联 createCanvasNode / createCanvasEdge / resolveCreationPosition

App.css
  ├─ 为 App.jsx 的 overlay 提供样式
  ├─ 为 TitanNode.jsx 提供节点壳样式
  └─ 为 CreationMenu.jsx 提供菜单样式
```

## 3.2 实际落地顺序
- `App.jsx` 必须先具备新 state 与新回调能力
- `TitanNode.jsx` 才能有地方把事件抛上来
- `CreationMenu.jsx` 才能有状态可读、有回调可触发
- `App.css` 可跟随两个新组件一起补样式

## 4. 哪一步必须先于哪一步

## 4.1 必须先改 `edges` 为可写 state
原因：
- 自动建 edge 是主链核心
- 没有 `setEdges`，后面所有 output create 都没法闭环

顺序：
- 这一步必须在 `TitanNode.jsx` 接入前完成

## 4.2 必须先在 `App.jsx` 建立 `pendingSource / creationMenu`
原因：
- 没有这两组 state，菜单既不知道来源，也不知道显示位置

顺序：
- 必须在 `CreationMenu.jsx` 接入前完成

## 4.3 必须先定义 `openCreationMenuFromOutput()`
原因：
- `screenToFlowPosition()` 的正确调用时机就在这里
- 这一步决定 `anchorClient + anchorFlow` 是否在菜单打开时被冻结

顺序：
- 必须早于 `handleCreationMenuSelect()`
- 必须早于 `CreationMenu.jsx` 真接入

## 4.4 必须先有 `TitanNode.jsx`
原因：
- 没有 custom node，就没有真正的输出点入口
- 只能继续停留在 pane click demo

顺序：
- 必须早于“点击输出点弹菜单”验收

## 4.5 必须先有 `handleCreationMenuSelect()`
原因：
- 菜单不应直接创建 node/edge
- P0 的创建总调度器必须在 `App.jsx`

顺序：
- 必须早于 `CreationMenu.jsx` 的最终联调

## 4.6 必须最后再接 `onConnectStart / onConnectEnd`
原因：
- 拖线到空白属于点击输出点链路的并行分支
- 如果点击输出点主链还没稳定，先接拖线会让调试面变宽

顺序：
- 建议晚于点击输出点链闭环

---

## 5. 哪些内容先临时放 `App.jsx`，哪些值得立刻抽模块

## 5.1 先临时放 `App.jsx`
P0 建议先放在 `App.jsx` 的函数：
- `openCreationMenuFromOutput()`
- `closeCreationMenu()`
- `createCanvasNode()`
- `createCanvasEdge()`
- `resolveCreationPosition()` 的最简单版
- `handleCreationMenuSelect()`
- `handleConnectStart()`
- `handleConnectEnd()`

原因：
- 当前 app 太薄
- 这些函数都紧贴 `nodes / edges / screenToFlowPosition()`
- 先保证主链跑通，比先追求模块化更重要

## 5.2 值得立刻抽模块
P0 就值得立刻抽出去的只有：
- `app/src/components/nodes/TitanNode.jsx`
- `app/src/components/CreationMenu.jsx`

原因：
- 一个是交互入口壳
- 一个是菜单 UI 壳
- 它们天然不该继续塞在 `App.jsx` JSX 里

## 5.3 暂不建议立刻抽的模块
- `nodeFactory.js`
- `edgeFactory.js`
- `creationPosition.js`

原因：
- 这 3 个模块更多是结构优化
- 不是 P0 第一阻塞点
- 可以等主链打通后再抽，避免过早抽象

## 6. `TitanNode.jsx / CreationMenu.jsx / App.jsx` 三者之间的事件流

## 6.1 点击输出点链

```text
TitanNode.jsx
  -> data.onOutputClick({ nodeId, handleId, clientX, clientY })
  -> App.jsx.openCreationMenuFromOutput(...)
  -> App.jsx.setPendingSource(...)
  -> App.jsx.setCreationMenu({ anchorClient, anchorFlow, allowedTypes, mode })
  -> CreationMenu.jsx 渲染
  -> 用户选择目标类型
  -> CreationMenu.jsx.onSelect(type)
  -> App.jsx.handleCreationMenuSelect(type)
  -> App.jsx.createCanvasNode(...)
  -> App.jsx.createCanvasEdge(...)
  -> App.jsx.closeCreationMenu()
```

## 6.2 拖出输出点链

```text
TitanNode source handle
  -> ReactFlow onConnectStart
  -> App.jsx.handleConnectStart(...)
  -> setPendingSource(...)

ReactFlow onConnectEnd
  -> App.jsx.handleConnectEnd(event, connectionState)
  -> 若落空白且无有效 target
  -> App.jsx.openCreationMenuFromOutput(... mode='from-output-drag')
  -> CreationMenu.jsx
  -> 用户选择目标类型
  -> App.jsx.handleCreationMenuSelect(type)
  -> create node + edge
```

## 6.3 三者边界冻结
- `TitanNode.jsx`：只抛事件，不做创建
- `CreationMenu.jsx`：只选类型，不做创建
- `App.jsx`：P0 唯一创建调度器

## 7. 哪些点会和状态机线、数据链线冲突，需要预留接口

## 7.1 状态机线冲突点
未来会碰到的状态机包括：
- `idle`
- `creating-from-output`
- `focus-pick`
- `grouping`

P0 现在先不完整实现状态机，但必须预留：

```ts
pendingSource.mode
creationMenu.mode
```

建议字段：
- `'from-output-click'`
- `'from-output-drag'`

必要性：
- 否则后面接 `canvasMode` 时会返工
- 也无法区分菜单是从 click 还是 drag 打开的

## 7.2 数据链线冲突点
后续真实数据链会要求：
- source handle 白名单
- target node family 白名单
- 边数据里保留来源模式
- target 初始化时可能写入 source 引用

P0 应预留接口：

### A. `allowedTypes`
由 `App.jsx` 注入到 `CreationMenu.jsx`

### B. `sourceMeta`
`createCanvasNode()` 支持：
```ts
sourceMeta?: {
  sourceNodeId,
  sourceHandleId,
}
```

### C. `edge.data`
`createCanvasEdge()` 支持：
```ts
data?: {
  creationMode?: 'from-output-click' | 'from-output-drag'
}
```

### D. `handleId`
`TitanNode.jsx` 输出点不要写死为单一无 id handle，至少预留 `handleId`

## 8. 完整 P0 完成后，最小可验证行为是什么

P0 完成后，至少要能验证下面 5 条：

## 8.1 点击输出点弹菜单
- 点击任一 titan node 输出点
- 菜单稳定出现在鼠标附近

## 8.2 选择类型后自动创建 node + edge
- 菜单选择目标类型
- 新节点出现在正确位置
- 自动创建从 source 到 target 的 edge

## 8.3 缩放/平移后落点仍正确
- 先 zoom / pan 画布
- 再从输出点发起创建
- 新节点仍落在正确的 flow 坐标附近

## 8.4 拖到空白区也能走同一条菜单链
- 从输出点拖线到空白 pane
- 弹出同一个 creation menu
- 选择类型后自动建 node + edge

## 8.5 旧链路不回归
- 顶部工具条 + pane click 创建仍然可用
- 右侧 Inspector / StatusPanel 不被破坏

只有这 5 条全部成立，P0 才算完成。

## 10. 最终冻结结论

P0 的正确顺序不是“先抽一堆模块”，而是：

1. 先把 `App.jsx` 变成真正的创建调度容器。
2. 再补 `TitanNode.jsx` 作为输出点入口壳。
3. 再补 `CreationMenu.jsx` 作为菜单显示壳。
4. 最后把拖线到空白这条分支接到同一条创建主链上。

这样排期的好处是：
- 每一步都有明确前置依赖
- 每一步都能单独验证
- 不会在 P0 过早陷入“模块抽得很漂亮，但主链没跑通”的假推进。
