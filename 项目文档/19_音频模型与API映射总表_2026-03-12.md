# titanX 音频模型与 API 映射总表

记录时间：2026-03-12
状态：API 调研 round1 整合稿

---

## 1. 总体结论

音频节点当前至少要拆成两条主运行时：
- TTS 语音合成家族
- Music 文生音乐家族

结合已确认的产品截图与官方 API 文档，round1 后可以先冻结成：
- TTS 主干先按 ElevenLabs V2 / V3 家族理解
- Music 主干先按 Eleven Music 家族理解
- `数字人视频` 仍不单列为独立音频 runtime，而是 Video Node 的特化入口

---

## 2. ElevenLabs TTS 家族

## 2.1 当前产品对齐结论
用户已确认音频节点 `文字转语音` 模式至少出现：
- `ElevenLabs V2`
- `ElevenLabs V3`

并带以下联动参数：
- 音色
- 语速
- 稳定性
- 相似度
- 风格强度

因此 titanX Audio Node 的 TTS 模式，已经足够按 `model-aware param panel` 来设计，而不是一张固定表单。

## 2.2 第一版产品字段映射
- `文字转语音` -> `mode='tts'`
- `模型` -> `model`
- `音色` -> `voiceId / voiceLabel`
- `语速` -> `speed`
- `稳定性` -> `stability`
- `相似度` -> `similarity`
- `风格强度` -> `styleStrength`

---

## 3. Eleven Music 家族

## 3.1 已确认官方能力
Eleven Music compose 文档 round1 已确认：
- 主入口为“从 prompt 或 composition plan 生成音乐”
- `prompt` 与 `composition_plan` 互斥
- `music_length_ms` 范围：`3000 - 600000`
- `model_id`: `music_v1`
- `force_instrumental: boolean`
- `respect_sections_durations: boolean`
- `seed`
- `output_format` 可配置

## 3.2 与 titanX 当前产品字段的对应关系
当前 titanX 音频节点 `音乐` 模式里已确认：
- 模型：`ElevenLabs Music`
- `纯音乐` 开关
- 时长：`Auto / 30s / 1m / 2m / 3m / 4m`

可以直接映射成：
- `音乐` -> `mode='music'`
- `ElevenLabs Music` -> `model='music_v1'` 或 registry display label -> model id
- `纯音乐` -> `force_instrumental`
- `30s / 1m / 2m / 3m / 4m` -> `music_length_ms`
- `Auto` -> 不传 `music_length_ms`

## 3.3 当前工程结论
- titanX 的 `音乐` 模式先不需要实现 `composition_plan`
- 第一阶段只做 `prompt + optional length + instrumental`
- `composition_plan` 可作为 P1 高级模式预留

## 4. Audio Node 第一版 compiled config

```ts
interface CompiledAudioConfig {
  providerFamily: 'elevenlabs-tts' | 'elevenlabs-music'
  mode: 'tts' | 'music'
  model: string
  prompt?: string
  uploadedAudioUrl?: string
  playbackRate?: number
  outputFormat?: string
  ttsConfig?: {
    voiceId?: string
    voiceLabel?: string
    speed?: number
    stability?: number
    similarity?: number
    styleStrength?: number
  }
  musicConfig?: {
    forceInstrumental?: boolean
    musicLengthMs?: number
    seed?: number
  }
}
```

关键点：
- `playbackRate` 是播放器字段，不属于生成 payload 主参数
- 上传音频后的素材态与 `tts/music` 生成态要分开理解
- `uploadedAudioUrl` 当前主要服务于播放器、替换素材、下游视频/数字人，不等于已经进入 STT runtime

---

## 5. 数字人视频的归属

round1 后结论继续保持不变：
- `数字人视频` 不应在 Audio Node 内部落成第三套音频 runtime
- 更准确的做法是：Audio Node 右输出创建 Video Node
- 自动写入：
  - `subtype='digital-human'`
  - 默认视频模型（当前先按 Wan 2.6）
  - 当前音频素材或 TTS 结果 URL

这保证了：
- 音频仍是音频节点
- 说话视频仍是视频节点
- 运行时边界不会混乱

---

## 6. 当前暂不建议过早做的事

1. 不要把 TTS / Music / 数字人 三条线揉成一张静态参数面板
2. 不要把上传音频素材态误写成 STT 已完成
3. 不要因为 Eleven Music 支持 `composition_plan`，就第一阶段把音频节点做成复杂编曲器
4. 不要先定义过多 mode（`dialogue` / `sfx` / `audio-isolation`）再回头找 API 对应物

---

## 7. 当前最重要的结论

1. Audio Node 至少要拆成 `elevenlabs-tts` 与 `elevenlabs-music` 两个 provider family
2. `ElevenLabs Music` 当前已经足够映射到 `prompt + music_length_ms + force_instrumental`
3. 音频播放器态和音频生成态必须分层，不要混成一个 payload schema
4. `数字人视频` 继续归 Video Node，Audio Node 只负责把音频结果送过去
5. round1 后，音频节点的 API 主线已经能支撑 model registry 和 payload builder 第一版
