# titanX 文本分析模型与 API 映射总表

记录时间：2026-03-12
状态：API 调研 round1 整合稿

---

## 1. 总体结论

Text Node 当前最准确的运行时定位，不是“普通富文本框 + Analyze 按钮”，而是：
- 统一走 Gemini `generateContent`
- 按输入媒介差异切换 inline data / Files API
- 分析结果默认写回当前 Text 节点正文区

这和前面已经确认的产品规则一致：
- Text 节点上游可接 Text / Image
- 但不会自动分析
- 必须由底部任务输入区显式触发分析
- 结果默认挂当前节点，而不是自动拆成独立分析节点

---

## 2. Gemini 多模态分析主链

## 2.1 核心能力入口
当前 round1 已确认，Gemini 文本分析主链统一可按：
- `models.generateContent`

来理解。

这意味着 titanX Text Node 不需要为图片分析、PDF 分析、文本问答各做一套完全不同的节点 runtime；
真正应该变的是输入 parts 组合方式。

## 2.2 输入组织原则
Gemini 请求的核心不是“表单字段”，而是 `contents[].parts[]`。
因此 titanX Text Node 更适合先编译成：
- 任务指令文本 part
- 上游文本摘要 part
- 当前正文/任务输入 part
- 图片 part / PDF part / 其他文件 part

再统一发给 Gemini。

---

## 3. 图片分析映射

## 3.1 已确认能力
Gemini 官方图片理解文档当前已明确：
- 可直接做 image understanding
- 小文件可 inline 传入
- 较大图片或需复用图片时，建议走 Files API
- 支持多图输入

## 3.2 产品映射
- 当前 Text Node 左输入接图片节点 -> 编译为 image part
- 当前 Text Node 上传图片附件 -> 编译为 image part
- 当前底部任务输入，例如“分析这张图的构图和灯光” -> 编译为 text part
- 当前节点正文区已有内容 -> 作为上下文 text part

## 3.3 titanX 实现建议
当图片输入存在时，compiled request 至少包括：
- `analysisPrompt`
- `content`
- `upstream image parts`
- `uploaded image parts`

当前不建议：
- 把图片 URL 只当普通文本塞进 prompt
- 让前端自己拼接一长段“请分析这张图：URL...”替代真实多模态输入

---

## 4. PDF / 文档分析映射

## 4.1 已确认能力
Gemini 官方文档当前已明确：
- 可原生处理 PDF
- 最长可理解到 1000 页文档
- 可总结、问答、提结构化输出
- 小文档可 inline 传 `application/pdf`
- 大文档或多轮复用建议走 Files API

## 4.2 对 titanX 的直接影响
这直接说明：
- Text Node 上传 PDF 是合理主线，不需要再单独造一个“PDF 专属节点”才能分析
- 后续如果要做“合同分析 / 论文总结 / 报表抽取”，都可以先复用当前 Text Node
- `pdf` 在 titanX runtime 里更像附件类型，而不是必须分裂成新的节点族

## 4.3 产品字段映射
- 上传 PDF -> `mimeType='application/pdf'` part
- 分析任务 -> `analysisPrompt`
- 输出到正文区 -> `content`
- 结构化摘要 -> `result.structured`

## 5. Text Node 第一版 compiled request 结构

```ts
interface CompiledTextAnalyzeRequest {
  providerFamily: 'gemini-generate-content'
  model: string
  systemInstruction?: string
  textParts: string[]
  imageParts?: Array<{
    source: 'inline' | 'file-api'
    mimeType: string
    data?: string
    fileUri?: string
  }>
  pdfParts?: Array<{
    source: 'inline' | 'file-api'
    mimeType: 'application/pdf'
    data?: string
    fileUri?: string
  }>
}
```

关键点：
- Text 节点真正统一的是 `providerFamily` 和 request compiler
- 多模态类型差异放在 `parts` 层，而不是拆多个 Analyze Node

---

## 6. Files API 与 inline data 的边界

当前 round1 可先冻结成下面这条工程规则：
- 小附件、一次性分析：优先 inline
- 大附件、复用附件、多轮对话：优先 Files API

这条规则的价值在于：
- 避免一次性把 Files API 强耦合到所有场景
- 也避免后续大文件分析时继续硬塞 base64，导致 request 又大又慢

---

## 7. 与 titanX 当前产品规则的对齐结论

1. Text Node 的运行时主入口已经足够清晰：Gemini `generateContent`
2. Text / Image / PDF 分析不需要拆成三个节点 runtime，统一走 parts 编译
3. “附件分析”真正的技术核心是多模态 parts，不是多几个上传控件
4. Text 分析结果继续挂当前节点，是合理且和 Gemini 能力边界一致的
5. 后续要补的不是重新定义产品，而是把 `analysisPrompt + content + upstream + files` 真正接进 compiler

---

## 8. 当前最推荐的下一步

1. 在 `textAnalysisAction` 前增加 `compileTextAnalyzeRequest()`
2. 明确 `inline` 与 `file-api` 的切换阈值和适用场景
3. 先让图片分析和 PDF 分析都统一落到 `generateContent`
4. 再补 `result.structured` 的最小 schema，而不是继续停留在纯文本回写
