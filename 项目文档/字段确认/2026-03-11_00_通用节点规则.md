# titanX 字段确认 · 通用节点规则

记录时间：2026-03-11
状态：用户口述 + 截图确认中 / 持续补充

---

## 1. 适用范围
本文件记录 titanX 中所有节点共用的基础显示与行为规则。

当前适用节点至少包括：
- Text
- Image
- Video
- Audio

后续新增节点类型时，也应优先对照这里确认是否继承这些通用规则。

---

## 2. 当前已确认的通用规则

### 2.0 所有节点必须有稳定业务 ID，且按类型独立排序
你这次明确补充的核心规则是：

- 每个节点都必须有可排序、可复用的业务 ID
- 编号按创建顺序递增，例如：`01`、`02`、`03`、`04`
- 如果删除了中间编号，例如删除 `02`，那么下次新建时应优先复用 `02`
- 在复用了最小缺失编号后，后续再创建才继续往后递增，例如：当前已有 `01`、`03`、`04`，新建得到 `02`；再新建才得到 `05`

为了适配你后面可能做的自动化版本，这条规则必须是稳定规则，而不是仅前端显示规则。

推荐正式落地为：
- `text-01`、`text-02`、`text-03`
- `image-01`、`image-02`
- `video-01`
- `audio-01`

也就是：**按节点类型独立编号，而不是全局混排一个总序号。**

### 2.1 所有节点都有类型 ID / 类型标识
从截图可确认，每个节点顶部左上侧都会显示当前节点类型标识，例如：
- Image
- Video
- Audio
- Text

这说明：
- 所有节点默认具备可见的类型标识
- 用户在画布上可直接区分节点类型
- 该标识属于节点头部的基础展示信息

> 当前先记录为“类型 ID / 类型标识”。
> 如果后面你再补充出业务 ID（如 text-01 / image-01），我再把“类型标识”和“业务 ID”拆成两层。

### 2.2 节点默认可以是空态
从截图可确认，各类节点在未输入内容、未上传素材、未生成结果之前，可以先以空态节点存在于画布中。

表现包括：
- Image 节点显示图片占位符
- Video 节点显示视频占位符
- Audio 节点显示音频占位符
- Text 节点显示空文本/占位文案

这说明：
- titanX 支持“先创建空节点，再逐步填充内容”的工作方式
- 节点创建与节点内容填充是两个不同阶段

### 2.3 没有内容时默认不展开
你这次明确补充的规则是：

**如果节点没有上传内容，或者没有实际内容，默认不展开。**

这说明：
- 节点存在“默认紧凑态 / 未展开态”
- 是否展开，和节点是否已有内容/素材/结果强相关
- 空节点默认以简洁卡片形态停留在画布中，而不是自动进入完整编辑态

这是后面节点 UI schema 里必须统一继承的全局规则。

### 2.4 所有生成型节点都支持“多个乘数 / 多结果生成”
你这次补充的关键通用规则是：

- 当前所有生成型节点，生成按钮旁边都有“生成多少个”的乘数切换
- 这条规则不是图片节点特例，而是通用到：
  - 文本
  - 图片
  - 视频
  - 音频
- 生成后得到的多个结果，不是平铺死数据，而是支持主次切换

这说明后续所有生成型节点都应统一具备：
- `count / multiplier / generations` 切换入口
- 多结果容器
- 主结果与副结果的切换机制

也就是说：
- 图片节点的“主图 + 副图”不是孤立规则
- 视频、音频、文本后面也应按“主结果 + 次结果”去理解

## 3. 当前可以先锁定的产品判断

### 3.0 业务 ID 不是展示细节，而是自动化基础设施
因为你后面可能会做第二版自动化，所以节点 ID 不能只是前端显示号，而应视为：
- 稳定引用节点的主键之一
- 模板保存与加载时的定位锚点
- 自动化执行时的节点寻址基础
- 删除/插入节点后仍可维持规则化排序的基础

所以这条业务 ID 规则后面应被所有节点 schema、模板结构、自动化执行层共同继承。

### 3.1 节点生命周期至少分三段
根据目前信息，节点生命周期至少可以拆为：

1. **创建空节点**
   - 节点被放到画布上
   - 只有类型标识和空态占位
   - 默认不展开

2. **填充内容 / 上传素材 / 输入任务**
   - 节点开始具备真实内容
   - 可能进入展开态或可编辑态

3. **生成结果 / 进入可传递状态**
   - 节点有真实结果
   - 可继续作为下游输入

### 3.2 “展开态”不是默认态，而是条件触发态
这和很多 demo 式节点不同。当前 titanX 更像是：
- 默认先给你一个紧凑卡片
- 只有在有内容或进入编辑时才展开

所以后续所有节点文档都要注意区分：
- 默认态
- 空态
- 展开编辑态
- 结果态

### 3.3 多结果不是附属显示，而是统一结果模型
既然文本、图片、视频、音频都支持“生成多个”，那后续结果模型不能只按单结果设计。

统一建议按以下语义理解：
- 每个生成型节点都有 `primaryResult`
- 同时保留 `secondaryResults[]`
- 用户可在多个结果之间切换主次
- 下载、继续生成下游、创建资产等动作，默认作用于当前主结果

这条后面会影响：
- 节点结果 UI
- 输出口引用的是哪个结果
- 下载默认下载哪一个
- 快照/增强/解析等衍生动作是基于主结果还是指定结果

## 4. 暂定字段整理建议（草案）
```ts
interface CommonNodeRulesDraft {
  nodeTypeLabelVisible: boolean
  supportsEmptyState: boolean
  defaultExpanded: false
  expandCondition?: string[]
  idRule?: {
    perTypeSequence: boolean
    reuseSmallestMissingNumber: boolean
    formatExample?: string[]
  }
  generationMultiplicity?: {
    supportedNodeTypes?: string[]
    hasCountSwitcher: boolean
    supportsPrimarySecondarySwitching: boolean
  }
  lifecycleStages?: string[]
}
```

推荐先按以下语义理解：
- `nodeTypeLabelVisible = true`
- `supportsEmptyState = true`
- `defaultExpanded = false`
- `expandCondition = ['hasUploadedContent', 'hasActualContent', 'enterEditMode']`
- `idRule.perTypeSequence = true`
- `idRule.reuseSmallestMissingNumber = true`
- `idRule.formatExample = ['text-01', 'text-02', 'image-01']`
- `generationMultiplicity.supportedNodeTypes = ['text', 'image', 'video', 'audio']`
- `generationMultiplicity.hasCountSwitcher = true`
- `generationMultiplicity.supportsPrimarySecondarySwitching = true`

## 5. 本轮确认结论
当前先把这条锁成 titanX 的通用节点规则：

1. 所有节点都带有可见的类型标识
2. 所有节点都允许先以空态创建
3. 如果节点没有上传内容或没有实际内容，默认不展开
4. 节点的展开态应视为条件触发态，而不是默认态
5. 文本 / 图片 / 视频 / 音频这四类生成型节点都支持“生成多个”的乘数切换
6. 多结果必须支持主次切换，后续下游引用默认跟随当前主结果
7. 上传本地素材后，图片 / 视频 / 音频节点都可能从“可继续接收上游输入”切到“素材浏览 / 结果分发态”，此时左输入口可收起，仅保留右输出口
8. 文本输入区右上角的放大按钮是通用交互：放大后只是同一编辑区的放大版，不应改变下方按钮结构、积分位与生成按钮逻辑
9. 左侧添加节点面板底部的 `上传` 入口是通用创建入口，当前至少支持图片、视频、音频、文本/TXT，并应按文件类型自适应创建对应节点，而不是统一落成单一素材卡
