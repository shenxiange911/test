### 3.4 底部编辑区与焦点入口
你这次补图后，这块也能先锁定：
- 视频节点在下方独立展开编辑区里，存在模式切换
- 单独创建视频节点时，默认是 `文生视频`
- 当接入图片节点后，会自动切到 `图生视频`
- 接入图片后，该图片会加载到视频节点首帧/参考图区
- 左侧存在参考图/控制画板截图入口
- 顶部存在 `焦点编辑` 模式条
- 焦点编辑态提示为：点击其他节点以提取元素
- 焦点编辑右侧有明确 `退出` 按钮

这说明：
- 视频节点底部编辑区不是只有 prompt 和参数，还继承了与图片节点一致的“焦点提取 -> 参考图/标签写回”逻辑
- 焦点编辑是画布级 picking mode，不是视频节点内部的局部开关
- 视频节点本身存在明确的 `文生视频 / 图生视频` 模式切换，且切换条件与是否接入图片上游有关

### 3.5 底部编辑区参数条
当前已确认，视频节点底部编辑区至少包含：
- 焦点按钮
- 文本输入框

- 参数汇总按钮：比例 / 清晰度 / 生成时长 / 生成音频开关
- 生成数量切换：当前视频只有 `1x / 2x`
- 积分显示
- 生成按钮

- 比例：`16:9 / 4:3 / 1:1 / 3:4 / 9:16 / 21:9`
- 清晰度：`480p / 720p / 1080p`
- 生成时长：`5s / 10s`

## 4. 输入输出与模式切换规则

### 4.1 左侧输入口
当前已确认，视频节点左侧输入创建/接入菜单至少支持：
- 文本生成
- 图片生成
- 音频生成

从用户补充语义看，更准确理解是：
- 视频节点允许接入 `文本 / 图片 / 音频` 类上游节点
- 其中图片接入会直接影响视频节点模式与首帧载入
- 其中音频接入后，可把当前视频理解为“带音频驱动的视频生成态”，为后续数字人 / 对口型语义提供输入基础

### 4.2 右侧输出口
当前已确认，视频节点右侧输出创建菜单只支持：
- 文本生成
- 视频编辑

其中：
- `视频编辑` 本质上就是下一个视频节点
- 不支持从视频结果右侧直接继续派生图片生成或音频生成

### 4.3 文生视频 / 图生视频切换
当前已确认：
- 单独创建视频节点时，只支持 `文生视频`
- 只有接入图片节点后，才切到 `图生视频`
- 接入图片节点后，会把图片加载到视频节点首帧

所以这条要写死成模式规则：
- `text-to-video` 是默认空节点模式
- `image-to-video` 由上游图片接入触发
- 模式切换不是单纯 tab 手动切换，而是受上游输入结构驱动

### 4.4 多图输入规则：首帧 / 尾帧 / 风格图 / 多帧模型
你这轮补图后，这块已经能正式锁定：
- 视频多图输入是否可用，取决于当前视频模型是否支持
- 如果当前模型不支持多图输入，需要切换到支持的模型
- 当有 2 张图片节点接入时：
  - 第一个接入的视频参考图作为 `首帧`
  - 第二个接入的视频参考图作为 `尾帧`
- 在首尾帧模式下，还可以额外再加 1 张 `风格图`
- 也就是：`首帧 + 尾帧 + 风格图 = 3 张图片`
- 点击 `风格参考` 后，会打开资产选择面板，从“我的资产/公共资产”里选择后添加到风格位
- 当接入 3 张图片节点，并切到多帧模型时：
  - 当前进入的是 `多帧模型` 理解
  - 此时不再单独提供 `风格图片` 选择位

这说明视频节点的图片输入不是单一 `referenceImages[]` 就能表达清楚，而至少有几种语义位：
- `firstFrameImage`
- `lastFrameImage`
- `styleReferenceImage`
- `multiFrameImages[]`

## 5. 快照规则

### 4.1 当前已确认的行为
点击快照按钮后：
- 按当前视频播放位置截图，或按暂停所在帧截图
- 自动生成一个新的图片节点
- 该图片节点功能与普通图片节点一致

### 4.2 产品含义
这说明视频节点不仅是结果浏览器，还承担：
- 从视频中抽帧
- 把帧转入图片工作流
- 让视频结果继续衍生为图片编辑/图片生成链路

### 4.3 当前应写死的工程规则
- `snapshot` 是节点级命令，不是播放器临时功能
- 快照生成结果必须走统一 `createNode({ type: 'image' })`
- 新生成的图片节点应继承普通图片节点全部能力，而不是特殊轻量节点

## 6. 顶部工具栏逐项确认

### 5.1 第一个按钮：剪辑（beta）
当前已确认：
- 这是 beta 功能
- 交互形态类似 PR 的时间轴剪辑
- 但能力只收敛到当前视频节点素材本身
- 只能设置当前这个视频节点的 `入点` 与 `出点`
- 本质上是把当前素材裁切成一段
- 左边是取消
- 右边是确定
- 确定后裁切当前视频素材
- 如果剪得不满意，可以回到原节点后再次进入剪辑重做

这说明：
- 剪辑不是新建节点，而是修改当前素材时间范围
- 剪辑不是完整 NLE 编辑器，而是当前节点级的时间段裁切器
- 该模式至少需要临时编辑态与确认/取消两步交互

### 5.2 第二个按钮：解析
当前已确认：
- 主要是上传素材给 AI 做分镜头解析
- 可能使用 Gemini 3.0 PRO 以上模型
- 解析结果上方存在两种视图切换：
  - 详细分析列表
  - 创意图缩略图
- 当前产品里图片中的文字需要双击才能复制，用户认为交互不好
- 更理想的行为是：按序号可直接复制该镜头下的全部内容，包括秒数、景别等

#### 5.2.1 创意图缩略图视图
当前已确认缩略图模式每个条目至少包含：
- 序号
- 多少秒
- 场景序号
- 简单内容
- 景别
- 镜头视角
- 镜头运动

#### 5.2.2 详细分析列表视图
当前已确认详细列表每个镜头至少包含：
- 镜头序号
- 画面截图缩略图
- 画面形容
- 画面内容
- 开始点
- 结束点
- 时长（秒）
- 景别
- 摄像机角度分析
- 摄像机运动
- 焦距与景深
- 光线
- 音乐
- 人声

#### 5.2.3 解析结果当前真实落地形态
你这次最后补充后，这一点已经明确：
- `解析` 结果不是简单写回节点内部的一块固定区域
- 它会生成一个可拖拽的结果文本框/结果面板
- 这个结果框里同时包含图片与文字内容
- 结果框支持用户自己拖拽位置
- 结果框支持用户自己放大缩小
- 初始显示时内容可能不全，需要用户自己拉大后查看完整内容

这说明：
- 解析结果更接近画布上的独立结果框，而不是节点内部静态详情区
- 它至少具备：独立位置、独立尺寸、内容滚动或展开能力

#### 5.2.4 当前产品限制
用户已明确：
- 现在解析结果并不是自动拆成可单独操作的一组镜头节点
- 当前无法直接把每个镜头单独分解并继续链接下一个节点
- 当前文本复制体验很差：要双击进入编辑模式才能复制文本
- 当前没有“整镜头一键复制”按钮
- 图片部分也没有很好的一键复制/复用链路
- 如果后续不做“镜头拆分打组”，用户就得先拷贝镜头图和提示词，再继续生成

因此这条要先记为后续优化方向：
- 是否允许 `解析结果 -> 每个镜头图片节点 + 文本提示 + 自动打组`
- 是否为每个镜头结果项提供 `复制整条信息` / `复制提示词` / `复制图片` 按钮

#### 5.2.5 当前更合理的实现建议
按你现在的使用场景，我判断优先级应该这么排：

1. **P0：给每个镜头结果项加一键复制按钮**
   - `复制整条镜头信息`
   - `复制该镜头提示词`
   - `复制图片`
   - 这是最小改动、最快改善体验的方案

2. **P1：把解析结果转换成可拖拽的镜头卡片/图片节点并自动打组**
   - 每个镜头生成：图片节点 + 文本节点（或复合镜头节点）
   - 自动放进一个 storyboard group
   - 支持整组移动
   - 支持解组后单独拿某个镜头继续连下游

3. **不推荐只继续强化双击编辑复制**
   - 因为这条路还是太重
   - 用户真实目标不是“编辑文本”，而是“快速复用镜头结果”

### 5.3 第三个按钮：高清
当前已确认：
- 点击后会自动链接一个 `视频增强` 节点
- 该节点具有输出框，可以链接到下一个节点
- 下方存在增强设置

当前已确认的视频增强参数至少包括：

- 视频分辨率：1080P / 2K / 4K
- 帧数：自适应 / 30 / 60 / 90

这说明：
- `高清` 不是原地修改，而是“从当前视频节点派生一个视频增强节点”
- 该节点是视频处理分支节点，不是播放器设置面板

### 5.4 第四个按钮：替换

### 5.6 第六个按钮：下载

## 10. 当前可以先锁定的实现判断

### 8.1 视频节点不是单纯生成节点
它至少同时承担：
- 视频素材承载
- 播放器
- 快照抽帧
- AI 分镜解析入口
- 派生视频增强分支
- 替换/移除/下载/全屏查看

### 8.2 视频节点上传后是“结果浏览态 + 工作分发态”
因为上传后：
- 只保留输出口
- 支持快照生成图片节点
- 支持解析
- 支持增强派生节点

所以视频节点不是被动结果卡，而是后续创作分发器。

### 8.3 `解析` 和 `高清` 都不应只做弹窗
- `解析` 更像生成结构化镜头结果
- `高清` 更像创建新的处理节点

后续实现时不要把这两个功能都做成一次性临时 modal 而失去链路语义。

### 8.4 视频节点焦点逻辑与图片节点保持一致
你这轮补图已经把这一点明确了：
- 视频节点存在 `焦点编辑` 模式
- 焦点来源可以是其他节点内容
- 提取出的焦点会写成参考图与标签
- 这套逻辑与图片节点一致

因此后续产品与 schema 应统一按下面理解：
- `视频节点` 也有参考图槽位
- `视频节点` 也有标签 chips
- `焦点编辑` 进入的是统一的画布 picking mode
- 从其他节点提取到的焦点，不是只服务图片节点，而是可以服务视频生成

## 13. 本轮确认结论

**一个以视频素材为中心的复合工作节点**，上传后具备：
- 顶部 7 个视频操作按钮

- 底部播放器控制条
