# titanX 节点 UI 验证经验 · 三段式展开规则

记录时间：2026-03-11
来源：基于 `titanZ` 静态 UI 测试版反复校正后的确认结论
状态：已通过用户肉眼校正，可作为后续节点 UI 基线

---

## 1. 本轮最重要的结论

此前最容易做偏的一点是：
- 把“展开态”理解成一个整体大卡片里塞更多内容

用户明确确认后，正确结构应该是：

### 正确规则
- **闭合态**：只保留中间那块“浏览/显示区”
- **展开态**：必须是明确的 **上 / 中 / 下 三段结构**
  - 上：标题区
  - 中：显示区
  - 下：编辑区

也就是说：
- 默认态不是一张独立缩略卡去替换完整节点
- 默认态也不是“大卡片的简化版”
- 默认态本质上是 **展开态里的中间显示块单独存在**

---

## 2. 闭合态规则（Collapsed State）

### 2.1 视觉结构
闭合态时，节点应表现为：
- 顶部只有很薄的一行类型标识（如 `Text` / `Image`）
- 主体只显示中间浏览块 / 显示块
- 不显示标题编辑区
- 不显示底部编辑区

### 2.2 交互规则
- 点击节点后，节点从闭合态进入展开态
- 闭合态本身主要承担“浏览 / 预览 / 占位”作用
- 闭合态不是功能面板，不应该堆过多按钮

### 2.3 设计意图
闭合态的目的不是展示全部参数，而是：
- 保持画布清爽
- 方便浏览多个节点
- 让用户先看结果或内容摘要，再决定是否展开编辑

---

## 3. 展开态规则（Expanded State）

### 3.1 必须拆成三段
展开态必须是肉眼可分辨的三段，而不是视觉上仍像一整块：

1. **上段：标题区**
   - 节点标题
   - 顶部样式按钮 / 标题控制按钮
   - 标题相关操作

2. **中段：显示区**
   - 当前节点的结果展示
   - 文本内容预览 / 图片结果 / 视频结果 / 音频结果
   - 这是节点最核心的“内容显示块”

3. **下段：编辑区**
   - prompt 输入
   - 模型选择
   - 参数按钮
   - 提交按钮
   - 其他编辑动作

### 3.2 中段是展开与闭合的共用核心
最关键的产品理解：
- **中段显示区是节点的核心本体**
- 闭合态保留的就是它
- 展开态只是在它上面增加标题区，在它下面增加编辑区

这条后续写节点组件时必须作为硬规则。

---

## 4. 本轮验证中踩过的坑

### 错误做法 1：把闭合态做成独立缩略卡
问题：
- 看起来像“另一种卡片”
- 不是从同一个节点自然展开出来
- 和目标产品感受不一致

### 错误做法 2：展开态仍然是一整块大卡
问题：
- 用户看不出明确的上中下三段
- 标题区、显示区、编辑区混在一起
- 失去节点结构感

### 错误做法 3：过早做功能逻辑，而不是先还原 UI
问题：
- 容易把注意力放到 React Flow 架构、handle、交互逻辑
- 却把最核心的视觉结构做偏

### 本轮修正后的原则
- **先画准 UI 结构，再接功能**
- **先做静态数据和静态按钮，再做真实逻辑**
- **先确保闭合态 / 展开态正确，再扩展节点功能**

---

## 5. 后续节点统一继承的 UI 基线

后面 `Text / Image / Video / Audio` 节点，统一先按这套结构：

### 通用结构
- 顶部薄类型条
- 中间核心显示块（默认可见）
- 展开后补上标题区
- 展开后补上底部编辑区

### 通用状态
- `collapsed`：只显示中间核心显示块
- `expanded`：显示上中下三段

### 通用优先级
1. 先确认默认态长相对不对
2. 再确认展开态三段结构对不对
3. 再确认每段里的按钮和控件位置
4. 最后才接真实逻辑

---

## 5.1 本轮新增验证结论（地图视角 + 展开观感）

这轮继续测试后，又确认了两条很重要的 UI 经验：

### 5.1.1 三段是成立的，但必须能在画布里完整看见
如果画布视角太近，虽然技术上已经展开成上中下三段，但用户肉眼只会觉得：
- 节点被往下拉长了
- 底部编辑区把地图往下顶开了
- 看不到完整结构

所以测试节点展开态时，必须同时验证：
- 节点内部确实拆成三段
- 在当前画布视角下，用户能一眼完整看清三段

换句话说：
- **结构正确** 不等于 **观感正确**
- UI 验收必须看完整画布视角下的真实观感

### 5.1.2 底部编辑区不能只是“把节点往下撑高”
当前测试中，用户明确指出一个问题：
- 底部编辑区虽然独立了，但观感上像“节点整体被往下拉长”
- 而不是“从中间显示块下面自然展开出的独立面板”

所以后续优化重点要改成：
- 底部编辑区更像下方独立附属面板
- 减少整块节点被粗暴拉高的感觉
- 让上中下三段既分离，又视觉上属于同一节点

这条后续要继续靠样式和间距微调，而不是改结构方向。

## 6. 对 titanX 文档和实现的直接影响

### 文档层
后续所有节点文档都要明确拆出：
- 默认态
- 展开态
- 标题区
- 显示区
- 编辑区

不要再只写“节点由几个模块组成”，而要明确：
- 哪个模块默认可见
- 哪个模块只有展开才出现

### 实现层
后续组件实现建议：
- 中间显示块先做成稳定核心组件
- 标题区和编辑区按展开态挂载
- 不要把三段又重新包成一个看不出结构的大壳

---

## 7. 当前可直接作为硬规则的版本

### 节点 UI 三段式展开规则
- 闭合态 = 只显示中间显示区
- 展开态 = 上标题区 + 中显示区 + 下编辑区
- 中显示区是节点核心本体
- 默认不要把节点做成另一张缩略卡
- 默认不要把展开态做成视觉上仍然一整块的大卡
- 先还原静态 UI，再接真实功能

---

## 8. 给后面开发者的一句话

**如果用户一眼看过去还觉得这是“一整块卡”，那就说明这版 UI 还没真正做到三段式。**
