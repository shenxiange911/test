## 1. 节点名称
图片节点

## 2. 节点定位
图片节点是 titanX 工作流中的图像生成、图像承载、图像结果展示与下游传递节点。

它的职责不会只是“放一张图”，而更像一个复合执行节点，至少会同时涉及：
- 图像结果展示
- 参考图输入
- 图像任务描述
- 模型与生成参数选择
- 执行提交
- 向下游节点继续传递图片结果或图像语义

结合你前面已经确认的整体产品方向，图片节点大概率会是 titanX 中最典型的复合节点之一。

---

## 3. 当前先继承的通用规则
在你还没继续发图片节点细图前，图片节点先默认继承以下已确认通用规则：

1. **节点默认可空态创建**
2. **没有上传内容或没有实际结果时，默认不展开**
3. **节点有可见类型标识**
4. **节点应继承三段式展开规则**
   - 闭合态：只显示中间浏览 / 显示区
   - 展开态：上标题区 + 中显示区 + 下编辑区
5. **节点必须有稳定业务 ID**
   - 推荐格式：`image-01`、`image-02`
   - 删除后复用最小缺失编号

这意味着图片节点后续确认具体字段时，不能脱离：
- `2026-03-11_00_通用节点规则.md`
- `2026-03-11_09_节点UI验证经验_三段式展开规则.md`

## 4. 当前可以先锁定的整体判断

### 4.1 图片节点不是单纯素材容器，而是生成节点
按照 titanX 当前产品方向，图片节点更合理的定义不是“上传一张图片放在画布上”，而是：
- 可以承载输入参考图
- 可以承载图像结果
- 可以配置生成参数
- 可以触发图像生成
- 可以把输出继续传给别的节点

所以它应优先按“执行节点”理解，而不是“附件节点”理解。

### 4.2 图片节点大概率同时包含“显示层”和“控制层”
结合文本节点和复合节点实现方案，图片节点至少应拆成两大部分：
- **显示层**：预览图、结果图、占位图、缩略图
- **控制层**：prompt、参考图、模型、比例、尺寸、参数、执行按钮

也就是说，后续字段不能只保留一个 `imageUrl` 就算结束。

### 4.3 图片节点的上游输入可能不止一种
从工作流角度判断，图片节点可能会接收：
- 上游文本节点传来的图像描述
- 上游图片节点传来的参考图或风格图
- 用户手动上传的图片素材

因此后续字段设计应预留：
- 文本语义输入
- 图片素材输入
- 结果图片输出

### 4.4 图片节点很可能是参数最复杂的节点之一
按你之前对 titan 系产品的要求，图片节点后面很可能涉及：
- 模型
- 比例
- 尺寸
- 数量 / 倍率
- 参考图
- 风格图
- 镜头 / 视角 / 构图类参数
- 结果回写

所以图片节点不适合做成一个平铺大对象，而应该按层拆分字段。

## 5. 当前建议的结构理解
在你继续发截图前，图片节点可以先按三段结构理解。

### 5.1 闭合态（默认态）
闭合态应只显示中间浏览 / 结果区，例如：
- 空态占位图
- 已生成图片缩略图
- 已上传参考图缩略图
- 类型标识 `Image`

闭合态目的：
- 快速浏览画布中的图片节点
- 看结果而不是看参数
- 保持画布清爽

### 5.2 展开态
展开态应拆成三段：

#### 上段：标题区
可能包括：
- 节点标题
- 节点颜色 / 节点标记
- 快捷操作按钮
- 结果切换 / 视图切换入口

#### 中段：显示区
可能包括：
- 当前结果图
- 参考图缩略图
- 空态占位图
- 多图结果轮播 / 切换入口

#### 下段：编辑区
可能包括：
- 图片任务描述输入
- 模型选择
- 比例选择
- 尺寸选择
- 参数位（如数量、倍率）
- 上传参考图 / 风格图
- 提交生成按钮

## 6. 当前可先锁定的功能方向

### 6.1 核心职责
- 承载图片生成任务
- 展示图片生成结果
- 接收参考图或素材图
- 配置图片生成参数
- 向下游节点输出图片结果
- 可能接收上游文本描述作为 prompt 来源

### 6.2 节点组成（暂定）
- 顶部标题 / 操作区
- 中间图片显示区
- 底部参数 / 输入 / 执行区
- 左右连接 / 扩展入口

### 6.3 图层角色
图片节点在工作流里至少可能同时承担三种角色：
- **素材输入节点**：承接上传图片
- **生成执行节点**：发起图片生成
- **结果输出节点**：把结果继续传给下游

这说明后续 schema 不应把“输入图片”和“输出图片”混成一个字段。

## 7. 当前重点判断

### 7.1 图片节点至少要区分“输入图片”和“结果图片”
这点非常关键。

因为图片节点里可能同时出现：
- 用户上传的参考图
- 风格图
- 模型生成的结果图

如果后续只用一个 `imageUrl` 字段，会直接把语义搞乱。

至少应预留：
- `referenceImages`
- `styleImages`
- `resultImages`

### 7.2 图片节点的 prompt 不一定只是一段文本
图片节点的输入语义很可能来自多路合成：
- 用户手写任务描述
- 上游文本节点输出
- 上游图片节点的语义解析
- 参考图附带的结构约束

所以后续更合理的是：
- 保留原始输入
- 保留编译后的 prompt / payload snapshot
- 不要只保留最终一段字符串

### 7.3 图片节点很可能存在“参数显示层”和“模型执行层”分离
例如：
- UI 上显示的是“宽屏 / 方图 / 竖图”
- 真正执行层写入的是 ratio code 或 size code

所以后续字段设计要预留：
- 面向用户的展示值
- 面向执行器的标准化值

### 7.4 图片节点很可能需要多结果管理
图像生成往往不是单结果，因此图片节点后续很可能需要：
- 多张结果图
- 当前选中结果
- 历史结果
- 重新生成
- 基于结果继续扩展

这一点后面发图时要重点确认。
