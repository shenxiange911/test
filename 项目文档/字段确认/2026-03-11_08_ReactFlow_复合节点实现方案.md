# titanX · React Flow 复合节点实现方案

记录时间：2026-03-11
目的：把 titanX 的节点系统，明确映射到 React Flow 官方能力，避免后面节点都确认完了却不知道该怎么落代码。

---

## 1. 先说结论

titanX 后续最合理的 React Flow 实现，不是把“一个业务节点”拆成多个真实 React Flow 节点乱连，而是：

- **一个业务节点 = 一个 React Flow custom node**
- **一个 custom node 内部 = 多个业务子模块组合**
- **只有在子块本身也要独立拖拽、独立连线、独立选中时，才升级成真正的子 flow / parent-child nodes**

也就是：

### 正确主线
- Text Node / Image Node / Video Node / Audio Node
  - 每个都是一个真实 React Flow node
  - 每个 node 内部再拆成多个 React 组件模块

### 不建议主线
- 把“标题区”“模型区”“输入区”都做成独立 React Flow node
- 把一个文本节点内部的每个按钮、每个参数都做成单独 node
- 用 subflow 强行模拟一个普通业务节点内部结构

原因很简单：
- 你现在描述的这些模块，本质上是**一个节点内部的 UI / schema / prompt 编译模块**
- 不是多个需要在画布上被用户单独拖拽管理的真实节点

---

## 2. React Flow 官方能力，对应 titanX 应该怎么用

### 2.1 Custom Nodes
React Flow 官方最核心、最适合 titanX 的能力就是 `custom node`。

官方文档明确支持：
- 节点里塞表单输入
- 节点里塞交互元素
- 节点里塞自定义内容
- 节点里放多个 handles

这正好对应 titanX 的节点现实：
- 文本节点有顶部样式区
- 中间有内容区
- 底部有模型区
- 左右有扩展入口
- 上方可能有 toolbar

所以 titanX 每种节点都应该先定义成：

```ts
node.type = 'text' | 'image' | 'video' | 'audio'
```

然后在 `nodeTypes` 里挂自己的 React 组件：

```ts
const nodeTypes = {
  text: TextNode,
  image: ImageNode,
  video: VideoNode,
  audio: AudioNode,
}
```

---

### 2.2 Handles
React Flow 官方支持一个节点有多个 handle，而且每个 handle 都可以有独立 id。

这对 titanX 非常重要，因为你现在已经确认：
- 文本节点左侧 `+` 和右侧 `+` 的能力不一样
- 后面图片、视频、音频节点很可能也会出现不同方向、不同语义的连接点

所以不要只做一个默认 handle，而应该按语义设计 handle：

```ts
left-insert
right-create
input
output
style-ref
asset-ref
```

如果后面 edge 真要精确知道“这条线是从哪个口出来的”，就必须用：
- `sourceHandle`
- `targetHandle`

### 适合 titanX 的 handle 设计思路
- 左侧 handle：前置输入 / 插入来源
- 右侧 handle：下游创建 / 输出延伸
- 顶部 handle：可选，给引用素材、风格图、控制参数
- 底部 handle：可选，给执行结果或状态输出

注意：
- 你现在 UI 上看到的是 `+`，实现上不一定必须是原生小圆点 handle 的样子
- 可以用 React Flow 官方支持的 **custom handle** 做成自己的 `+` 按钮外观

---

### 2.3 Node Toolbar
React Flow 官方有 `NodeToolbar`。

这个能力很适合 titanX 的：
- 文本节点顶部富文本工具条
- 图片节点顶部快捷按钮
- 节点 hover / 选中后显示的一排操作按钮

所以不要把顶部那排按钮硬塞进节点内容流里，应该明确区分：

- **节点主体内容区**
- **节点浮动工具条**

推荐做法：
- 常驻且和内容强绑定的控件，可放在节点内部 header
- 选中后才显示的编辑动作，优先放 `NodeToolbar`

这样好处是：
- 节点默认态更干净
- 空节点默认不展开这条规则更容易成立
- 工具条逻辑和内容逻辑能分层

---

### 2.4 Group / ParentId / Sub Flow
React Flow 官方支持：
- group node
- parentId
- sub flow

但这块要非常谨慎。

### 什么时候该用
只有当你真的要实现下面这种结构时才该用：
- 一个父节点内部包多个可独立拖拽的小节点
- 子节点位置是相对父节点的
- 子节点要跟随父节点移动
- 用户能明确感知“这是一个节点组 / 子流程”

### 什么时候不该用
以下情况不要上来就用 sub flow：
- 文本节点内部有标题、内容、模型选择三个面板
- 图片节点内部有镜头、焦距、风格、上传风格图参数
- 只是想表达“这个节点内部有很多模块”

这些都更适合：
- 一个 custom node 内部模块化
- 而不是多个真实 child nodes

### 结论
对 titanX 来说：
- **普通业务节点：用 custom node 内部模块化**
- **真正的模板组 / 工作流片段 / 宏节点：才考虑 group / sub flow**

## 3. titanX 节点应该拆成哪几层

为了后面手动版、自动版都能共用，我建议所有节点统一拆成 5 层。

### 3.1 View Layer（展示层）
只负责 UI 展示，不直接决定模型执行。

包括：
- 节点颜色
- 节点标题显示
- 占位图/占位文案
- 折叠态 / 展开态
- 缩略图显示

### 3.2 Content Layer（内容层）
负责用户真正编辑和看到的业务内容。

文本节点例子：
- 标题
- 正文
- 富文本结构

图片节点例子：
- 参考图
- 风格图
- 图片描述

视频节点例子：
- 视频任务描述
- 镜头说明
- 节奏/时长说明

### 3.3 Semantic / Prompt Layer（语义层 / Prompt 编译层）
这是最关键的一层。

它负责把用户输入的内容、风格、参数，整理成结构化语义，再编译成真正给模型用的 prompt / payload。

比如：
- 文本节点里的标题、颜色、字号、强调关系
  - 以后可能会转成 prompt 权重或关键词
- 图片节点里的镜头、焦距、风格、参考图
  - 会转成图像模型参数与提示词片段

所以这里不能只存 UI 值，必须留：
- 原始用户输入
- 结构化语义字段
- 编译后的 prompt snapshot

### 3.4 Execution Layer（执行层）
负责模型和运行，不负责内容本身。

包括：
- model
- ratio
- size
- quality
- multiplier / count
- submit
- run status
- result metadata

这个层后面大概率是多个节点共享同一套执行壳子，只是字段不同。

### 3.5 Graph Layer（图层）
负责 React Flow 层面的编排信息。

包括：
- nodeId
- nodeType
- 业务 ID（text-01 / image-01）
- position
- handles
- connections
- collapsed / expanded
- selected
- zIndex

## 4. 文本节点应该怎么落 React Flow

### 4.1 不要写成一个平铺大对象
不要这样：

```ts
TextNodeData = {
  text: '',
  color: '',
  title: '',
  model: '',
  size: '',
  prompt: '',
  output: '',
}
```

这会让：
- UI 层字段
- 语义层字段
- 执行层字段
- 结果层字段

全部混在一起，后面一定炸。

### 4.2 推荐结构
```ts
interface TextNodeData {
  bizId: string
  nodeType: 'text'

  view: {
    collapsed: boolean
    colorToken: string
    showToolbar: boolean
  }

  content: {
    title?: string
    body?: string
    richTextJson?: unknown
  }

  semantic: {
    extractedKeywords?: string[]
    styleSignals?: string[]
    promptFragments?: string[]
    compiledPrompt?: string
  }

  execution: {
    model: string
    multiplier?: string
    status?: 'idle' | 'running' | 'success' | 'error'
  }

  io: {
    promptInput?: string
    outputText?: string
  }
}
```

### 4.3 UI 映射
文本节点的 UI 推荐这样分：
- 顶部：Toolbar / Header controls
- 中部：内容卡片
- 底部：任务输入 + 模型选择 + 提交
- 左右：自定义 handle / 扩展入口

也就是：

```tsx
<TextNode>
  <NodeToolbar />
  <TextHeader />
  <TextEditorPanel />
  <TextExecutionPanel />
  <LeftInsertHandle />
  <RightCreateHandle />
</TextNode>
```

## 5. 图片节点应该怎么落 React Flow

图片节点很可能比文本节点更能证明“复合节点”思路是对的。

因为图片节点天然就有：
- 参考图模块
- 风格图模块
- 镜头/焦距参数模块
- 比例/尺寸模块
- 模型选择模块
- 结果图显示模块

所以图片节点建议直接定义成：

```tsx
<ImageNode>
  <ImagePreviewModule />
  <ReferenceImageModule />
  <StyleReferenceModule />
  <CameraParamsModule />
  <PromptModule />
  <ExecutionModule />
  <HandlesModule />
</ImageNode>
```

### 关键点
这些模块都还是**一个业务节点内部模块**。

只有当你以后真要做：
- “风格图节点”在画布上可单独拖拽
- “镜头参数节点”也是独立节点
- “参考图节点”也可以自己连线到很多节点

那时候再拆成真正多个节点。

在当前阶段没必要这样搞，会把系统复杂度拉爆。

## 6. 空态 / 折叠态 / 展开态 应该怎么设计

你已经确认一条很重要的规则：
- 没有内容时默认不展开

这条在 React Flow 里非常适合做成 `view.collapsed` 状态，而不是靠 CSS 临时糊。

推荐统一规则：

```ts
view: {
  collapsed: boolean
  empty: boolean
  expandedByUser: boolean
}
```

### 推荐行为
- 新建空节点：`empty = true`, `collapsed = true`
- 上传内容后：允许展开，但不一定自动展开
- 用户主动编辑时：`expandedByUser = true`, `collapsed = false`
- 内容清空后：回到空态，可重新折叠

这样后面所有节点行为才统一。

## 7. 业务 ID 应该放在哪一层

你刚确认的规则：
- 每种节点独立编号
- 删除后复用最小缺失编号
- 例如 text-01, text-02, image-01

这条不应该只挂在 React Flow 的 `id` 上。

推荐双 ID：

### 7.1 React Flow id
给内部图系统用：
```ts
id: 'rf_x7k29a'
```

### 7.2 业务 ID
给产品和自动化用：
```ts
bizId: 'text-02'
```

### 为什么必须双 ID
因为：
- React Flow id 更适合内部稳定引用
- bizId 更适合用户可读、模板保存、自动化寻址
- 后面如果有复制、导入、模板实例化，双 ID 更安全

---

## 8. 后面自动化版怎么复用这套设计

你现在最对的一点，就是提前在手动节点里把“参数层 / 语义层 / 执行层”分开。

因为自动化版真正复用的不是 UI，而是：
- 节点 schema
- 业务 ID
- prompt 编译规则
- execution payload 规则

所以后面自动化版最好的模式是：

- 手动版里，用户在节点上填：
  - 文本内容
  - 风格参数
  - 模型
  - 图像参考
- 系统把这些保存成结构化节点数据
- 自动化版载入同样的数据结构
- 替换变量
- 直接走同一套 prompt 编译 + execution pipeline

所以：
- 自动化版复用的是 `schema + compiler + executor`
- 不是复用“用户在 화면上怎么点按钮”

## 9. 对 titanX 的明确落地建议

### 9.1 第一优先级：先统一 NodeData 结构
先别急着继续拼 UI，先统一每个节点的数据骨架至少有：
- `bizId`
- `view`
- `content`
- `semantic`
- `execution`
- `graph`

### 9.2 第二优先级：每种节点都按“复合节点”实现
- 一个节点 = 一个 custom node component
- 内部拆模块组件
- 不要把一个节点的内部模块拆成多个 React Flow nodes

### 9.3 第三优先级：统一 handles 语义
例如：
- `left-insert`
- `right-create`
- `input-main`
- `output-main`
- `style-ref`

后面 edge 才不会混乱。

### 9.4 第四优先级：统一 toolbar 策略
- 通用动作：删除、复制、颜色、折叠
- 节点特有动作：模型切换、风格参数、预览操作
- 能放 NodeToolbar 的尽量放 NodeToolbar

### 9.5 第五优先级：不要过早上 sub flow
除非未来明确出现“一个宏节点里包含多个可独立拖拽子节点”的需求，否则先别上 parentId / group。

---

## 10. 最终判断

### React Flow 官方到底支不支持你这种“复合节点”？
支持。

但官方更接近的写法是：
- **custom node + 内部多个 UI 子模块**

而不是：
- **多个真实 flow node 假装成一个节点**

### titanX 最合理的实现路线
- 每个业务节点先作为一个 custom node
- 节点内部做模块化分层
- 用 handles 解决连接语义
- 用 NodeToolbar 解决浮动工具条
- 用 group / sub flow 只处理真正的节点组，不处理普通节点内部结构

---

## 11. 给后面写代码的人一句硬规则

**不要把“一个节点内部有多个功能区”误判成“应该拆成多个 React Flow 节点”。**

先问四个问题：
1. 这个子块是否需要独立拖拽？
2. 这个子块是否需要独立连线？
3. 这个子块是否需要独立选中？
4. 这个子块是否需要独立保存位置？

只要四个问题大部分答案是否定的，那它就应该：
- 留在同一个 custom node 内部
- 作为节点内部模块实现

而不是升级成真实 node。
