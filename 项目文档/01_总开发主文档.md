# titanX 总开发主文档

## 1. 项目定位
TitanX 是基于 React Flow 的节点式 AI 工作流系统。当前阶段目标不是同时完成手动 + 自动化双模系统，而是先把 KIE / Gemini / Audio 能力组织成**手动可执行、可写回、可验收**的节点工作台。

## 1.1 当前版本定义
**titanX 当前阶段为 Manual-first Baseline。**
- 当前只做手动闭环
- 自动化暂不纳入本阶段验收
- 自动化 / skillBinding / agent 驱动仅作为后续阶段预留方向

### 当前技术实现概览（以代码为准）
- 前端：React 19 + TypeScript + Vite
- 画布：`@xyflow/react`
- 当前状态管理：React Flow 受控模式 + React state
- 节点范围：当前代码仅稳定落地 `Text Node` 与 `Image Node`
- 后续可引入更完整状态层，但当前代码并未正式落地 Zustand

## 3. 当前核心锚点

### [ANCHOR-UI-001] 节点交互总原则
- 节点不是点击展开整个节点本体。
- 节点必须保持缩略图/摘要卡片常驻。
- 只有点击节点缩略图时，才显示下方编辑框。
- 节点选中高亮与编辑器展开必须解耦。
- 点击画布空白处时，应关闭当前编辑器。
- 文本 / 图片 / 视频 / 音频四类生成型节点都要统一支持“生成多个”的乘数切换。
- 多结果生成后必须支持主次切换；后续下载、创建资产、继续连接下游时，默认作用于当前主结果。

### [ANCHOR-TEXT-001] Text 节点的结构化提示词语义
- Text 节点顶部工具条当前不应按“普通富文本排版器”理解，而应按“结构化提示词标注器”理解。
- 标题、小标题、列表、强调等工具的核心作用，不是单纯让文本好看，而是给后续图片 / PPT / 版式类生成提供结构信号。
- 这类结构信号的产品语义，应视为与图片节点的摄像机参数、风格按钮同层级：都是隐式 prompt 注入来源。
- 底层实现不一定必须保存为原始 HTML，但必须保留结构化语义，并能进入 prompt 编译链。
- 当前按用户截图已确认的顶部工具条顺序为：`颜色`、`H1`、`H2`、`H3`、`正文`、`粗体`、`斜体`、`无序列表`、`有序列表`、`分割线`、`复制全部文本`、`放大文本框`。
- 另新增一个产品增强按钮：`插入到下层开头`，作用是把当前 Text 节点文本内容一键插入到下游 Text / Image / Video / Audio 节点文本框开头。

### [ANCHOR-UI-002] Image Node 编辑区结构
- Image Node 选中后，下方编辑框必须包含：
  - 模型型号
  - 比例
  - 大小
  - 当前输入文本
  - 上游图片输入预览
  - 上游文本输入预览
  - 参考图槽位
  - 焦点按钮
  - 标签 chips
- 模型/比例/大小都属于编辑框内容，不属于展开区。
- 底部编辑区不是单纯 prompt 框，而是“自然语言 + `/` 快捷命令 + `@Image N` 引用 + 参考图写回”的混合输入系统。

### [ANCHOR-UI-003] Image Node 摄像机控制区
- 图片节点编辑框里必须包含摄像机控制区。
- 至少包含：
  - 摄像机型号
  - 镜头型号
- 预留：
  - 焦段
  - 光圈
- 该区和模型/比例/大小属于同一层配置。
- 摄像机参数不是纯 UI 选项，而属于隐式全局提示词注入体系；点击生成时，应自动编译进底层 prompt 前缀。

### [ANCHOR-UI-004] Image Node 风格参考图区
- 风格按钮展开的是风格参考图，而不是单纯风格枚举。
- 选中的风格参考图必须同时写回：
  - 当前节点参考图区
  - 底层 prompt 编译链
- 用户文本框中不显示这段系统自动注入的风格提示词。

### [ANCHOR-UI-005] Image Node 结果展示规则
- 多图生成结果必须按“主图 + 右侧副图展开”展示。
- 当前主图右上角应显示张数标识。
- 副图必须支持：
  - 设为主图
  - 主副图切换
  - 下载当前副图

### [ANCHOR-UI-006] Image Node 顶部按钮分段实现原则
- 当前阶段优先做通底部编辑区和链接主链路。
- 顶部 7 个主编辑动作按钮：
  - 重绘
  - 擦除
  - 增强
  - 扩图
  - 抠图
  - 多角度
  - 打光
  当前只保留字段与入口，不作为第一阶段必须打通项。
- 右侧通用按钮：
  - 标注
  - 裁剪
  - 下载
  - 全屏
  属于当前主链路实现范围。

### [ANCHOR-DATA-001] 输入输出真实绑定
- 节点输入输出不能是假数据。
- 必须按 edge 真实解析上游内容。
- Text → Image、Image → Image、Text/Image → Video、附件 → Gemini Text 都要按真实节点数据流走。

### [ANCHOR-RUNTIME-001] KIE 真实运行时
- 禁止 mock 图 / 假结果。
- 图片与视频生成必须走真实 KIE 服务端代理。
- 结果必须写回节点的 preview / result / runtime。
- 视频节点的 `解析`、`快照`、`高清派生增强节点` 也应视为真实运行链的一部分，不要只做静态 UI。
- API round1 后，运行时 family 也已需要明确拆层：
  - video: `kie-jobs-video` / `veo-family`
  - image: `kie-jobs-image` / `flux-kontext`
  - text analyze: `gemini-generate-content`
  - audio: `elevenlabs-tts` / `elevenlabs-music`

### [ANCHOR-VIDEO-001] 视频节点素材态结构
- 视频节点可通过左侧栏、双击画布空白处、图片节点输出创建菜单等入口创建。
- 上传视频后，节点进入素材态：
  - 出现顶部 7 按钮工具栏：`剪辑 / 解析 / 高清 / 替换 / 移除 / 下载 / 全屏`
  - 中部是视频浏览框，左上角静音，右上角替换上传
  - 底部是播放控制条、时间、进度条、快照按钮
- 上传视频后当前节点只保留输出口，说明该态更接近“结果浏览态 + 下游分发态”。
- 单独创建视频节点默认是 `文生视频`；接入图片上游后切到 `图生视频`，并把图片加载到首帧/参考图区。
- 视频节点底部编辑区当前至少包含：焦点按钮、文本输入框、模型选择、参数汇总按钮、数量切换、积分显示、生成按钮。

### [ANCHOR-VIDEO-002] 视频快照与分镜解析
- 视频节点底部快照按钮必须按当前播放/暂停时刻截图，并生成新的图片节点。
- 该图片节点功能应与普通图片节点完全一致，不能做成特殊阉割节点。
- `解析` 功能当前至少支持两种结果视图：
  - 详细分析列表
  - 创意图缩略图
- 解析结果至少应包含镜头序号、截图、时段、景别、镜头角度、运动、焦距/景深、光线、音乐、人声等结构化信息。
- 解析结果当前真实形态是画布上一个可拖拽、可缩放的结果框，里面同时包含图片与文字；初始尺寸可能显示不全，需要用户自行拖大查看。
- 当前复制体验也已确认存在缺陷：要双击进入编辑态才能复制文本，没有按镜头的一键复制按钮。
- 当前产品限制也要记录：解析结果暂不能直接拆成可单独链接的镜头节点，后续可作为优化方向。

### [ANCHOR-VIDEO-002C] 视频解析结果的优化优先级
- P0：先给每个镜头结果项补一键复制能力，至少包括：
  - `复制整条镜头信息`
  - `复制该镜头提示词`
  - `复制图片`
- P1：把解析结果转换为可拖拽镜头卡片/图片节点并自动打组。
- 自动打组后应支持：
  - 整组移动
  - 解组
  - 组内单镜头继续连下游
- 不推荐继续把“进入编辑态后复制文本”当作主方案，因为用户真正要的是快速复用而不是文本编辑。

### [ANCHOR-VIDEO-002A] 视频节点焦点编辑与参考图写回
- 视频节点底部编辑区当前已确认存在 `图生视频` 模式。
- 视频节点也继承与图片节点一致的 `焦点编辑` 逻辑。
- 进入焦点编辑后，画布进入 picking mode，提示为“点击其他节点以提取元素”，并带 `退出` 按钮。
- 从其他节点提取出的焦点，应写回为视频节点的参考图与标签，而不是只做一次性临时选择。
- 因此视频节点不能只设计成 `prompt + model + duration`，还必须预留参考图槽位与标签写回结构。
- 当前左侧输入口允许接入 `文本 / 图片 / 音频`；右侧输出创建菜单当前只允许 `文本生成 / 视频编辑（下一个视频节点）`。

### [ANCHOR-VIDEO-002B] 视频剪辑(beta)规则
- `剪辑(beta)` 的交互形态类似 PR 时间轴，但能力只作用于当前视频节点素材本身。
- 当前只允许设置这个视频节点的 `入点 / 出点`，把素材裁切成一段。
- 左侧是 `取消`，右侧是 `确定`；确定后写回当前节点素材时间范围。
- 如果用户对裁切结果不满意，应可返回后再次进入剪辑重做。
- 这不是完整视频编辑器，而是节点级片段裁切器。

### [ANCHOR-VIDEO-003] 视频高清增强派生规则
- 点击视频节点顶部 `高清` 后，应自动创建并连接一个视频增强节点。
- 视频增强节点当前至少包含：
  - 处理时长：5 到 10 分钟
  - 输出分辨率：1080P / 2K / 4K
  - 帧数：自适应 / 30 / 60 / 90
  - 帧率增强：自适应 / 2x
- 增强后当前只允许继续连接：
  - 文本节点
  - 视频编辑节点（当前先按新的视频节点族理解）

### [ANCHOR-VIDEO-004] 视频全屏查看器
- 第七个按钮 `全屏` 必须进入独立的视频全屏查看器，而不是简单放大弹窗。
- 当前全屏模式至少包含：
  - 播放/暂停
  - 进度条
  - 声音按钮
  - 下载
  - 放大
  - 缩小
  - 右上角叉号关闭

### [ANCHOR-VIDEO-005] 视频参数条与多结果规则
- 视频节点底部参数汇总当前至少覆盖：比例、清晰度、生成时长、生成音频开关。
- 当前已确认的视频参数档位至少包括：
  - 比例：`16:9 / 4:3 / 1:1 / 3:4 / 9:16 / 21:9`
  - 清晰度：`480p / 720p / 1080p`
  - 时长：`5s / 10s`
  - 生成音频：`开启 / 关闭`
- 当前视频生成数量切换只有 `1x / 2x`。
- 视频生成多个结果时，主次切换逻辑与图片节点一致：
  - 可切换当前主视频
  - 下游引用默认跟随当前主视频
  - 下载默认下载当前主视频
- 比例 / 清晰度 / 秒数 / 自适应等参数必须根据当前视频模型动态切换可选项；不能假设所有模型共享同一套参数面板。

### [ANCHOR-VIDEO-006] 视频多图输入语义
- 视频多图输入是否可用，取决于当前视频模型是否支持；不支持时需要切换模型。
- 当有 2 张图片节点接入视频节点时：
  - 第一张作为 `首帧`
  - 第二张作为 `尾帧`
- 在首尾帧模式下，还可额外添加 1 张 `风格图`，形成 `首帧 + 尾帧 + 风格图` 的三图结构。
- `风格参考` 入口当前会打开资产选择面板，从资产库中选择后写入风格位。
- 当接入 3 张图片节点并切到多帧模型时，当前按 `多帧模型` 理解，不再单独提供风格图选择位。
- 视频节点完整整合入口已单独落盘：`titanX/项目文档/12_视频节点整合总表_2026-03-11.md`。

### [ANCHOR-AUDIO-001] 音频节点三段式与素材态
- 音频节点继承统一三段式：上方上传/操作区、中间音频缩略浏览区、下方独立编辑区。
- 上传本地音频后，下方会出现波形图、播放/暂停、时间信息等播放器能力。
- 顶部当前至少包含：播放速度调节、下载、上传/替换当前音频素材。
- 播放速度默认 `1x`，最高 `2x`；这是播放器控制，不是生成参数。
- 上传音频后当前节点只保留输出口，不再保留左输入口；该态应按素材浏览/结果分发态理解。

### [ANCHOR-AUDIO-002] 音频节点编辑区模式与参数
- 音频节点底部编辑区基础骨架与其他生成型节点一致：文本输入框、放大按钮、模式位、模型位、参数展开位、字数统计、积分显示、生成按钮。
- 第一按钮切换 `文字转语音` / `音乐` 两种模式。
- `音乐` 模式当前至少有 1 个模型，且第三按钮展开后至少包含：`纯音乐` 开关、`Auto / 30s / 1m / 2m / 3m / 4m` 时长位。
- `文字转语音` 模式当前至少有 2 个模型，且参数区至少包含：音色选择、语速、稳定性、相似度、风格强度。
- 这些参数必须按当前音频模型动态切换，不允许把所有模型硬塞进同一静态参数面板。
- 文本框放大只是同一编辑区的放大版，不应改变按钮结构、积分位与生成按钮逻辑。

### [ANCHOR-AUDIO-003] 音频节点输入输出与数字人语义
- 音频节点左侧输入当前只支持文本节点。
- 音频节点右侧输出当前只支持：`视频`、`数字人`。
- 上传本地音频后的素材态也保持同样的右侧输出能力。
- `数字人` 当前更准确地理解为视频节点的特化模式，而不是独立运行时。
- 当从音频节点创建 `数字人` 时，当前应默认理解为：创建视频节点、自动套用 `Wan 2.6`、自动载入音频输入。
- 除 `Wan 2.6` 的高级设置外，数字人参数当前基本继承普通视频模型；其中高级设置可能承担多镜头、音频驱动口型或说话视频生成语义。

### [ANCHOR-RUNTIME-002] API 结果 URL 保存规则
- 使用在线 API 生成图片/视频后，必须保存返回 URL 到项目目录 JSON。
- 标准格式：
```json
{
  "taskId": "...",
  "model": "...",
  "status": "success",
  "resultUrls": ["..."],
  "createTime": "2026-03-10T15:00:00+08:00"
}
```
- 原因：媒体文件太大，优先保存 URL 供复用。

### [ANCHOR-CANVAS-001] 画布基础操作

- 左侧添加节点面板底部的 `上传` 入口当前至少支持：`图片 / 视频 / 音频 / 文本(TXT)`。
- `上传` 不是单纯导入资产，而是按文件类型自适应创建对应节点。
- 按住鼠标拖动可框选多个节点。
- 画布空白处右键菜单当前至少包含：
  - 上传
  - 添加资产
  - 添加节点
  - 撤消
  - 重做
  - 粘贴

### [ANCHOR-CANVAS-002] 单节点与多选右键能力
- 选中单个节点右键菜单当前至少包含：

  - 创建资产
  - 复制
  - 粘贴
  - 副本
  - 删除

  - 创建资产

### [ANCHOR-CANVAS-003] 打组 / 模板规则
- 多节点框选后应支持打组。
- 打组不是纯视觉框，而是有独立语义：
  - 组颜色
  - 组布局
  - 组执行工作流
- 当前已确认的组颜色入口为顶部工具条最左侧颜色圆点，点击后弹出背景颜色选择面板。
- 当前已确认的组布局入口会展开至少两种布局：
  - 宫格布局
  - 水平布局
- 打组后的组级快捷栏当前至少包含：
  - 整组执行
  - 创建模板
  - 解组
- `整组执行` 已确认是实际执行入口，而不是占位按钮。
- 当前截图已确认：当组内包含未填写/未就绪节点时，整组执行会失败，并给出失败提示。
- 打组后应支持“创建模板”，本质上是把打组结果保存为工作流模板。
- 打组时仍可继续操作组内单个节点。
- 选中组框空白处时，应支持整组移动。
- 解开打组后，节点恢复为独立单节点状态。

### [ANCHOR-ID-001] 节点业务 ID 规则
- 每个新建节点必须使用按类型独立编号的业务 ID。
- 例如：`text-01`、`text-02`、`image-01`、`image-02`、`video-01`。
- 删除后复用该类型最小缺失编号。
- 当前阶段至少要先保证**手动创建路径**稳定使用同一套 allocator。
- 自动化路径复用同一 allocator 作为后续阶段要求保留，但不作为当前验收项。

### [ANCHOR-CONTRACT-001] 当前阶段 contract 原则
- 当前阶段保留最小 contract 思维，但只要求服务**手动闭环开发**。
- 必须尽量明确：
  - 使用的工具/执行入口
  - 输入
  - 输出
  - 写回目标
- 自动化 skill/tool contract 的完整闭环可延后到后续阶段补齐。

## 4. 产品边界与非目标
- titanLX 不是通用可编程工作流平台。
- titanLX 不是只有表单式 storyboard 壳子。
- titanLX 必须是可执行节点 + 结果写回 + 本地/索引持久化的媒体工作台。
- V1 要避免任意脚本节点、任意 provider 控制面板化、过度暴露底层 runtime 细节。

## 5. 参数分层规则
所有字段必须先归类，再决定是否放进 UI：
1. operator-facing：用户直接编辑的高价值稳定字段
2. flow-level：画布节点和边结构相关字段
3. compiled-config-level：编译后、可执行前的标准化配置
4. adapter-level：桥接 provider/runtime 的内部适配字段
5. runtime-internal：轮询、原始响应、重试等内部状态

规则：
- 只把 operator-facing 字段直接暴露给用户
- 不把 raw provider payload 直接暴露进画布
- 节点 UI 不应退化成 provider 控制台

- Text Node

## 7. 运行时与结果写回原则
- 当前阶段的执行模式主入口只认：`manual`
- `skill / acp / 自动批量执行` 等模式保留为后续阶段扩展方向，但不纳入当前阶段验收
- 每个可执行节点执行后必须写回：
  - node status
  - runtime info
  - remote result url(s)
  - local artifact path(s) 或结果索引引用
  - summary preview info
- 节点级 Analyze 是核心交互，分析结果也属于节点写回的一部分。
- 运行时可靠性建设按阶段推进：先完成**手动真实闭环**，再补结构化错误处理、有限重试、并发控制。

## 8. 结果持久化原则
在线生成结果不能只显示不落记录。
至少要保存：
- taskId
- model
- status
- resultUrls[]
- createTime

后续如本地落盘可用，还要补：
- local_path
- preview_path
- metadata_path
