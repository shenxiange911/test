## 2. 精简方案：三层文档架构

### 2.1 推荐新架构

```
titanX/项目文档/
├── README.md                          # 项目总入口
├── 00_架构总览.md                      # 新增：统一架构概念层
│
├── core/                              # 核心规范（单一真相来源）
│   ├── schema.md                      # 统一 Schema 定义
│   ├── runtime.md                     # 统一 Runtime 规范
│   ├── state-machine.md               # 统一状态机定义
│   └── data-flow.md                   # 统一数据流规范
│
├── nodes/                             # 节点实现规范
│   ├── text-node.md                   # Text 节点完整规范
│   ├── image-node.md                  # Image 节点完整规范
│   ├── video-node.md                  # Video 节点完整规范
│   └── audio-node.md                  # Audio 节点完整规范
│
├── api/                               # API 映射层
│   ├── provider-families.md           # Provider 家族总表
│   ├── gemini.md                      # Gemini API 映射
│   ├── kie-jobs.md                    # KIE Jobs API 映射
│   ├── flux-kontext.md                # Flux Kontext API 映射
│   └── veo.md                         # Veo API 映射
│
├── implementation/                    # 实现指南
│   ├── current-anchor.md              # 当前开发锚点
│   ├── progress.md                    # 实现进度追踪
│   └── code-mapping.md                # 代码映射总表
│
└── archive/                           # 归档（调研过程/字段确认）
    ├── research/
    └── field-confirmation/
```

### 2.2 精简原则

#### 原则 1：单一真相来源（Single Source of Truth）
- 每个概念只在一个文件中定义
- 其他文件通过引用链接，不重复定义
- 例如：`BaseNodeShell` 只在 `core/schema.md` 定义一次

#### 原则 2：分层不重复
- **Core 层**：定义通用概念和接口
- **Nodes 层**：继承 Core，只写节点特有逻辑
- **API 层**：只写 API 映射，不重复 Schema
- **Implementation 层**：只写实现指南，不重复规范

#### 原则 3：调研归档
- 调研过程文档移入 `archive/research/`
- 字段确认文档移入 `archive/field-confirmation/`
- 正式规范不再引用调研文档

## 6. 实施建议

### 6.1 分阶段实施

#### 阶段 1：创建新架构骨架（1-2 小时）
1. 创建 `core/` `nodes/` `api/` `implementation/` 目录
2. 创建空白模板文件
3. 更新 `README.md` 指向新架构

#### 阶段 2：合并 Core 层（2-3 小时）
1. 提取 `BaseNodeShell` / `CommonRuntimeState` 到 `core/schema.md`
2. 提取状态机定义到 `core/state-machine.md`
3. 提取数据流规范到 `core/data-flow.md`

#### 阶段 3：重构 Nodes 层（3-4 小时）
1. 合并 Text/Image/Video/Audio 节点文档
2. 删除重复的 Schema 定义

#### 阶段 4：重构 API 层（2-3 小时）

2. 拆分 KIE Jobs / Flux / Veo / Gemini

### 6.2 风险控制

#### 风险 1：丢失重要信息
**缓解措施**：
- 不删除任何文件，只移动到 `archive/`
- 在新文档中添加"迁移自"注释
- 保留 Git 历史

#### 风险 2：破坏现有引用
**缓解措施**：
- 在旧文件位置添加重定向说明
- 更新 `08_当前开发锚点` 中的文件路径
- 逐步迁移，不一次性删除

#### 风险 3：团队不适应新架构
**缓解措施**：
- 在 `README.md` 中添加"如何查找文档"指南
- 提供旧文档到新文档的映射表
- 保留 1-2 周过渡期

## 8. 总结

### 8.1 核心问题
titanX 项目文档当前最大的问题不是"文档不够"，而是：
1. **重复定义**：同一个接口在 3-4 个文件中重复
2. **分层混乱**：调研、规范、实现混在一起
3. **维护困难**：修改一个概念需要同步多个文件

### 8.2 精简方案核心
1. **单一真相来源**：每个概念只定义一次
2. **三层架构**：Core（通用）→ Nodes（特化）→ Implementation（实现）
3. **调研归档**：历史文档移入 `archive/`，不污染主文档

### 8.3 预期收益
- 文档行数减少 50%
- 重复定义减少 90%
- 维护成本降低 70%
- 新人上手时间缩短 60%

2. **本周完成**：合并 Core 层和 Nodes 层
