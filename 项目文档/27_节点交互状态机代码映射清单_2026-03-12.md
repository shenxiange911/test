## 3. 哪些状态应该继续放 React Flow node data

### 3.1 继续放 `node.data` 的状态

下面这些状态会随节点本体持久化、复制、保存模板、运行写回一起存在，因此应继续放在 React Flow node data：

| 字段 | 原因 | 当前代码映射 |
| --- | --- | --- |
| `businessId` | 节点业务身份 | 扩展 `data` |
| `nodeType` | 节点家族本体 | 由当前 `kind` 升级 |
| `contentMode` | 决定节点主体显示空态/素材态/结果态 | 扩展 `data` |
| `runtimeStatus` | 决定运行中/成功/失败展示 | 扩展 `data` |
| `hasLeftInput` | 与节点主体模式绑定 | 扩展 `data` |
| `hasRightOutput` | 与节点主体模式绑定 | 扩展 `data` |
| `runtime` | taskId / 错误 / 原始结果写回 | 扩展 `data` |
| `resultState` | 多结果与主结果切换必须持久化 | 扩展 `data` |
| 节点内容字段 | 如 `prompt` / `uploadedAudioUrl` / `inputImages[]` | 扩展 `data` |

### 3.2 当前代码的起点

`/home/liudehua/.openclaw/workspace/titanX/app/src/App.jsx` 当前节点数据是：

```js
data: { label: 'Analysis', kind: 'analysis' }
```

建议第一步先升级成：

```js
data: {
  label,
  nodeType,
  businessId,
  contentMode: 'empty',
  runtimeStatus: 'idle',
  hasLeftInput: true,
  hasRightOutput: true,
  runtime: null,
  resultState: { primaryResultId: null, results: [] },
}
```

### 3.3 为什么这些必须留在 node data

因为它们会影响：
- 节点本体渲染
- 节点复制
- 节点保存
- 节点结果写回
- 模板导出

如果把这些状态放在纯 UI state，会导致节点一旦重新渲染或复制就丢语义。

## 7. picking mode 适合挂在哪层，如何避免污染普通选中逻辑

### 7.1 适合挂在画布级 UI state

推荐挂在：
- 当前阶段：`/home/liudehua/.openclaw/workspace/titanX/app/src/App.jsx`
- 后续阶段：独立 `flowStore`

不应挂在：
- 节点 `data`
- `Inspector.jsx` 本地 state
- 单节点组件本地 state

### 7.2 推荐结构

```ts
type CanvasMode = 'idle' | 'creating-from-canvas' | 'creating-from-output' | 'focus-pick'

interface FocusPickState {
  active: boolean
  sourceNodeId: string | null
  targetKind?: 'focus-ref' | 'style-ref' | 'image-ref'
}
```

### 7.3 避免污染普通选中逻辑的关键

关键不是再造一个“特殊 selected”，而是在 `onNodeClick` 分流：

```ts
if (canvasMode === 'focus-pick') {
  handleFocusPick(targetNode)
  return
}

handleNormalNodeSelect(targetNode)
```

也就是说：
- 普通选中逻辑继续服务 `selectedNodeIds`
- picking mode 只是临时接管点击解释权
- 退出 picking 后，恢复普通选中逻辑

### 7.4 推荐实现边界

可直接做：
- 进入 / 退出 picking mode
- 选择一个目标节点后写回 source node data

需要包装后做：
- 从视频解析结果框里挑镜头截图
- 自动生成标签 chips
- 多种 targetKind 共用统一拾取入口

不建议 v1：
- 多轮 picking 会话栈
- picking 中多选
- picking 中复杂候选高亮编排

### 7.5 风险点 / 踩坑点

- 如果把 picking state 写进节点 data，会污染节点持久化
- 如果让普通 `selected` 承担 picking 含义，会导致模式退出后状态错乱
- 如果不在 `onNodeClick` 层分流，而在子组件里各自判断，最终一定会散乱
