# titanX 节点交互状态机总表

记录时间：2026-03-12
定位：Text / Image / Video / Audio 四类节点的统一交互状态机定义
状态：可直接作为 titanX 手动版节点实现的状态层参考

---

## 1. 文档目的

这份文档不讲泛化 UI 理论，只回答 titanX 当前最实际的一个实现问题：

**在 React Flow 画布中，Text / Image / Video / Audio 四类节点，如何共用一套能直接落代码的交互状态机。**

本次只覆盖这些规则：
- 缩略图常驻
- 点击缩略图展开下方编辑区
- 选中高亮与展开解耦
- 点击画布空白关闭编辑器
- 上传素材后切素材态
- 多结果主次切换
- picking mode 进入/退出
- 节点选中、编辑、运行、结果态切换

---

## 2. 统一状态机定义

### 2.1 不要只用一个 `status`

titanX 节点交互至少要拆成 4 个独立维度：

1. `selection state`：节点是否被选中
2. `editor state`：节点下方编辑区是否展开
3. `content mode`：节点当前处于空态 / 素材态 / 生成态 / 结果态哪一种内容模式
4. `runtime state`：节点是否正在执行、成功、失败

如果只保留一个 `status`，会出现这些问题：
- 选中节点时无法区分是不是展开了编辑区
- 上传素材后无法表达“是素材浏览态，不是运行成功态”
- 多结果存在时无法区分“当前主结果”和“节点整体成功”
- picking mode 会被错误地塞进节点自身状态里

### 2.2 统一状态维度

推荐统一拆成：

```ts
selection: 'unselected' | 'selected'
editor: 'closed' | 'open'
contentMode: 'empty' | 'draft' | 'asset' | 'generated'
runtimeStatus: 'idle' | 'running' | 'success' | 'error'
```

说明：
- `selection` 只管高亮和当前聚焦对象
- `editor` 只管下方编辑区是否显示
- `contentMode` 只管节点本体中部显示什么
- `runtimeStatus` 只管执行链是否在跑

### 2.3 统一状态语义

| 维度 | 值 | 语义 |
| --- | --- | --- |
| selection | `unselected` | 节点未被选中，不高亮 |
| selection | `selected` | 节点被选中，高亮 |
| editor | `closed` | 下方编辑区关闭 |
| editor | `open` | 下方编辑区展开 |
| contentMode | `empty` | 新建空节点，还没有内容或素材 |
| contentMode | `draft` | 有草稿/参数，但还没有素材或结果 |
| contentMode | `asset` | 已上传素材，进入素材浏览态 |
| contentMode | `generated` | 已有生成结果，进入结果态 |
| runtimeStatus | `idle` | 未执行，或执行完成后静止 |
| runtimeStatus | `running` | 正在执行 |
| runtimeStatus | `success` | 最近一次执行成功 |
| runtimeStatus | `error` | 最近一次执行失败 |

### 2.4 统一状态组合约束

不是所有状态组合都合理，建议明确这些约束：

| 组合 | 是否允许 | 说明 |
| --- | --- | --- |
| `selected + editor=open` | 允许 | 点击缩略图的标准路径 |
| `selected + editor=closed` | 允许 | 框选、移动、右键菜单、仅高亮不编辑 |
| `unselected + editor=open` | 不建议 | 会造成“谁在编辑”语义混乱 |
| `contentMode=asset + runtimeStatus=idle` | 允许 | 已上传素材但未生成 |
| `contentMode=generated + runtimeStatus=success` | 允许 | 标准成功结果态 |
| `contentMode=generated + runtimeStatus=running` | 允许 | 已有旧结果，同时正在跑新任务 |
| `contentMode=empty + runtimeStatus=running` | 允许 | 从空态直接发起生成 |

## 3. 四类节点共性与差异

### 3.1 共性

四类节点都共用这些交互骨架：
- 缩略图/摘要区常驻
- 点击缩略图时展开下方编辑区
- 节点选中高亮与编辑器展开解耦
- 点击画布空白关闭当前编辑器
- 有运行态、成功态、失败态
- 可以被框选、多选、打组
- 可以通过右键进入命令动作
- 结果一旦存在，都应有“当前主结果”的概念

### 3.2 差异表

| 节点类型 | 是否有素材态 | 是否有多结果主次切换 | 是否有 picking mode 写回 | 特别说明 |
| --- | --- | --- | --- | --- |
| Text | 否 | 弱（通常不是多结果卡片） | 否 | 更偏结构化文本编辑与分析结果写回 |
| Image | 是 | 是 | 是 | 主图/副图切换、参考图写回最强 |
| Video | 是 | 是 | 是 | 素材态、快照、解析、高清派生更复杂 |
| Audio | 是 | 是 | 可预留 | 既有 TTS / music 编辑态，也有上传音频后的播放器素材态 |

### 3.3 Text 节点差异

Text 节点不应硬套素材态。

推荐理解：
- `empty`：新建空文本节点
- `draft`：已有内容或结构化提示词内容
- `generated`：如果未来支持分析结果回写或多版本，也可进入结果态

但 Text 节点当前不建议为了“统一”而强造波形、缩略图式素材态。

### 3.4 Image 节点差异

Image 节点是统一状态机最标准的参考节点：
- 空态：无图、无结果
- 草稿态：有 prompt / 参数 / 上下文，但未执行
- 素材态：上传图片后，以素材浏览/编辑为主
- 结果态：生成一个或多个结果图

多结果主次切换在 Image 节点里必须做成一等能力，而不是临时 UI。

### 3.5 Video 节点差异

Video 节点除了统一状态机外，还额外带这些特殊扩展：
- 上传视频后进入素材态
- 素材态下顶部有剪辑 / 解析 / 高清 / 替换 / 移除 / 下载 / 全屏
- 已有结果时也可能继续运行新的生成任务
- 视频解析结果当前建议写回节点字段，并以独立结果框展示，而不是先拆成子节点

所以 Video 节点的关键不是另起一套状态机，而是在统一状态机上叠加“视频专属动作”。

### 3.6 Audio 节点差异

Audio 节点和 Video 很像，但更强调模式切换：
- `tts`
- `music`
- `dialogue`
- 上传本地音频后的播放器素材态

要点：
- 播放速度是播放器状态，不是生成状态
- 上传素材后只保留输出口的规则，要由 `contentMode=asset` 驱动

## 4. 哪些状态放 React Flow node data

原则：
- **会随节点持久化、复制、保存模板、写回结果一起保存的状态，放 node data**
- **只属于当前界面瞬时交互的状态，不放 node data**

### 4.1 应放 `node.data` 的状态

| state | why |
| --- | --- |
| `businessId` | 节点业务身份的一部分 |
| `nodeType` | 节点类型本体 |
| `contentMode` | 节点当前内容模式，会影响节点本体渲染 |
| `runtimeStatus` | 节点执行状态，会影响卡片显示 |
| `hasLeftInput` | 与节点当前模式直接相关 |
| `hasRightOutput` | 与节点当前模式直接相关 |
| `runtime` | 任务、错误、结果写回都依赖 |
| `resultState` | 多结果、主结果必须持久化 |
| 各节点自己的内容字段 | 如 `prompt`、`uploadedAudioUrl`、`inputImages[]` |

### 4.2 推荐的 `node.data` 最小字段

```ts
interface CommonNodeViewStateInData {
  contentMode: 'empty' | 'draft' | 'asset' | 'generated'
  runtimeStatus: 'idle' | 'running' | 'success' | 'error'
  hasLeftInput: boolean
  hasRightOutput: boolean
  runtime?: CommonRuntimeState
  resultState?: MultiResultState
}
```

### 4.3 为什么 `contentMode` 要放 node data

因为它直接决定节点中部显示：
- 空白占位
- 素材浏览器
- 结果展示区
- 草稿摘要区

这些都属于节点本体，而不是外层浮层 UI。

## 5. 哪些状态放外层 UI state

原则：
- **只影响当前画布会话的瞬时交互，不属于节点内容本体的状态，放外层 UI state**

### 5.1 应放外层 UI state 的状态

| state | why |
| --- | --- |
| `selectedNodeIds` | React Flow 选择态本来就属于画布层 |
| `editorNodeId` | 当前哪个节点编辑器展开，属于全局单例/少量实例控制 |
| `contextMenu` | 临时浮层 |
| `creationMenu` | 临时浮层 |
| `pendingConnection` | 临时建边过程态 |
| `canvasMode` | 画布级行为模式，不属于单节点 |
| `focusPickingSourceNodeId` | picking mode 源节点，属于画布流程态 |
| `hoveredHandle` | 瞬时 hover 态 |

### 5.2 `editorExpanded` 不建议直接落 node data

虽然它看起来和节点相关，但更适合作为外层状态：
- 因为“当前谁展开了编辑器”通常是画布级约束
- 点击空白关闭编辑器时，需要统一清空
- 多选时一般不应该多个编辑器同时开

推荐不要在每个节点 data 里放一个布尔 `editorExpanded`，而是改成：

```ts
interface FlowUiState {
  editorNodeId: string | null
}
```

判断逻辑：

```ts
const editorOpen = ui.editorNodeId === node.id
```

### 5.3 `selected` 不建议长期冗余写入 node data

因为 React Flow 本身就有 selection 体系。
除非确有导出/调试需求，否则不要双写：
- 一份在 React Flow selection
- 一份在 node.data.selected

否则很容易出现不同步。

## 6. picking mode 进入/退出建议

### 6.1 picking mode 是画布态，不是节点态

推荐只放在外层：

```ts
type CanvasMode = 'idle' | 'creating-from-canvas' | 'creating-from-output' | 'adding-context' | 'focus-pick' | 'grouping'
```

并配一个载荷：

```ts
interface FocusPickState {
  active: boolean
  sourceNodeId: string | null
  targetKind: 'image-ref' | 'focus-ref' | 'style-ref'
}
```

### 6.2 进入规则

触发：点击当前节点编辑区中的焦点按钮

状态变化：
- `canvasMode = 'focus-pick'`
- `focusPickState.active = true`
- `focusPickState.sourceNodeId = currentNodeId`
- 当前编辑器保持打开
- 画布出现“退出”按钮与提示文案

### 6.3 退出规则

触发：
- 点击退出按钮
- 完成一次合法拾取
- 点击不合法区域后用户取消

状态变化：
- `canvasMode = 'idle'`
- `focusPickState.active = false`
- `focusPickState.sourceNodeId = null`

### 6.4 可直接做

- 进入/退出 picking mode
- 显示提示条
- 从图片/视频节点取主结果作为参考图写回

### 6.5 需要包装后做

- 从解析结果框中抽取镜头截图再写回
- 从多节点中批量提取焦点集合
- picking 完成后自动生成标签 chips

### 6.6 不建议放进 v1

- 多轮 picking 会话栈
- 复杂的跨组/跨结果框拾取回溯
- picking 过程中联动高亮所有候选槽位

### 6.7 风险点 / 踩坑点

- 把 picking mode 误塞进节点 data，会导致退出流程很乱
- 如果 editor 跟 selected 不解耦，进入 picking 时容易把当前节点编辑器错误关掉
- picking 的目标如果直接写 raw provider payload，会污染统一 schema

## 7. 多结果主次切换规则

### 7.1 统一结果结构

沿用已有统一 schema：

```ts
interface NodeResultItem {
  resultId: string
  kind: 'text' | 'image' | 'video' | 'audio'
  url?: string
  previewUrl?: string
  thumbnailUrl?: string
  text?: string
  duration?: number
  width?: number
  height?: number
  model?: string
  promptSnapshot?: string
  createTime?: number
}

interface MultiResultState {
  primaryResultId?: string
  results: NodeResultItem[]
}
```

### 7.2 统一规则

- 主结果只有一个
- 副结果可以有多个
- 下游默认引用主结果
- 下载默认下载主结果
- 创建资产默认用主结果
- 切换主结果不应改动节点 `runtimeStatus`

### 7.3 可直接做

- `primaryResultId` 切换
- 根据 `primaryResultId` 渲染主卡片
- 下载当前主结果
- 从副结果设为主结果

### 7.4 需要包装后做

- 主副图动画切换
- 结果版本历史对比
- “设为主结果后同步改写所有下游预览”这种联动刷新

### 7.5 不建议放进 v1

- 多结果打分排序
- 结果收藏夹/结果 pin
- 主结果与下游边的版本锁定体系

### 7.6 风险点 / 踩坑点

- 如果直接把主结果对象整份复制到节点顶层，很容易和 `results[]` 不一致
- 必须只存 `primaryResultId`，不要存重复结果副本
- Text 节点如果暂时没有多结果，不要为了统一而伪造结果数组 UI

## 8. 上传素材后切素材态规则

### 8.1 统一素材态定义

素材态不是运行态，它是 `contentMode='asset'`。

适用节点：
- Image
- Video
- Audio

### 8.2 素材态统一行为

进入素材态后：
- 中部显示素材浏览器，而不是空白占位
- 顶部显示素材相关动作
- 输入口可按节点规则隐藏
- 输出口通常保留
- 底部编辑区仍可存在，但其语义从“纯生成”切到“基于素材的再编辑/再生成”

### 8.3 各类节点差异

| 节点类型 | 上传后 contentMode | 输入口变化 | 中部变化 |
| --- | --- | --- | --- |
| Image | `asset` | 可保留或按模式切换 | 显示图片素材预览 |
| Video | `asset` | 隐藏左输入，仅保留输出 | 显示视频播放器 + 顶部工具条 |
| Audio | `asset` | 隐藏左输入，仅保留输出 | 显示波形/播放器 |
| Text | 不建议用 `asset` | - | - |

### 8.4 可直接做

- 上传后切 `contentMode='asset'`
- 依据 `contentMode` 改节点主体渲染
- 依据 `contentMode` 改输入口显示

### 8.5 需要包装后做

- 资产替换与结果态之间的平滑切换
- 素材态 + 旧结果并存视图
- 上传后自动推断模式（如视频节点自动切 `image-to-video`）

### 8.6 不建议放进 v1

- 素材态和多版本结果态同时分屏
- 上传多个素材形成资产栈浏览器
- 复杂素材回滚历史

### 8.7 风险点 / 踩坑点

- 把素材态误当成 `runtimeStatus='success'` 会让语义混乱
- `asset` 是内容模式，不是执行结果
- Video / Audio 上传后隐藏左输入口的逻辑，要由 `contentMode` 驱动，而不是写死在组件 if/else

## 9. 节点选中、编辑、运行、结果态切换

### 9.1 推荐最小切换图

```text
新建节点
  -> contentMode=empty, runtimeStatus=idle

输入内容/参数
  -> contentMode=draft, runtimeStatus=idle

点击缩略图
  -> selection=selected, editor=open

点击生成
  -> runtimeStatus=running

生成成功
  -> runtimeStatus=success, contentMode=generated

生成失败
  -> runtimeStatus=error

上传素材
  -> contentMode=asset, runtimeStatus=idle

素材基础上再生成成功
  -> contentMode=generated, runtimeStatus=success
```

### 9.2 点击缩略图规则

统一推荐：
- 点击缩略图：`selected + editor=open`
- 点击节点非缩略图区域：只 `selected`
- 点击空白处：`editor=closed`，是否清空选择可按当前产品交互再定

### 9.3 编辑区展开关闭规则

推荐：
- 同一时刻只允许一个主编辑器展开
- 关闭编辑器不等于清空结果、不等于取消选中
- 关闭编辑器只影响下方编辑区

### 9.4 可直接做

- 单编辑器展开模型
- 点击空白关闭编辑器
- 运行态 loading
- 成功/失败态 badge

### 9.5 需要包装后做

- 多节点同时编辑
- 编辑区停靠/钉住
- 编辑区浮窗化

### 9.6 不建议放进 v1

- 编辑区 tab 化
- 跨节点编辑栈历史
- 多编辑器并开 + 同步滚动

### 9.7 风险点 / 踩坑点

- 如果缩略图点击和节点整体点击没有分开，会导致“一点就乱开编辑器”
- 如果 `editor=open` 直接写回模板，会把瞬时界面态污染到持久化数据里
- 如果 runtimeStatus 成功就自动强制展开编辑器，用户体验会很吵

## 10. 最小 v1 状态集合

### 10.1 最小必须实现集合

```ts
selection: 'unselected' | 'selected'
editor: 'closed' | 'open'
contentMode: 'empty' | 'draft' | 'asset' | 'generated'
runtimeStatus: 'idle' | 'running' | 'success' | 'error'
primaryResultId?: string
results: NodeResultItem[]
canvasMode: 'idle' | 'focus-pick' | 'creating-from-output'
```

### 10.2 对应落实建议

| 字段 | v1 是否必须 | 存放位置 |
| --- | --- | --- |
| `selectedNodeIds` | 是 | 外层 UI / React Flow |
| `editorNodeId` | 是 | 外层 UI |
| `contentMode` | 是 | node data |
| `runtimeStatus` | 是 | node data |
| `runtime` | 是 | node data |
| `resultState.results` | Image/Video/Audio 是 | node data |
| `resultState.primaryResultId` | Image/Video/Audio 是 | node data |
| `canvasMode` | 是 | flowStore |
| `focusPickState` | Image/Video 可先做 | flowStore |

### 10.3 推荐最小实现方案

按顺序落地：
1. 单节点选中 / 缩略图点击 / 编辑区展开关闭
2. `contentMode` 驱动节点主体渲染
3. `runtimeStatus` 驱动运行态和结果态
4. `resultState.primaryResultId` 驱动主次切换
5. `canvasMode='focus-pick'` 驱动 picking mode

## 11. 不建议 v1 过早实现的复杂态

### 11.1 清单

| complex state | why not v1 |
| --- | --- |
| 多编辑器同时展开 | 会显著增加全局协调复杂度 |
| 组级状态机 | 当前先做组命令，不先做组状态机 |
| 视频解析结果自动拆镜头子节点 | 依赖更多节点派生与打组能力 |
| 结果版本锁定到下游边 | 会引入边级版本管理 |
| picking 多步会话栈 | 当前用户只需要进入/退出即可 |
| 素材态与结果态并排双面板 | 增加节点体复杂度，不利于先通主链路 |

### 11.2 不建议放进 v1 的原因归纳

- 这些都不是“有没有状态机”的问题，而是“要不要先做复杂产品层”的问题
- titanX 当前是 Manual-first Baseline，不该一开始就做多层嵌套交互态

## 12. 推荐 TS 类型草案

### 12.1 共用状态类型

```ts
type NodeSelectionState = 'unselected' | 'selected'
type NodeEditorState = 'closed' | 'open'
type NodeContentMode = 'empty' | 'draft' | 'asset' | 'generated'
type NodeRuntimeStatus = 'idle' | 'running' | 'success' | 'error'

type CanvasMode =
  | 'idle'
  | 'creating-from-canvas'
```

## 13. 可直接做 / 需要包装后做 / 不建议放进 v1

### 13.1 可直接做

- 缩略图点击展开下方编辑区
- 选中高亮与展开解耦
- 点击画布空白关闭编辑器
- `contentMode` 切 `empty / draft / asset / generated`
- `runtimeStatus` 切 `idle / running / success / error`
- 多结果 `primaryResultId` 主次切换
- picking mode 的进入/退出
- 上传素材后切素材态并按模式改输入输出口

### 13.2 需要包装后做

- Video / Audio 素材态与结果态之间的平滑切换
- Image / Video picking 完成后自动写回标签 chips
- 多结果切换后联动下游预览刷新
- Video 解析结果框和主节点之间的联动状态

### 13.3 不建议放进 v1

- 多编辑器并开
- 复杂组级状态机
- 视频解析结果自动拆镜头节点
- 版本锁定到边级
- 多轮 picking 会话栈
- 素材态和结果态双面板并存

### 13.4 推荐最小实现方案

1. 外层先建立 `selectedNodeIds + editorNodeId + canvasMode`
2. node data 里补齐 `contentMode + runtimeStatus + resultState`
3. 节点本体统一按 `contentMode` 渲染空态/素材态/结果态
4. 编辑区统一按 `editorNodeId === node.id` 决定是否展开
5. 运行按钮统一只改 `runtimeStatus` 与 `runtime`
6. Image / Video / Audio 先落地 `primaryResultId` 逻辑
7. 最后再接入 `focus-pick`

### 13.5 风险点 / 踩坑点

- 把 `selected`、`editorExpanded`、`runtimeStatus` 混成一个字段，会导致状态语义崩掉
- 把 `editorExpanded` 写进 node data，会污染持久化与模板导出
- 把 `asset` 误当成运行成功态，会导致 Video / Audio 上传后逻辑错乱
- 把主结果对象复制存两份，会造成主次切换与结果列表不同步
- 让每个节点组件自己维护一套局部开关，会和 React Flow selection、flowStore 打架

## 14. 最终工程结论

### 14.1 真正应该统一的不是“节点 UI 长得像不像”
而是这 3 件事：
- 统一的内容模式 `contentMode`
- 统一的执行模式 `runtimeStatus`
- 统一的外层交互模式 `editorNodeId / canvasMode`

### 14.2 对 titanX v1 最重要的约束

- **节点本体状态放 node data**
- **画布交互状态放外层 UI state**
- **不要把瞬时 UI 开关写进持久化 schema**

### 14.3 这份文档可以直接服务的实现点

- React Flow 自定义节点渲染
- node data 结构补齐
- flowStore 增补 `editorNodeId` / `canvasMode`
- 点击缩略图、点击空白、切换主结果、进入 picking mode 的事件实现
