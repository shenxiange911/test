## 1. 当前代码基线判断

当前 `titanX/app/src` 仅有：
- `App.jsx`
- `components/Inspector.jsx`
- `components/StatusPanel.jsx`
- `main.jsx`
- 样式文件

现状不是完整业务系统，而是：
- 已有 React Flow 画布原型
- 已有节点创建交互壳
- 已有右侧 inspector 与状态面板壳
- 但没有业务数据主链

因此本轮映射原则是：
- **保留当前 3 个可复用 UI 文件**
- **新增最小数据主链模块**
- **不把业务逻辑继续塞回组件**

## 2. businessId allocator 映射

## 2.1 titanX/app 当前最适合承接 allocator 的文件

### 第一落点
- `app/src/App.jsx`

因为当前唯一新建节点入口就在：
- `handlePaneClick()`

### 但正式承载位置不推荐是 `App.jsx`
因为后续这些入口都会复用 allocator：
- 双击空白新建
- 输出点建下游节点
- 上传附件自动建节点
- 复制节点
- 模板实例化

所以 `App.jsx` 只能是**接入点**，不是**实现本体**。

## 2.2 allocator 最终应做成什么形态

### 推荐结论
**做成独立模块函数，不做 hook，不做 store 私有能力。**

### 推荐文件
- `app/src/lib/allocateBusinessId.js`

### 原因
- allocator 是纯函数，天然适合独立模块
- 不依赖 React 生命周期，不需要 hook
- 不应被任何未来 store 实现绑死
- 方便 nodeFactory、模板实例化、测试代码复用

### 推荐接口
```ts
allocateBusinessId(nodes, nodeType): string
```

### 推荐调用链
```text
App.jsx handlePaneClick
  -> nodeFactory.createNode(...)
  -> allocateBusinessId(nodes, nodeType)
```

## 3. upstreamResolver 映射

## 3.1 upstreamResolver 最适合挂在哪个模块

### 推荐文件
- `app/src/lib/upstreamResolver.js`

### 不推荐挂在组件层
- 不应挂在 `Inspector.jsx`
- 不应挂在 `StatusPanel.jsx`
- 不应直接写死在 `App.jsx`

原因：
- 它是图关系解析能力
- 不是 UI 展示能力
- 后续所有执行动作都要复用

## 3.2 输入输出接口怎么定

### 输入
```ts
resolveUpstream({
  nodeId,
  nodes,
  edges,
})
```

### 输出
```ts
{
  sourceNodeIds: string[],
  texts: string[],
  imageUrls: string[],
  videoUrls: string[],
  audioUrls: string[],
  resultRefs: Array<{
    sourceNodeId: string,
    sourceBusinessId: string,
    sourceNodeType: string,
    resultId?: string,
    role: 'text' | 'image' | 'video' | 'audio',
    url?: string,
    text?: string,
    isPrimary: boolean,
  }>
}
```

## 3.3 当前 app 里的调用位置

### 最合适调用层
- `app/src/lib/actions/runNodeAction.js`

### 不建议在这些位置调用
- 不建议在节点编辑输入变化时实时解析
- 不建议在 Inspector 渲染时顺手解析

### 推荐调用流程
```text
runNodeAction(nodeId)
  -> get current nodes / edges
  -> resolveUpstream({ nodeId, nodes, edges })
  -> compileNodeExecutionPlan(node, upstream)
```

## 4. BaseNodeShell / runtime / result / providerFamily 落点映射

## 4.1 BaseNodeShell 应落在哪些 types 或 state 层

### 推荐类型文件
- `app/src/lib/types.js`

### 推荐状态落点
- `node.data.record.shell`

### 原因
当前 React Flow node 外壳还是：
```js
{ id, type, data, position }
```

所以 titanX 业务数据最稳的做法是：
- 保留 React Flow node 外层结构
- 在 `node.data.record` 下承接业务数据主链

## 4.2 runtime 应落在哪些 types 或 state 层

### 推荐类型文件
- `app/src/lib/types.js`

### 推荐状态落点
- `node.data.record.runtime`

### 说明
当前 app 还没有独立 flowStore，最小实现时：
- `nodes` state 就是唯一真相
- runtime 回写必须直接更新对应 node.data.record.runtime

## 4.3 result 应落在哪些 types 或 state 层

### 推荐类型文件
- `app/src/lib/types.js`

### 推荐状态落点
- `node.data.record.result`

### 说明
主结果切换、下游消费、Inspector 展示都依赖它；不能只挂在 `StatusPanel`

## 4.4 providerFamily 应落在哪些 types 或 state 层

### 推荐类型文件
- `app/src/lib/types.js` 中的 `NodeCompiledConfig`

### 推荐状态落点
- 主归属：`node.data.record.compiled.providerFamily`
- 执行快照：`node.data.record.runtime.executedProviderFamily`

### 不推荐落点
- 不建议放到 `shell`
- 不建议在 `Inspector` 里直接给用户编辑

## 5. 节点执行后的 writeback 责任层

## 5.1 结论
**应由动作层统一负责回写 node data。**

### 推荐文件
- `app/src/lib/actions/runNodeAction.js`
- `app/src/lib/applyExecutionWriteback.js`

### 原因
如果让组件自己写回：
- `Inspector` 点按钮一套写法
- 右键菜单执行一套写法
- 批量执行再来一套写法
最后 runtime/result 会散。

## 5.2 推荐责任拆分

### `runNodeAction.js`
负责：
- 组织执行流程
- 调 resolver
- 调 compiler
- 调 adapter
- 拿到执行结果

### `applyExecutionWriteback.js`
负责：
- 统一把执行结果映射回 `node.data.record`
- 更新 `runtime`
- 更新 `result`
- 更新 `shell.canvasStatus`

### `persistRuntimeRecord.js`
负责：
- 把 writeback 后的结果落到 runtime record 持久化副本

## 7. 最小手动闭环在 titanX/app 里先接哪 3 个点

## 7.1 第 1 个点：节点创建主链

### 当前文件
- `app/src/App.jsx`

### 当前位置
- `handlePaneClick()`

### 先接内容
- `nodeFactory.createNode()`
- 内部统一分配 `businessId`
- 创建 `node.data.record = { shell, operator, runtime, result }`

### 为什么先接它
没有稳定 `businessId`，后面 runtime/result/record 都无法落稳。

## 7.2 第 2 个点：节点执行主链

### 当前可接 UI 入口
- `Inspector.jsx` 里的“运行节点”按钮

### 应新增动作层
- `runNodeAction(nodeId)`

### 先接内容
- `resolveUpstream()`
- `compileNodeExecutionPlan()`
- mock adapter 也行，但结果结构必须按正式 schema 写回

### 为什么第二个接它
manual-first 的真正主链不是“能创建节点”，而是“能执行并写回”

## 7.3 第 3 个点：writeback 展示链

### 当前展示文件
- `Inspector.jsx`
- `StatusPanel.jsx`

### 先接内容
- `node.data.record.runtime`
- `node.data.record.result`
- `shell.canvasStatus`

### 为什么第三个接它
只要 writeback 能被看见，manual-first 最小闭环就成立，后续再替换真实 adapter 成本最低。

## 8. titanX/app 空壳程度与最小补壳路径

## 8.1 当前空壳程度结论
`titanX/app` 当前属于：
**React Flow 原型壳已存在，业务数据主链基本未落。**

## 8.2 缺失程度表

| capability | current state |
| --- | --- |
| 画布显示 | 已有 |
| 节点创建 UI | 已有 |
| businessId allocator | 缺失 |
| nodeFactory | 缺失 |
| node record 数据壳 | 缺失 |
| upstreamResolver | 缺失 |
| compiled config | 缺失 |
| providerFamily routing | 缺失 |
| runtime writeback | 缺失 |
| result 主次结构 | 缺失 |
| runtime record persistence | 缺失 |

## 8.3 最小补壳路径

### Step 1：把创建逻辑收口成 nodeFactory
- 让所有新节点带 `businessId`
- 让所有新节点拥有统一 `record` 壳

### Step 2：把执行逻辑收口成 runNodeAction
- 先不追求真实 API 全接
- 但必须先有 `resolver -> compiled -> writeback` 骨架

### Step 3：把 Inspector / StatusPanel 改成读取 record
- UI 开始真正消费 `runtime/result`
- 形成最小手动闭环

### Step 4：再接真实 provider adapter
- 图片先接 `kie-jobs-image`
- 文本先接 `gemini-generate-content`
- 视频/音频后补

## 8.4 明确判断
当前 app 不是“只差一个 adapter”就能闭环。
它缺的是**中间那层数据主链骨架**。
如果不先补这层，直接接 provider，只会把执行逻辑继续写散在组件里。

## 9. 本轮最终映射结论

1. `businessId allocator` 最适合先从 `App.jsx > handlePaneClick` 接入，但正式实现必须抽成独立模块 `src/lib/allocateBusinessId.js`
2. allocator 应做成独立模块，不做 hook，不做 store 私有能力，因为它是纯函数、要跨入口复用、还要便于测试
3. `upstreamResolver` 最适合新建在 `src/lib/upstreamResolver.js`，输入固定 `{ nodeId, nodes, edges }`，输出统一 `ResolvedUpstream`
4. `BaseNodeShell / runtime / result / providerFamily` 应统一放进 `src/lib/types.js`，并挂到 `node.data.record` 的不同层级下
5. 节点执行后的 writeback 应由动作层负责，推荐 `runNodeAction.js + applyExecutionWriteback.js`，不要让组件直接改 node data
6. 当前可复用文件只有 `App.jsx`、`Inspector.jsx`、`StatusPanel.jsx`；数据主链相关文件基本都必须新建
7. 最小手动闭环先接 3 个点：`nodeFactory/createNode`、`runNodeAction`、`applyExecutionWriteback`
8. `titanX/app` 当前空壳程度较高：画布有了，但业务数据主链几乎还没开始；最小补壳路径应先补“node record + allocator + resolver + writeback”，再谈真实 provider 接入
