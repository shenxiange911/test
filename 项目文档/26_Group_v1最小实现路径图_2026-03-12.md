记录时间：2026-03-12
定位：把 `23_Group_Template实现边界总表_2026-03-12.md` 映射到 `titanX/app` 真实代码结构，给出可执行的最小实现路径，而不是停留在概念层。

## 1. 现状结论

当前 `titanX/app` 仍是一个 React Flow 原型壳：
- 只有单文件画布：`app/src/App.jsx:36`
- 只有 pane click 创建节点：`app/src/App.jsx:52`
- 没有多选状态
- 没有 group node
- 没有 group commands
- 没有 template serialize 模块

所以 Group v1 的正确推进方式不是：
- 直接往现有 `selected` 上硬叠 group 功能

而是：
- 先把“多选 + 组命令入口 + group node 壳”补出来
- 再把 template 作为 group 的导出能力接上

## 3. 混合模型下，state 怎么分层

## 3.1 parent-child 底层
放在 React Flow `nodes` 主 state。

也就是：
- group 本身是一个 node
- child 节点继续在 `nodes[]` 中
- child 与 group 的归属关系直接挂在 node 结构上

这是 Group v1 的底层承接层。

## 3.2 视觉框表现层
放在两部分：

### A. 常驻字段
挂到 group node data：
```ts
{
  groupId,
  title,
  color,
  layout,
  executable,
  templateable,
}
```

### B. 临时 UI 态
放在 `FlowCanvas` 顶层 state：
```ts
selection
selectedGroupId
toolbarMode
activeInspectorTarget
```

### 为什么不全塞 node data
因为这些状态里有大量瞬时 UI 态，例如：
- toolbar 开没开
- 当前是不是多选态
- 当前是不是 group 选中态

这些不该污染 node 持久字段。

## 3.3 template 业务层
不放在 group node data。

应放：
- `templateSerializer.js` 导出的标准结构
- 后续如果接模板库，再单独进入模板存储模块

正式结论：
- parent-child：React Flow nodes 层
- group metadata：group node data 层
- group / selection UI：FlowCanvas UI state 层
- template payload：独立 serializer 模块层

## 4. 关键交互与代码入口映射

## 4.1 整组移动

### 当前最适合的底层入口
- `onNodesChange`：`app/src/App.jsx:43-45`

原因：
- 无论是普通 node 拖动还是未来 group node 拖动，最终都会通过 node change 回流到这里
- 只要底层 parent-child 建好，group move 的底层承接口仍是这条链

### 需要新增的上层入口
- `GroupNode.jsx` 中的组框空白区命中逻辑

实现目标：
- 点组框空白处 -> 选中 group
- 拖组框空白处 -> 移动整个 group

## 4.2 解组

### 当前代码里没有任何现成入口
必须新建 command：
```ts
ungroupGroup(groupId)
```

### 最适合挂的位置
- P0：先挂 `app/src/flow/groupCommands.js`
- `App.jsx` 负责调用

### 需要做的事
- 找到 group node
- 找到所有 child nodes
- 将 child 坐标从 group 相对坐标还原为画布绝对坐标
- 清除 child 的 parent 归属字段
- 删除 group node

## 4.3 组内单节点操作

### 当前已有入口
- `onNodeClick={(e, node) => setSelected(node)}`：`app/src/App.jsx:98`

### Group v1 处理原则
- 这条链继续保留
- child node 依然能进 Inspector
- group 不能吞掉 child 的点击事件

这正是混合模型里“底层 parent-child + 组内仍可单节点操作”的代码化落点。

## 5. template 从 group 导出时，最小序列化结构挂在哪个模块

## 5.1 模块落点
最适合新建：
- `app/src/flow/templateSerializer.js`

并由：
- `app/src/flow/groupCommands.js`
调用它
- `App.jsx` 触发它

## 5.2 最小序列化结构
v1 建议固定为：

```ts
{
  templateId: string,
  name: string,
  sourceGroupId?: string,
  group: {
    title?: string,
    color?: string,
    layout?: 'grid' | 'horizontal',
  },
  nodes: Array<{
    localId: string,
    nodeType: string,
    data: unknown,
    position: { x: number, y: number },
  }>,
  edges: Array<{
    source: string,
    target: string,
    sourceHandle?: string,
    targetHandle?: string,
  }>,
}
```

### 这里的关键规则
- `position` 是 group 内相对坐标，不是画布绝对坐标
- 只导出 group 内成员
- 不导出 `selected / dragging / runtime` 之类实例态

## 6. 当前 reactflow 版本下，parentId / extent / selection 是否能直接承接

## 6.1 parent-child
结论：能承接，但开工前必须统一当前版本字段口径。

当前依赖仍是：
- `reactflow@^11.10.0`

并且已从实际 node_modules 确认：
- 内部存在 `parentNode || parentId` 的兼容逻辑
- `extent: 'parent'` 已有实现
- `selection` / `onSelectionChange` / 多选拖动链已存在

所以正式实现前要先做一件事：
- 确认项目内部统一写法，避免一半 `parentNode`、一半 `parentId`

正式建议：
- 文档层继续坚持“parent-child 底层模型”
- 代码层不要混用两套字段名

## 6.2 extent
结论：当前版本可承接，但不适合进入 P0 主路径。

原因：
- `extent` 更适合 child movement constraint
- Group v1 的第一目标不是限制 child 不能越界，而是先打通：
  - 打组
  - 整组移动
  - 解组
  - 组内单节点继续操作

正式建议：
- P0 不重压 extent
- P1 再补 child in parent 的边界限制

## 6.3 selection
结论：React Flow 的 selection 能直接承接，但当前 app 还没把它真正接出来。

这意味着：
- 不是库不支持
- 是 app 层还缺：
  - `selection`
  - `selectedGroupId`
  - `onSelectionChange`
  - selection toolbar

## 8. 当前 app 如果太空壳，应该从哪个最小壳开始补

正式判断：是，当前 `titanX/app` 在 group/template 这条线上属于明显空壳。

### 缺失程度
- 还没有多选命令入口
- 还没有 group node 壳
- 还没有 group command 层
- 还没有 template serializer

### 最小补壳路径

#### 第一步：`SelectionToolbar.jsx`
理由：
- 没有它，就没有“从多选到打组”的明确入口

#### 第二步：`GroupNode.jsx`
理由：
- 没有它，就无法承接“parent-child 为底层 + 视觉框表现 + group metadata”这条混合模型

#### 第三步：`groupCommands.js`
理由：
- 打组 / 解组 / 导出模板不能散在按钮 onClick 里

#### 第四步：`templateSerializer.js`
理由：
- 否则 template 会和 group 当前实例态混在一起

正式最小路径图如下：

```text
App.jsx
  -> 接出 selection / selectedGroupId / activeInspectorTarget
  -> 渲染 SelectionToolbar
  -> 注册 GroupNode
  -> 调用 groupCommands

SelectionToolbar.jsx
  -> 触发 groupSelection()

GroupNode.jsx
  -> 承接组框显示、整组选中、整组拖动命中区

groupCommands.js
  -> groupSelection()
  -> ungroupGroup()
  -> createTemplateFromGroup()

templateSerializer.js
  -> serializeGroupToTemplate()
```

## 9. 新增补强：真实代码入口 + 行号 + 为什么是这里

这里是当前所有顶层状态的唯一集中区。

为什么是这里：
- `selection / toolbarMode / activeInspectorTarget / lastTemplateDraft` 都应该在这一段向外扩

- **P0 临时承接**：`selection`、`toolbarMode`、`activeInspectorTarget`

为什么是这里：

### C. `app/src/App.jsx:52-68` — `handlePaneClick`
为什么是这里：
- 这是当前唯一“向图中新增普通节点”的入口
- 未来 Group 线会和这里发生点击语义冲突，所以必须明确它继续只负责普通建节点

结论：
- 这是普通节点创建入口，应继续复用

### D. `app/src/App.jsx:73-79` — 顶栏 `canvas-toolbar`
为什么是这里：

标注：
- **P0 临时承接**：mode 文案、轻量入口按钮
- P1 再拆出独立 toolbar 组件

### E. `app/src/App.jsx:81-91` — `node-toolbar`
为什么是这里：
- 当前已经是主要操作按钮区
- P0 最低成本方案，是在这一区域临时插入“打组 / 解组 / 创建模板”入口

标注：

### F. `app/src/App.jsx:94-104` — `ReactFlow` 组件本体
为什么是这里：
- `onEdgesChange`、`onSelectionChange`、`nodeTypes`、未来 `GroupNode` 注册，都只能从这里接入

这是：
- selection 的真正挂载入口

### G. `app/src/App.jsx:98` — `onNodeClick`

为什么是这里：

## 9.7 新增补强：未来最可能冲突的接缝在哪里

### A. Group 线 vs 输出点建节点线
冲突点：
- `handlePaneClick` 当前只负责空白区创建普通节点：`app/src/App.jsx:52-68`
- 未来如果同时引入多选态 / group 选中态 / template 态，空白区点击语义会变复杂

最容易冲突的接缝：
- 当前准备创建节点时点击 pane
- 当前处于多选态时点击 pane
- 当前 group 选中时点击 pane

结论：
- `handlePaneClick` 未来必须受 `toolbarMode` + `pendingNodeType` 双重约束
- 这是 Group 线和普通建节点线的第一冲突面

### B. Group 线 vs 状态机线
这里的“状态机线”指容器级状态流：
- `selected`
- `pendingNodeType`
- future `selection`
- future `toolbarMode`
- future `activeInspectorTarget`

最容易冲突的接缝：
- 单节点选中 -> 多选态 -> group 选中态 的切换规则
- `onNodeClick` 是否覆盖 `onSelectionChange`
- Inspector 是看 node、看 group，还是看 selection

结论：
- `selected` 不能继续作为唯一选择状态
- 必须升级成 target router，否则 group/selection 很快互相打架

### C. Group 线 vs 数据主链线
这里的数据主链指：
- `nodes`
- `edges`
- future template draft

最容易冲突的接缝：
- `groupSelection()` 改写 child 坐标
- `ungroupGroup()` 还原 child 坐标
- `serializeGroupToTemplate()` 又要读取相对坐标

这三条链共用同一批数据，但读写意图不同：
- 画布编辑态要绝对+相对混用
- 解组要回绝对坐标
- 模板要只拿相对坐标净化输出

结论：
- `groupCommands.js` 和 `templateSerializer.js` 的边界必须非常清楚
- 最容易出错的不是 UI，而是坐标口径

### D. Group 线 vs 连线 / 边系统
当前 edges 还很简单：`app/src/App.jsx:31-34`、`app/src/App.jsx:38`
但 group 上来后，最容易冲突的是：
- group 内 child 连外部节点
- 解组后 edge 是否还能稳定指向原 child
- template 导出时哪些 edge 算组内边，哪些算跨组边

结论：
- 现在 edges 还没复杂，但 group/template 一接进来，这条线会迅速变成高风险接缝
- 所以 `collectGroupMembers(groupId, nodes, edges)` 必须尽早稳定

## 10. 最终回答 7 个必答问题

## 10.1 titanX/app 现有代码里，group 相关逻辑最适合从哪些文件开始接
- 先从 `app/src/App.jsx` 接主状态与主命令入口
- 配套新增：`SelectionToolbar.jsx`、`GroupNode.jsx`、`groupCommands.js`、`templateSerializer.js`

## 10.2 如果采用混合模型，state 分别放哪一层
- parent-child：React Flow `nodes` 层
- 视觉框常驻字段：group node data 层
- 组级临时 UI：`FlowCanvas` 顶层 state
- template 导出结构：独立 serializer 模块

## 10.3 整组移动 / 解组 / 组内单节点操作，各自对应哪些代码入口
- 整组移动：底层 `onNodesChange` + `GroupNode` 命中区
- 解组：`ungroupGroup(groupId)` command
- 组内单节点操作：继续走 `onNodeClick`

## 10.4 template 从 group 导出时，最小序列化结构应该挂在哪个模块
- 挂在 `app/src/flow/templateSerializer.js`

## 10.5 当前 reactflow 版本下，parentId / extent / selection 相关 API 是否能直接承接
- parent-child：能承接，但字段口径要先统一版本
- extent：可用，但 P0 不重度使用
- selection：API 可承接，当前只是 app 还没接状态

## 10.6 哪些功能适合 P0 直接做，哪些必须后推
- P0：多选、打组、整组移动、解组、组内单节点操作、导出最小 template
- 后推：自动吸附、嵌套 group、高级布局、完整模板库、完整组执行编排器

## 10.7 如果 titanX/app 当前太空壳，要明确指出缺失程度，并给出“从哪个最小壳开始补”的路径
- 当前确实太空壳
- 最小补壳路径：`SelectionToolbar.jsx` -> `GroupNode.jsx` -> `groupCommands.js` -> `templateSerializer.js`

## 11. 一句话冻结结论

Group v1 在 titanX/app 里，不该从“先想模板”开始，而应从“先把多选打组这条真实画布命令链补出来”开始；模板只是 group 的导出层，不是 group 的替代物。
