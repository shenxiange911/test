# titanX 调研过程｜输出点建节点 P0 落地顺序

记录时间：2026-03-12
范围：基于 `25_输出点建节点代码改造清单_2026-03-12.md`，把“输出点建节点”继续拆成真正可排期的 P0 文件落地顺序与依赖关系，不直接改代码。

---

## 1. 当前代码起点

当前 `titanX/app` 仍是单文件 React Flow 原型：
- `app/src/App.jsx` 同时承担画布 state、节点创建、ReactFlow 渲染
- `edges` 还是只读 state：`app/src/App.jsx:38`
- 只有 `pendingNodeType`，没有 `pendingSource / creationMenu`
- 没有 custom node、没有 creation menu、没有 edge factory

这说明：
- P0 落地顺序不能按“概念模块优雅度”排
- 必须按“先把链路撑起来，再抽壳，再补事件，再闭环验证”排

---

## 2. P0 需要新增/修改的完整文件清单

### 2.1 必改现有文件
1. `app/src/App.jsx`
2. `app/src/App.css`

### 2.2 P0 必建新文件
3. `app/src/components/nodes/TitanNode.jsx`
4. `app/src/components/CreationMenu.jsx`

### 2.3 P0 可选但不必立刻新建的文件
5. `app/src/flow/nodeFactory.js`
6. `app/src/flow/edgeFactory.js`
7. `app/src/flow/creationPosition.js`

当前建议：
- 这 3 个 flow 模块先不要一开始就建
- 先临时放 `App.jsx`，待 P0 主链打通后再判断是否值得立刻抽

原因：
- 当前 app 过薄
- 如果还没把 TitanNode / CreationMenu / setEdges 接通，就先抽模块，容易做出一堆“形式上很整齐、实际上没跑通”的空壳

## 3. 每个文件的职责边界

## 3.1 `app/src/App.jsx`
P0 里它仍是总调度器，职责边界应冻结为：
- 维护 nodes / edges 主 state
- 维护 `pendingNodeType`
- 维护 `pendingSource / creationMenu`
- 维护 `screenToFlowPosition()` 调用点
- 承接 `TitanNode` 抛出来的 output click 事件
- 承接 `ReactFlow onConnectStart / onConnectEnd`
- 调度 node 创建与 edge 创建
- 渲染 `CreationMenu`

不应该让它做的：
- 直接承载菜单 UI 细节
- 直接渲染节点内部输出点 DOM
- 在 JSX 内部散写一长串 node/edge 构造逻辑

## 3.2 `app/src/App.css`
P0 里职责边界：
- 给 `.canvas-shell` 提供 overlay 定位上下文
- 提供 `CreationMenu` 的基础样式
- 提供 `TitanNode` 的最低样式

不应该做的：
- 承担业务逻辑
- 依赖运行时状态机字段命名过深

## 3.3 `app/src/components/nodes/TitanNode.jsx`
P0 里职责边界：
- 只负责节点视觉壳和输出点交互入口
- 点击输出点时，向上抛出 `onOutputClick`
- 若允许拖线，也在这里承接 source handle 视觉与交互

不应该做的：
- 直接写 `setNodes`
- 直接写 `setEdges`
- 直接算 `screenToFlowPosition()`
- 直接决定 creation menu 打开什么类型

## 3.4 `app/src/components/CreationMenu.jsx`
P0 里职责边界：
- 只负责菜单显示
- 只根据 `anchorClient` 定位
- 只把“选中了哪个 type”回调给上层
- 只负责关闭和轻量空白 dismiss

不应该做的：
- 直接创建节点
- 直接创建边
- 直接访问 React Flow instance

## 3.5 `nodeFactory / edgeFactory / creationPosition`
P0 阶段如果不抽文件，职责依然要先在逻辑层冻结：
- `createCanvasNode()`：只管产出 node 对象
- `createCanvasEdge()`：只管产出 edge 对象
- `resolveCreationPosition()`：只管从 `anchorFlow` 计算目标位

这 3 个函数即便先放在 `App.jsx`，也要保持“像独立模块那样写”，避免后面抽不动。

## 4. 文件之间的依赖顺序

## 4.1 最小依赖链

```text
App.jsx
  -> TitanNode.jsx
  -> CreationMenu.jsx
  -> (临时内联) createCanvasNode / createCanvasEdge / resolveCreationPosition
```

因为当前没有 store，也没有 flow 目录落地结构，所以：
- `App.jsx` 必须先成为统一容器
- `TitanNode.jsx` 与 `CreationMenu.jsx` 都依赖 `App.jsx` 提供回调和状态
- 两者之间不应互相依赖

## 4.2 更精确的先后关系

### 第一层：状态先行
- `App.jsx` 必须先能维护 `setEdges + pendingSource + creationMenu`

否则：
- `TitanNode.jsx` 抛出的事件无处接
- `CreationMenu.jsx` 无状态可读

### 第二层：入口先行
- `TitanNode.jsx` 必须先于 `CreationMenu` 真正进入主链

原因：
- 菜单是被 output trigger 打开的
- 没有 output trigger，菜单组件只能悬空存在

### 第三层：渲染后置
- `CreationMenu.jsx` 在 `App.jsx` 和 `TitanNode.jsx` 都连通后再接入最合理

### 第四层：位置策略最后补
- `resolveCreationPosition()` 可先做最简单版，放到最后一批补

---

## 5. 哪一步必须先于哪一步

### 5.1 必须先把 `edges` 改为可写，才能谈自动建 edge
当前 `App.jsx:38` 是：
```ts
const [edges] = useState(initialEdges)
```

这一步不先改：
- 后面所有 output -> create edge 都没法落地

所以顺序上：
- `setEdges` 改造必须早于 `TitanNode` 接入主链

### 5.2 必须先有 `pendingSource / creationMenu`，才能接 `CreationMenu`
没有这两组状态：
- 菜单无法知道它服务哪一个 source
- 菜单无法知道显示在哪
- 菜单无法知道允许创建什么类型

所以顺序上：
- `App.jsx` 状态改造必须早于 `CreationMenu.jsx` 接入

### 5.3 必须先有 `TitanNode`，才能真正支持“点击输出点弹菜单”
在当前默认节点体系里：
- 没有业务 output trigger

所以顺序上：
- `TitanNode.jsx` 必须早于“输出点点击打开菜单”这条交互闭环

### 5.4 必须先定义 `handleCreationMenuSelect()`，才能让菜单不沦为纯 UI
否则菜单只能显示，不能闭环创建。

所以顺序上：
- `handleCreationMenuSelect()` 必须早于 `CreationMenu` 正式接入

### 5.5 必须先冻结 `screenToFlowPosition()` 调用时机，才能避免落点漂移
这个调用点必须在“打开菜单时”，不是“确认菜单时”。

所以顺序上：
- `openCreationMenuFromOutput()` 设计必须早于 node/edge 创建闭环

## 6. 哪些内容先临时放 `App.jsx`，哪些值得立刻抽模块

## 6.1 先临时放 `App.jsx`
P0 建议先临时放这里：
- `createCanvasNode()`
- `createCanvasEdge()`
- `resolveCreationPosition()` 的最简单版
- `openCreationMenuFromOutput()`
- `handleConnectStart()`
- `handleConnectEnd()`
- `handleCreationMenuSelect()`

原因：
- 当前 app 太薄
- 这些函数都紧贴 `nodes / edges / selected / screenToFlowPosition()`
- 先就地打通主链，最容易验证

## 6.2 值得立刻抽模块
P0 建议立刻抽出去的只有两类：

### A. `TitanNode.jsx`
因为它本质是新的交互入口壳，不应和 `App.jsx` JSX 混在一起。

### B. `CreationMenu.jsx`
因为它本质是 overlay UI，不应和 `App.jsx` 内联 DOM 混在一起。

## 6.3 暂不建议立刻抽模块
以下 3 个可等 P0 跑通再抽：
- `nodeFactory.js`
- `edgeFactory.js`
- `creationPosition.js`

原因：
- 它们更多是“结构优化”，不是当前主链的第一阻塞点

## 7. `TitanNode.jsx / CreationMenu.jsx / App.jsx` 三者之间的事件流

## 7.1 点击输出点链路

```text
TitanNode.jsx
  -> onOutputClick({ nodeId, handleId, clientX, clientY })
  -> App.jsx.openCreationMenuFromOutput(...)
  -> App.jsx.setCreationMenu({ anchorClient, anchorFlow, pendingSource, allowedTypes })
  -> CreationMenu.jsx 渲染
  -> 用户选择目标类型
  -> CreationMenu.jsx.onSelect(type)
  -> App.jsx.handleCreationMenuSelect(type)
  -> App.jsx.createCanvasNode(...)
  -> App.jsx.createCanvasEdge(...)
  -> App.jsx.closeCreationMenu()
```

## 7.2 拖出输出点链路

```text
ReactFlow onConnectStart
  -> App.jsx.handleConnectStart(...)
  -> setPendingSource(...)

ReactFlow onConnectEnd
  -> App.jsx.handleConnectEnd(event, connectionState)
  -> 若落空白且无有效 target
  -> App.jsx.openCreationMenuFromOutput(... mode='from-output-drag')
  -> CreationMenu.jsx
  -> onSelect(type)
  -> App.jsx.handleCreationMenuSelect(type)
  -> create node + edge
```

## 7.3 三者之间的边界冻结
- `TitanNode.jsx` 只负责抛事件，不负责创建
- `CreationMenu.jsx` 只负责选型，不负责创建
- `App.jsx` 是 P0 唯一创建调度器

---

## 8. 哪些点会和状态机线、数据链线冲突，需要预留接口

## 8.1 会和状态机线冲突的点
这里的“状态机线”指后续可能补的：
- `canvasMode`
- `creating-from-output`
- `focus-pick`
- `grouping`
- `editing`

输出点建节点 P0 当前还没正式上状态机，但会直接碰到这些问题：
- 打开 creation menu 时，是否需要锁住其他 mode
- 菜单未关闭时，pane click 是继续创建还是关闭菜单
- output drag 未完成时，能否同时进入别的 picking mode

因此 P0 应预留：
```ts
creationMenu.mode
pendingSource.mode
```

并建议未来兼容：
```ts
canvasMode?: 'idle' | 'creating-from-output'
```

P0 先不完整实现状态机，但要留字段，不然后面会返工。

## 8.2 会和数据链线冲突的点
这里的“数据链线”指后续真正的：
- source handle -> target node 输入绑定
- 上下游数据引用
- 节点 family 白名单

P0 当前只做“自动建 edge”，但会碰到这些边界：
- 哪些源节点能创建哪些目标类型
- sourceHandleId 是否要进入 edge data
- target node 初始化时是否要记来源信息

因此 P0 应预留接口：

### A. `allowedTypes`
不要写死在菜单组件里，应由 `App.jsx` 或未来规则函数注入

### B. `sourceMeta`
`createCanvasNode()` 最好允许接收：
```ts
sourceMeta?: {
  sourceNodeId,
  sourceHandleId,
}
```

### C. `edge.data`
`createCanvasEdge()` 最好预留：
```ts
data?: {
  creationMode?: 'from-output-click' | 'from-output-drag'
}
```

这样以后状态机线、数据链线接进来时，不至于重写主链。

## 9. 完整 P0 完成后，最小可验证行为是什么

P0 完成后，至少应能稳定验证下面这 5 条：

### 9.1 点击输出点弹菜单
- 点击某个 titan node 的输出点
- 菜单出现在鼠标附近

### 9.2 选择目标类型后自动创建 node + edge
- 在菜单里选一个类型
- 画布里新增目标节点
- 同时新增从 source 到 target 的 edge

### 9.3 缩放和平移后仍创建在正确位置
- 先 zoom / pan 画布
- 再从输出点打开菜单
- 新节点仍出现在正确的 flow 坐标附近

### 9.4 拖出输出点到空白区也能走同一条菜单链
- 从输出点拖线到 pane 空白
- 弹同一个 creation menu
- 选择类型后自动建 node + edge

### 9.5 旧的 pane click 创建链不被破坏
- 顶部工具条 + 空白处点击创建仍然可用
- 不因输出点主链接入而回归坏掉

这 5 条全部成立，才算 P0 真正闭环。

## 10. 推荐的 P0 文件落地顺序（过程稿结论）

### Phase 1：先改容器，不加新组件
1. 改 `App.jsx`：让 `edges` 可写
2. 改 `App.jsx`：加 `pendingSource / creationMenu`
3. 改 `App.jsx`：加 `openCreationMenuFromOutput / closeCreationMenu`
4. 改 `App.jsx`：加 `createCanvasNode / createCanvasEdge / handleCreationMenuSelect`

### Phase 2：补 UI 壳
5. 新建 `TitanNode.jsx`
6. 改 `App.jsx`：注册 `nodeTypes`，让现有节点切到 TitanNode
7. 新建 `CreationMenu.jsx`
8. 改 `App.jsx`：把 `creationMenu` 真正渲染出来
9. 改 `App.css`：补 overlay 和 TitanNode 样式

### Phase 3：补拖线入口
10. 改 `App.jsx`：加 `onConnectStart / onConnectEnd`
11. 改 `TitanNode.jsx`：让输出点既能 click，也能作为 source handle

### Phase 4：收尾验证
12. 验证点击输出点链
13. 验证拖空白链
14. 验证 zoom/pan 后坐标
15. 验证旧 pane click 创建未回归

这个顺序的核心思想是：
- 先让容器有能力
- 再给它真实入口和真实菜单
- 最后再把拖线这条分支接上
