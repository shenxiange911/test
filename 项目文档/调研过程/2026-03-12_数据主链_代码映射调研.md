# titanX 调研过程：数据主链代码映射

记录时间：2026-03-12
基准文档：`22_数据主链与业务ID总表_2026-03-12.md`
调研范围：仅看 `titanX/app` 现有代码，判断数据主链应该挂到哪里，不改代码

---

## 1. 当前 app 代码现实状态

当前 `titanX/app/src` 只有 6 个文件：
- `App.jsx`
- `components/Inspector.jsx`
- `components/StatusPanel.jsx`
- `main.jsx`
- `App.css`
- `index.css`

其中真正承载画布行为的只有：
- `App.jsx`

当前已有：
- React Flow 受控 `nodes`
- 固定 `edges`
- pane click 创建节点
- 选中节点后 inspector 显示基础字段

当前没有：
- `businessId allocator`
- `nodeFactory`
- `edgeFactory`
- `upstreamResolver`
- 类型层 `shell/operator/compiled/runtime/result`
- provider routing
- 执行动作
- 结果写回

结论：
- `titanX/app` 当前仍是**画布原型壳**，不是业务数据链实现体
- 但它并不是完全不可承接，反而因为文件少，适合先把主链骨架插进去

---

## 2. 逐项映射判断

## 2.1 哪些文件最适合承接 businessId allocator

### 当前最直接挂点
- `App.jsx`

原因：
- 当前 `handlePaneClick()` 就是唯一新建节点入口
- `nodes` 状态也只在 `App.jsx` 维护
- 如果本轮只是最小补壳，allocator 接入点一定先从这里切入

### 但长期不应直接写死在 `App.jsx`
因为：
- 双击建节点、上传建节点、输出点建节点、复制节点、模板实例化都要复用
- allocator 是纯业务能力，不应耦合在某个具体 UI 事件里

### 最终推荐
新建独立模块：
- `app/src/lib/allocateBusinessId.js`

并由未来的：
- `nodeFactory`
- 或 `flowActions/createNode`
统一调用

## 2.2 allocator 应该做成独立模块、hook、还是 store 内能力

### 结论
**应该先做成独立模块函数，不做 hook，不做 store 内私有能力。**

### 原因
1. allocator 是纯函数
- 输入：当前 nodes + nodeType
- 输出：新 businessId
- 不依赖 React 生命周期
- 不依赖副作用

2. 做成 hook 的坏处
- 只能在 React 环境里用
- 后续 nodeFactory、模板实例化、批量导入、测试代码复用都不方便

3. 做成 store 内私有能力的坏处
- 未来如果要做撤销/重做、导入、离线校验，很容易被 store 实现绑死
- 不利于单元测试

### 推荐挂法
- 独立模块：`src/lib/allocateBusinessId.js`
- 调用者：`nodeFactory` / `createNodeAction`

---

## 3. upstreamResolver 最适合挂在哪个模块

### 当前最合适的模块位置
新建：
- `app/src/lib/upstreamResolver.js`

### 不建议挂在这些地方
- `Inspector.jsx`：这是 UI 层，不该知道图关系解析
- `StatusPanel.jsx`：只是展示层
- `App.jsx`：可以临时调用，但不应承载解析逻辑本体

### 输入接口建议
```ts
resolveUpstream({
  nodeId,
  nodes,
  edges,
})
```

### 输出接口建议
```ts
{
  sourceNodeIds: string[],
  texts: string[],
  imageUrls: string[],
  videoUrls: string[],
  audioUrls: string[],
  resultRefs: []
}
```

### 当前 app 代码接入点
未来最适合在“执行节点动作”里调用，而不是在节点编辑时每次实时算。
也就是：
- `runNodeAction(nodeId)`
  - 先读 nodes/edges
  - 再 `resolveUpstream()`
  - 再 `compileNodeExecutionPlan()`

---

## 4. BaseNodeShell / runtime / result / providerFamily 各自适合落点

## 4.1 BaseNodeShell
### 推荐落点
新建：
- `app/src/lib/types.js` 或 `app/src/lib/nodeTypes.js`

### 原因
当前 `App.jsx` 里的 node 还是 React Flow 原生结构：
```js
{ id, type, data, position }
```

而 titanX 需要的是 node.data 里面有稳定业务壳。
所以更合理的做法是：
- React Flow 节点外层保持 RF 格式
- `node.data.record` 或 `node.data.nodeRecord` 放 titanX 业务壳

## 4.2 runtime
### 推荐落点
- 类型定义：`src/lib/types.js`
- 状态真相：先仍放在 `App.jsx` 持有的 `nodes` 里

### 原因
当前没有独立 flowStore，所以最小实现必须承认：
- `nodes` 就是当前 app 的唯一状态真相
- runtime 写回应先直接修改对应 node.data

## 4.3 result
### 推荐落点
- 类型定义：`src/lib/types.js`
- 实际数据：`node.data.record.result`

### 原因
主结果切换、下游引用都依赖它；它不能只放在 `StatusPanel`

## 4.4 providerFamily
### 推荐落点
- 类型定义：`compiled config` 类型里
- 运行时快照：`node.data.record.runtime.executedProviderFamily`

### 不推荐落点
- 不推荐直接放在 `BaseNodeShell`
- 不推荐做成 inspector 里的可编辑 input

---

## 5. 节点执行后的 writeback 应由哪个层负责

### 结论
应该由**动作层 / execution action 层**负责，而不是由组件直接写。

### 当前 app 现实
当前还没有动作层。
所以第一版最小补壳建议新建：
- `app/src/lib/actions/runNodeAction.js`

### 该层职责
1. 读取当前节点
2. 调 `resolveUpstream()`
3. 调 `compileNodeExecutionPlan()`
4. 选 provider adapter
5. 拿到执行结果
6. 统一回写 node data
7. 触发 `persistRuntimeRecord()`

### 为什么不能让组件自己写
如果把 writeback 写进：
- `Inspector.jsx` 的按钮点击
- `StatusPanel.jsx` 的 run button

后面会出现：
- 右键执行、快捷栏执行、批量执行各写一套
- runtime/result 字段更新不一致

---

## 6. 哪些现有文件可复用，哪些必须新建

## 6.1 可继续复用

### `App.jsx`
可复用为：
- 现阶段 flow 容器
- `nodes/edges` 主状态持有者
- create/run actions 的挂载入口

### `Inspector.jsx`
可复用为：
- operator-facing 字段编辑面板原型
- 当前节点 record 的可视化入口

### `StatusPanel.jsx`
可复用为：
- 当前运行态摘要
- 最近一次执行状态摘要

## 6.2 必须新建

### 类型与数据层
- `app/src/lib/types.js`

### 节点创建主链
- `app/src/lib/allocateBusinessId.js`
- `app/src/lib/nodeFactory.js`

### 上游解析与编译主链
- `app/src/lib/upstreamResolver.js`
- `app/src/lib/compileNodeExecutionPlan.js`
- `app/src/lib/modelRegistry.js`

### 执行与写回主链
- `app/src/lib/actions/runNodeAction.js`
- `app/src/lib/applyExecutionWriteback.js`
- `app/src/lib/persistRuntimeRecord.js`

### 如继续最小化，也可先合并成 3 个核心新文件
如果不想一次新建太多，最小也至少要有：
- `src/lib/nodeFactory.js`
- `src/lib/upstreamResolver.js`
- `src/lib/actions/runNodeAction.js`

## 7. 最小手动闭环在 titanX/app 里应该先接哪 3 个点

### 第 1 个点：节点创建
当前入口：`App.jsx > handlePaneClick`
先接：
- `nodeFactory.createNode()`
- 内部调用 `allocateBusinessId()`

价值：
- 先把业务 ID 主链建立起来
- 后面所有执行和 writeback 都有稳定 node identity

### 第 2 个点：节点执行
当前入口还不存在，应新增：
- `runNodeAction(nodeId)`

最容易从 `Inspector.jsx` 的“运行节点”按钮接进去。

价值：
- 先把“不是点按钮假动作，而是进入 compiled/runtime/result 主链”打通

### 第 3 个点：writeback
执行成功后先回写：
- `node.data.record.runtime`
- `node.data.record.result`

并同步刷新：
- `StatusPanel.jsx`
- `Inspector.jsx`

价值：
- 一旦这步接通，manual-first 的可见闭环就成立了

## 8. titanX/app 当前空壳程度评估

### 结论
`titanX/app` 当前属于：**低代码量原型壳，画布存在，业务数据主链基本未落。**

### 缺失程度分解
| 层 | 当前状态 |
| --- | --- |
| React Flow 画布 | 已有最小壳 |
| 节点创建 | 只有 UI 创建，没有业务 ID |
| 节点数据壳 | 未建立 |
| upstream 解析 | 未建立 |
| compiled config | 未建立 |
| providerFamily 路由 | 未建立 |
| runtime 写回 | 未建立 |
| result 主次结构 | 未建立 |
| 持久化 record | 未建立 |

### 判断
不是“有一半实现了只差接 API”，而是：
- 画布层已有
- 业务数据主链几乎全缺

所以这轮不能假设只要补一个 adapter 就能闭环。

---

## 9. 最小补壳路径

### Phase A：先把节点从 RF node 升级为 titanX node record
- 加 `nodeFactory`
- 加 `allocateBusinessId`
- 所有新建节点都写入 `node.data.record`

### Phase B：加最小执行主链
- `runNodeAction(nodeId)`
- `resolveUpstream()`
- `compileNodeExecutionPlan()`
- mock adapter 也行，但 writeback 结构必须按正式 schema 来

### Phase C：加最小 writeback 和可视化
- 成功后回写 `runtime/result`
- `Inspector` 显示 record
- `StatusPanel` 显示最后一次运行状态

### Phase D：再替换成真实 provider adapter
- 这一步才能接 KIE / Gemini / ElevenLabs
- 否则只会把 provider 调用接到一个不稳定的壳上

---

## 10. 本轮压缩结论

1. `businessId allocator` 最终应做成独立模块，由 `nodeFactory` 调用；当前 app 里最先接入点是 `App.jsx > handlePaneClick`
2. `upstreamResolver` 应新建为 `src/lib/upstreamResolver.js`，输入固定为 `{ nodeId, nodes, edges }`，输出统一 `ResolvedUpstream`
3. `BaseNodeShell / runtime / result` 不应散在组件里，应该进入 `src/lib/types.js`，并挂在 `node.data.record` 下
4. 节点执行后的 writeback 应由动作层 `runNodeAction` 统一负责，不应由 `Inspector` 或 `StatusPanel` 自己写
5. 可复用文件只有 `App.jsx`、`Inspector.jsx`、`StatusPanel.jsx`；真正的数据主链文件基本都必须新建
6. titanX/app 目前不是“主链只差最后一步”，而是“画布有了，业务数据主链基本还没开始”
7. 最小闭环先接 3 个点：`nodeFactory/createNode`、`runNodeAction`、`applyExecutionWriteback`
