记录时间：2026-03-12

已核对：
- `app/src/App.jsx`

- `app/src/components/Inspector.jsx`

结论先说：

- 只有“工具条选类型 -> 点击空白 pane 创建节点”这条链
- 还没有自定义节点组件、没有自定义 handle、没有 edge 写入函数、没有菜单组件、没有 flow store、没有 node factory / edge factory

这意味着：
- 本轮不能只说“在哪个现有函数里补一补”
- 必须明确指出：当前代码缺失到什么程度，以及最小需要补哪些壳

## 2. 现有结构怎么分层

### 2.1 `app/src/App.jsx` 现在承担了 4 件事
当前 `FlowCanvas()` 里同时承担：
- 画布 state
- 节点创建逻辑
- 顶部节点库工具条
- 整个 ReactFlow 渲染

对应位置：
- 节点/边 state：`App.jsx:37-40`
- 坐标换算实例：`App.jsx:41`
- pane 点击创建逻辑：`App.jsx:52-68`
- ReactFlow 渲染：`App.jsx:93-104`

问题：
- 这适合 demo，不适合加“输出点附近创建下游节点”这类第二条创建主链
- 如果继续把输出创建、菜单、边、避让都堆进 `FlowCanvas()`，会很快变成一坨大函数

### 2.2 现有组件里没有一个能直接承载输出点菜单入口
当前组件只有：
- `FlowCanvas`：总容器
- `Inspector`：右侧检查面板
- `StatusPanel`：右下状态面板

现状意味着：
- 没有 custom node component，因此没有地方挂“点击输出点”事件
- 现有节点都是 React Flow 默认 `input/default/output` 节点
- 默认节点只有默认 handle，不适合承载 titanX 的业务菜单入口

结论：
- 菜单入口不能挂在 `Inspector` 或 `StatusPanel`
- 也不适合继续挂在顶部 `node-toolbar`
- 真正的入口要落在“新建 custom node component + 在 node 内部暴露输出点事件”这条线上

## 3. 当前能复用的现有逻辑

### 3.1 `screenToFlowPosition()` 已经接好了
现有位置：`App.jsx:41`

现状价值：
- 说明 ReactFlowProvider 和 `useReactFlow()` 已经就位
- 现有代码已经验证过 pane click -> flow position 这一步

可直接复用：
- 输出拖拽落空白区时的落点换算
- 点击输出点时的默认创建锚点换算

### 3.2 `handlePaneClick()` 可作为“节点创建工厂”的最早雏形
现有位置：`App.jsx:52-68`

它现在做的是：
- 把 `pendingNodeType` 变成一个实际 node
- 直接 `setNodes([...prev, newNode])`

问题：
- 它现在是“pane click 专用函数”
- 没有抽象成可复用的 `createNode()`
- 没有 id allocator
- 没有 edge 逻辑
- 没有偏移/避让

结论：
- 不能继续拿 `handlePaneClick()` 当最终实现
- 但可以把里面“从 node type 组装 node 对象”的部分抽成 node factory 的第一版

### 3.3 `canvas-shell` 是天然的菜单 overlay 挂载位
现有 DOM：`App.jsx:71-106`
现有样式：`App.css:9-14`

为什么它适合：
- 它包住了 toolbar、node-toolbar、canvas
- 可以加 `position: relative`
- 可以在 ReactFlow 同级挂一个绝对定位菜单层
- 菜单不必跟 viewport 缩放

结论：
- 现有代码里菜单入口最适合挂在 `FlowCanvas` 内部新增 overlay 组件
- 而不是挂在 ReactFlow children 内部

## 4. 缺失到什么程度

### 4.1 缺 custom node component
当前 `ReactFlow` 没传 `nodeTypes`，所以没有 titanX 自定义节点。

直接后果：
- 没有业务化输出点
- 没有地方区分“点击节点本体”和“点击输出点”
- 没有 source handle 粒度的事件入口

### 4.2 缺 edges 可写链路
当前：
- `const [edges] = useState(initialEdges)` 只有读，没有 `setEdges`
- 没有 `onConnect`
- 没有 `addEdge`
- 没有 `onConnectStart / onConnectEnd`

直接后果：
- 现在根本没有新增 edge 的正式入口
- “选类型后自动建 edge”在现有结构中无处可挂

### 4.3 缺 creation menu state
当前只有：
- `pendingNodeType`

它表达的是：
- 顶部工具条先选一个节点类型，然后点空白创建

它不表达：
- source 是谁
- 从哪个 output handle 发起
- 菜单应该显示在哪里
- 节点应该落在哪里

### 4.4 缺 node/edge factory
当前 node 创建是内联对象拼接。

缺失：
- `createNode()`
- `createEdge()`
- 统一 id 规则
- 统一默认数据
- 统一位置求解

### 4.5 缺 collision / lane state
当前没有任何：
- 同源下游计数
- 布局稳定策略
- 轻量避让策略

结论：
- 当前 `titanX/app` 空到还不能谈“只改两三个函数就上输出创建”
- 最少需要先补：自定义节点壳、边 state、菜单 state、工厂函数

## 6. state 层放置判断

### 6.1 `pendingSource`
最适合放在 `FlowCanvas` 顶层 state，或其后续抽出的 `flow store`。

不适合放在：
- 单个 node 组件内部：因为 `onConnectEnd` 和菜单确认都要读它
- `Inspector`：不属于检查面板语义

推荐结构：
```ts
pendingSource = {
  sourceNodeId,
  sourceHandleId,
  mode: 'from-output-click' | 'from-output-drag',
}
```

### 6.2 `creationMenu`
最适合放在 `FlowCanvas` 顶层 state。

原因：
- ReactFlow 事件和 overlay 渲染都在这一层最方便汇合
- 菜单开关、可选类型、锚点位置都不该放进 node data

### 6.3 `anchorClient`
应归属 `creationMenu`，不是单独飘一个 state。

原因：
- 它只服务菜单定位
- 生命周期和菜单一致

### 6.4 `anchorFlow`
也应归属 `creationMenu`。

原因：
- 它只服务这一次创建动作
- 不应该写回节点本身
- 与 `anchorClient` 同时生成、同时清空最稳

### 6.5 `lane state`
建议不要和 `creationMenu` 混放。

更适合：
- `FlowCanvas` 内 `useRef(new Map())`
- 或未来抽到 `flow store`

因为它属于：
- 持续的布局记忆
- 不是一次性菜单状态

## 7. `screenToFlowPosition()` 该在哪些交互点调用

### 7.1 必须调用的点

#### A. 拖出输出点落空白区时
在 `onConnectEnd` 内调用。

原因：
- 这是最标准的“drop 点 -> flow 点”转换
- 要在菜单打开瞬间冻结 `anchorFlow`

#### B. 点击输出点直接弹菜单时
在输出点 click handler 内调用。

但这里不要直接拿 node.position 代替。

更稳做法：
- 读取点击时的 `clientX/clientY`
- 同步生成 `anchorClient + anchorFlow`

### 7.2 不建议调用的点

#### A. 菜单项点击确认时再调用
不建议。

原因：
- 菜单打开后用户可能平移/缩放画布
- 此时再算 flow position，会让节点落点漂移

正式结论：
- `screenToFlowPosition()` 应在“打开 creationMenu 的那一刻”调用，而不是在“确认创建”的那一刻调用

## 8. node 和 edge 分别由哪个函数负责

### 8.1 node 创建
当前没有正式函数，建议从 `handlePaneClick()` 中抽出。

推荐新增：
```ts
createCanvasNode({ type, label, kind, position, sourceMeta? })
```

放置位置建议：
- P0 可先放 `App.jsx` 内部
- P1 再抽到 `app/src/flow/nodeFactory.js`

### 8.2 edge 创建
当前完全不存在，必须新建。

推荐新增：
```ts
createCanvasEdge({ source, sourceHandle, target, targetHandle })
```

放置位置建议：
- P0 可先放 `App.jsx` 内部
- P1 再抽到 `app/src/flow/edgeFactory.js`

### 8.3 为什么 node / edge 不该继续直接在事件里内联拼对象
原因：
- pane click 创建
- output click 创建
- output drag 创建
三条链最终都要复用同一套创建规则。

如果继续内联写对象：
- 业务 ID 规则会散
- 默认 data 会散
- edge 字段会散
- 后面文档和代码会重新失配

## 9. 哪些文件能继续用，哪些需要新建

### 9.1 可以继续用
- `app/src/App.jsx`：继续做当前阶段的画布主容器
- `app/src/App.css`：继续承接全局画布和 overlay 样式
- `app/src/components/Inspector.jsx`：保留，不进主链
- `app/src/components/StatusPanel.jsx`：保留，不进主链

### 9.2 建议新建
至少建议新建下面这些壳：

#### A. `app/src/components/nodes/TitanNode.jsx`
用途：
- 自定义 titanX 节点壳
- 放输出点按钮/handle
- 放 `onOutputClick`

#### B. `app/src/components/CreationMenu.jsx`
用途：
- 纯 overlay 菜单
- 根据 `creationMenu` 状态渲染
- 不直接改 nodes/edges，只回调“选中了哪个类型”

#### C. `app/src/flow/nodeFactory.js`
用途：
- 统一 node 创建
- 后续承接业务 id / 默认 data / 默认 type

#### D. `app/src/flow/edgeFactory.js`
用途：
- 统一 edge 创建
- 后续承接 sourceHandle / targetHandle / edge type

#### E. `app/src/flow/creationPosition.js`
用途：
- `anchorFlow + baseOffset + lane + collisionNudge`
- 避免位置求解散在多个事件回调里

### 9.3 P0 是否必须马上上 store
不必须。

当前 app 很薄，P0 用 `FlowCanvas` 顶层 `useState/useRef` 足够。

但如果继续往下做：
- contextMenu
- creationMenu
- pendingSource
- history
- canvasMode

那很快就应该补 `flowStore`。

## 11. 回答本轮 7 个必答点的过程结论

### 11.1 菜单入口最适合挂在哪个文件/组件
- 真正的点击入口：新建 `app/src/components/nodes/TitanNode.jsx`
- 菜单渲染入口：`app/src/App.jsx` 内的 `FlowCanvas` overlay 区

### 11.2 `pendingSource / creationMenu / anchorClient / anchorFlow` 适合放哪层 state
- 都先放 `FlowCanvas` 顶层 state
- `anchorClient / anchorFlow` 归属 `creationMenu`
- `lane map` 用 `useRef` 单独放

### 11.3 `screenToFlowPosition()` 应在哪个交互点调用
- output click 打开菜单时
- `onConnectEnd` 落空白打开菜单时
- 不建议等到菜单确认时再调用

### 11.4 node 创建和 edge 创建分别由哪个函数负责
- node：新增 `createCanvasNode()`
- edge：新增 `createCanvasEdge()`
- P0 可先放 `App.jsx`，P1 再抽文件

### 11.5 哪些现有文件能继续用，哪些需要新建
- 继续用：`App.jsx`、`App.css`、`Inspector.jsx`、`StatusPanel.jsx`
- 新建：`TitanNode.jsx`、`CreationMenu.jsx`、`nodeFactory.js`、`edgeFactory.js`、`creationPosition.js`

### 11.6 最小改造顺序（P0 / P1）
- P0：补 edge state、菜单 state、custom node、菜单 overlay、node/edge 创建函数
- P1：补 lane、避让、位置模块化、再考虑 flow store

### 11.7 如果 `titanX/app` 太空，缺失到什么程度，建议从哪个壳开始补
当前缺失程度：
- 缺自定义节点壳
- 缺输出点业务入口
- 缺 edge 可写链路
- 缺 creation menu state
- 缺 node/edge factory

建议从这两个壳先补：
1. `TitanNode.jsx`
2. `CreationMenu.jsx`

因为没有这两个壳，输出点建节点这条产品语义在代码里根本无处安放。
