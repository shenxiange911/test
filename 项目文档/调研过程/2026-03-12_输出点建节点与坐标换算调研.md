# titanX 调研过程｜输出点建节点与坐标换算

记录时间：2026-03-12
任务范围：只验证 titanX 的“输出点附近创建下游节点”交互，不泛谈 React Flow 基础。

---

## 1. 本轮核对范围

本轮只围绕以下问题做定向验证：
- 点击输出点或拖出输出点后弹菜单
- 菜单显示在鼠标附近
- 选择节点类型后自动创建 node + edge
- 缩放和平移后的画布坐标换算
- 新节点默认偏移策略
- 避免新节点压住旧节点
- 多次创建时的布局稳定性

## 2. 已读 titanX 正式文档得到的前置约束

### 2.1 来自 `01_总开发主文档.md`
已明确存在产品锚点：
- `[ANCHOR-UX-001]`：点击输出点/输出按钮时，弹出新建节点菜单；新节点必须生成在鼠标附近；新建后自动建 edge。
- 当前 titanX 是 Manual-first Baseline，画布层要优先服务真实手动闭环，而不是抽象成通用工作流平台。
- 节点创建必须符合业务语义，不是随便往空白处掉一个 node 就结束。

### 2.2 来自 `11_画布交互与ReactFlow实现逻辑.md`
已明确提出：
- 应有统一 `creationMenu` 与 `pendingConnection`
- 输出拖拽到空白区域后，不直接建边，而是先弹菜单
- 菜单确认后执行 `createNode()` 与 `createEdge()`
- 双击空白与输出创建都应走统一 node factory

但当前文档还缺 3 个关键细节：
- 坐标到底记“屏幕坐标”还是“flow 坐标”
- 输出点附近的菜单挂载位置该跟 pane 走还是跟 viewport 走
- 连续多次创建时，新节点偏移和避让如何稳定

### 2.3 来自 `14_统一字段Schema总表_2026-03-11.md`
- `BaseNodeShell.position` 已有统一位置字段，可承接节点绝对落点
- 文档没有单独给“创建菜单状态”或“pendingConnection”定义 schema，说明这层应留在 flow/canvas store，而不是塞进 node data

### 2.4 来自模型映射文档 `15/17/18/19`
- 这些文档没有改变画布创建规则，但进一步说明 titanX 节点族不是纯 UI 占位，而是后面要接真实 provider family
- 所以“输出口创建下游”不能只做演示型拖拽，必须保留 source node / source handle / target node type 的业务语义

## 4. React Flow 官方能力核对结果

### 4.1 `screenToFlowPosition()` 能直接解决缩放/平移后的坐标换算
官方文档与示例均说明：
- `screenToFlowPosition(clientPosition)` 用于把屏幕像素位置转换成 flow 内坐标
- 适合 sidebar drag-and-drop 或鼠标落点建节点

这说明 titanX 不应自己手写：
- `(clientX - viewportX) / zoom`
- `(clientY - viewportY) / zoom`

更稳妥的做法是统一走 React Flow instance：
```ts
const flowPoint = screenToFlowPosition({ x: clientX, y: clientY })
```

验证结论：
- 缩放和平移后的坐标换算：可直接做
- 不需要额外包装数学公式
- 但前提是菜单确认创建节点时，仍保留“触发时的 client 坐标”或“菜单打开时的 flow 坐标”

### 4.2 `onConnectEnd` 可承接“拖出输出点后落空白处”的链路
React Flow 官方示例 `Add Node On Edge Drop` 说明：
- 用户从 handle 拖出连接线
- 若 drop 到 pane 且不是有效连接，则 `onConnectEnd` 触发
- 示例里直接在 drop 点创建 node，再建 edge

这证明 titanX 的输出创建主语义成立：
- 从输出 handle 拖出
- drop 空白区
- 记录 source node/source handle
- 在 drop 点附近弹菜单
- 用户选类型后自动建 node + edge

但 titanX 与官方 demo 的差异是：
- 官方 demo 直接创建默认 node
- titanX 必须先弹业务菜单
- titanX 还要按节点族限制 allowed target types

所以 React Flow 官方能力够用，但 titanX 需要包装在自己的 `creationMenu + pendingConnection` 之上。

### 4.3 菜单位置不能直接用 flow 坐标渲染到普通 DOM
如果菜单是普通 HTML 浮层：
- DOM 定位吃的是屏幕/pane 像素坐标
- 不是 flow 坐标

因此菜单状态需要同时保留：
- `anchorClient`: 菜单视觉锚点，用于 HTML 浮层定位
- `anchorFlow`: 节点实际创建点，用于 node.position

这是当前 `11_画布交互与ReactFlow实现逻辑.md` 还没写清的一点。

## 5. 菜单挂载位置验证

### 5.1 不推荐把菜单挂进随 viewport 一起缩放的节点层
如果菜单直接挂进 viewport：
- 会跟着 zoom 缩放
- 菜单字号和点击热区会变形
- 产品感受像“菜单是节点的一部分”，不适合创建动作菜单

### 5.2 推荐挂在 React Flow 外层 shell 或 pane 同级 overlay
推荐：
- 菜单用普通 DOM portal/floating layer
- 定位基于 `clientX/clientY` 或相对 pane 的像素坐标
- 菜单本身不参与画布缩放

原因：
- 菜单必须稳定显示在“鼠标附近”
- 缩放后视觉尺寸不应变化
- 点菜单不应再触发画布拖拽/缩放

结论：
- 菜单挂载位置：需要包装后做
- 不是 React Flow 做不到，而是 titanX 应主动把菜单从 viewport 图层剥离出来

## 6. 事件流验证

### 6.1 点击输出点
如果产品允许“点击输出点就弹菜单”：
- 这不是 React Flow 默认 connect 事件链
- 需要在自定义 handle 或节点右侧输出按钮上自己绑定 click

推荐事件流：
1. 点击输出点
2. 读取当前鼠标或 handle 的屏幕位置
3. 计算默认落点 `anchorFlow`
4. 写入 `creationMenu = { mode: 'from-output-click', sourceNodeId, sourceHandleId, anchorClient, anchorFlow, allowedTypes }`
5. 打开菜单
6. 选择节点类型
7. `createNode()`
8. `createEdge()`

结论：
- 点击输出点弹菜单：可直接做
- 但属于 titanX 自己的自定义 handle 交互，不是拿 `onConnectEnd` 直接代替

### 6.2 拖出输出点后弹菜单
推荐事件流：
1. `onConnectStart` 记录 `sourceNodeId/sourceHandleId`
2. `onConnectEnd` 判断 `connectionState.isValid === false`
3. 若落点在 pane 空白处，则打开 creation menu
4. 菜单锚点取 drop 时鼠标位置
5. 选择类型后建 node + edge

结论：
- 拖出输出点后弹菜单：可直接做
- 这是和 React Flow 官方示例最贴近的一条链

## 7. 节点落点策略验证

### 7.1 最小可用落点
最简单的节点落点方式：
- 直接以 `anchorFlow` 作为新节点位置

问题：
- 很容易压住源节点
- 连续创建多个下游节点会叠在一起
- 菜单如果在源节点右边，node 左上角直接落点会出现“节点中心不在预期位置”

### 7.2 推荐默认偏移策略
推荐把“鼠标附近”解释成：
- 不要求节点左上角正好等于鼠标点
- 而是新节点整体出现在鼠标右下或右侧轻微偏移处

建议默认偏移：
```ts
baseOffset = { x: 220, y: -24 }
```
或更通用地按 source -> target 方向：
- 右侧输出创建：优先向右偏移
- 纵向空间不足时再向下微调

原因：
- titanX 当前节点卡片不是小圆点，而是中等尺寸卡片
- 如果不偏移，边和节点会挤在一起

结论：
- 新节点默认偏移策略：需要包装后做
- 不是 API 限制，而是产品层必须定规则

### 7.3 节点 origin 对落点感知有直接影响
官方示例里用了：
```ts
origin: [0.5, 0]
```
即节点位置以“顶部中心”为锚点。

这意味着 titanX 若继续沿用默认 `[0, 0]`：
- `position` 代表左上角
- 鼠标附近创建时需要自己补 UI 偏移

如果 titanX 想让“边从输出点拖出去后，新节点更像顺着线落下”：
- 可以考虑某些节点统一使用 `origin: [0, 0.5]` 或 `[0.5, 0]`
- 但这会影响现有节点布局和拖拽语义

结论：
- v1 不建议为了这个交互全局修改 nodeOrigin
- 更稳的是保留现有 nodeOrigin，单独在创建工厂里做位置偏移

## 8. 避免压住旧节点的验证

### 8.1 React Flow 本身不提供“自动避让布局”主语义
React Flow 提供的是：
- 节点位置控制
- 交互与视口转换
- 相交检测接口 `getIntersectingNodes()`

它不会自动替你决定：
- 连续创建的 sibling node 该落哪里
- 哪种偏移算更符合工作台语义

### 8.2 titanX 可利用 `getIntersectingNodes()` 做轻量避让
官方实例接口可拿到：
- `getIntersectingNodes(nodeOrRect, partially?)`

可用于：
1. 先算出候选位置
2. 生成一个候选矩形
3. 若与已有节点相交，则按固定步长继续偏移
4. 直到找到空位或达到最大尝试次数

推荐步长：
```ts
stagger = { x: 36, y: 96 }
maxTries = 6
```

原因：
- 右侧输出创建的直觉通常是“同列下移”优先
- 这样比左右乱跳更稳定

结论：
- 避免新节点压住旧节点：需要包装后做
- 可用轻量检测，不必一上来接 ELK/Dagre

## 9. 多次创建时的布局稳定性

### 9.1 不建议每次都按“离最近空位”贪心乱找
如果每次新建都：
- 从源点开始四周搜索空位
- 找到最近没碰撞的位置就塞进去

会导致：
- 连续多次创建的下游节点像散弹一样分布
- 用户难形成“这个输出口的下游列”视觉记忆

### 9.2 推荐按 source handle 维度维护 creation lane
更稳定的策略：
- 对每个 `sourceNodeId + sourceHandleId` 维护一个 lane index
- 第 1 次创建：`base + (0, 0)`
- 第 2 次创建：`base + (0, 96)`
- 第 3 次创建：`base + (0, 192)`
- 如果某个目标位冲突，再在当前行上做二级横向微调

这比“每次重新算最近空位”更符合 titanX 的工作台体验。

推荐：
```ts
laneKey = `${sourceNodeId}:${sourceHandleId ?? 'default'}`
base = anchorFlow + { x: 220, y: -24 }
rowOffset = laneIndex * 96
collisionNudge = { x: 36, y: 0 }
```

结论：
- 多次创建时的布局稳定性：需要包装后做
- 但这套规则非常适合 titanX v1，不重也不虚

## 10. edge 自动创建时机验证

不建议在菜单打开时就先建临时 edge。

更稳的时机：
- 用户点击菜单项确认节点类型后
- `createNode()` 成功拿到 target node id
- 再调用 `createEdge()`

原因：
- 用户可能关闭菜单或取消创建
- 先建 edge 会产生脏状态
- titanX 当前不是连线预览产品，重点是“选了目标类型才真正落子”

可选增强：
- 拖拽过程中仍显示 React Flow 默认 connection line
- 但实际 edge 只在菜单确认后写入 edges state

结论：
- edge 自动创建时机：可直接做
- 应放在菜单确认之后，不放在菜单弹出时

## 11. 是否适合放进 titanX v1

### 11.1 可直接做
- `screenToFlowPosition()` 做缩放/平移坐标换算
- `onConnectStart/onConnectEnd` 承接“拖出输出点后弹菜单”
- 菜单确认后自动创建 node + edge
- 点击输出点触发菜单（需自定义 handle/button 事件）

### 11.2 需要包装后做
- 菜单挂载到 viewport 外的 overlay 层
- 同时维护 `anchorClient + anchorFlow`
- 新节点默认偏移策略
- 轻量碰撞避让
- 基于 source handle 的稳定 lane 排布

### 11.3 不建议放进 v1
- 一上来接完整自动布局引擎（ELK / Dagre）处理每次输出创建
- 菜单跟随 viewport 缩放做成“画布内节点化浮层”
- 菜单打开就先写临时 edge 再回滚
- 为了这一个交互去全局重构 nodeOrigin

## 12. 推荐最小实现方案（过程稿结论）

### 12.1 状态结构
```ts
creationMenu = {
  open: boolean,
  mode: 'from-output-click' | 'from-output-drag',
  sourceNodeId: string,
  sourceHandleId?: string | null,
  anchorClient: { x: number; y: number },
  anchorFlow: { x: number; y: number },
  allowedTypes: string[],
}
```

### 12.2 事件流
- 点击输出点：直接打开菜单
- 拖出输出点到空白：`onConnectEnd` 打开菜单
- 选中类型：创建 target node -> 创建 edge -> 关闭菜单

### 12.3 位置策略
```ts
basePosition = {
  x: anchorFlow.x + 220,
  y: anchorFlow.y - 24,
}

nextPosition = {
  x: basePosition.x,
  y: basePosition.y + laneIndex * 96,
}
```

### 12.4 冲突处理
- 先按 lane 位置放
- 若与已有节点相交，则最多尝试 6 次
- 每次向右 `+36` 微移

### 12.5 为什么这套适合 titanX
- 保住“鼠标附近创建下游节点”的产品语义
- 不会因为缩放/平移导致落点错位
- 多次创建能形成稳定下游列
- 工程复杂度低，能进入 v1

---

## 13. 风险点 / 踩坑点

1. 只记 `clientX/clientY` 不记 `anchorFlow`：若菜单打开后用户又缩放或平移，再创建时节点落点会漂。
2. 只记 `anchorFlow` 不记 `anchorClient`：菜单无法稳定显示在鼠标附近，只能粗暴定位到画布内部。
3. 菜单挂到 viewport 内：缩放时菜单会一起缩放，交互感很差。
4. 直接把 drop 点作为 node 左上角：新节点容易压住源节点或盖住边。
5. 没有 lane 计数：同一输出口多次创建会叠在一起。
6. 碰撞策略过于激进：若每次都全局搜索最近空位，会导致布局抖动，用户失去空间记忆。
7. 菜单弹出时就先写 edge：取消创建时需要回滚，状态会变脏。

---

## 14. 本轮最终判断

对于 titanX 的“输出点附近创建下游节点”语义：
- React Flow 原生能力已经足够支撑 v1
- 关键不在“能不能做”，而在 titanX 需要把官方 connect/drop 能力包装成自己的业务事件流
- v1 最值得冻结的不是复杂自动布局，而是：
  - 坐标双锚点
  - 菜单外层挂载
  - lane 化稳定落点
  - 菜单确认后再建 edge
