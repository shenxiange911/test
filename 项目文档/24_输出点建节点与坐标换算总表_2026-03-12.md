# titanX 输出点建节点与坐标换算总表

记录时间：2026-03-12
定位：针对 titanX“鼠标附近创建下游节点”语义的正式整合稿，不展开 React Flow 基础教程。

---

## 1. 结论先行

### 1.1 可直接做
- 点击输出点或从输出点拖出到空白区后弹菜单
- 菜单选定节点类型后自动创建 `node + edge`
- 用 `screenToFlowPosition()` 处理缩放/平移后的画布坐标换算
- 用 `onConnectStart/onConnectEnd` 承接“拖出输出点后创建下游节点”

### 1.2 需要包装后做
- 菜单显示在鼠标附近，但菜单本体不跟随 viewport 缩放
- 同时维护菜单视觉锚点与节点创建锚点
- 新节点默认偏移与轻量避让
- 同一输出口多次创建时的稳定排布

### 1.3 不建议放进 v1
- 引入完整自动布局引擎来处理每次输出创建
- 把菜单做成画布内缩放浮层
- 菜单一打开就先写临时 edge
- 为了这个交互全局改 `nodeOrigin`

### 1.4 推荐最小实现方案
- `creationMenu + pendingSource` 两段状态
- 菜单挂在 React Flow 外层 overlay
- 节点位置由 `anchorFlow + 默认偏移 + lane 偏移 + 轻量碰撞修正` 得出
- 选中菜单项后再统一 `createNode()` 与 `createEdge()`

### 1.5 风险点 / 踩坑点
- 只存一种坐标，菜单和节点会有一个必然错位
- 不做偏移和避让，新节点会压住源节点或叠成一团
- 没有 lane 规则，多次创建布局不稳定
- 菜单挂进 viewport 会随缩放变形

## 3. 事件流

### 3.1 点击输出点创建
推荐事件流：
1. 用户点击节点右侧输出点/输出按钮
2. 系统读取当前输出点对应的 `sourceNodeId/sourceHandleId`
3. 系统记录菜单锚点：
   - `anchorClient`：菜单显示位置
   - `anchorFlow`：节点创建基准位置
4. 系统根据当前节点类型计算 `allowedTypes`
5. 打开创建菜单
6. 用户选择节点类型
7. 调用 `createNode()` 创建 target node
8. 调用 `createEdge()` 自动连接 source -> target
9. 关闭菜单并清空 pending 状态

### 3.2 拖出输出点创建
推荐事件流：
1. `onConnectStart` 记录 source 信息
2. 用户把连接线拖到空白 pane
3. `onConnectEnd` 发现 `connectionState.isValid === false`
4. 系统在 drop 鼠标附近打开创建菜单
5. 用户选择节点类型
6. 调用 `createNode()`
7. 调用 `createEdge()`
8. 关闭菜单并清空 pending 状态

### 3.3 为什么 edge 不应提前创建
不建议在菜单弹出时先写 edge。

原因：
- 用户可能取消创建
- 先写 edge 会引入回滚问题
- titanX 当前主语义是“确认节点类型后才真正落子”

正式规则：
- 连接线预览可以由 React Flow 默认 connect line 承担
- 真实 edge 只在菜单确认后落入 `edges state`

## 4. 坐标换算方法

### 4.1 必须区分两套坐标
为了同时满足：
- 菜单显示在鼠标附近
- 节点在缩放/平移后的正确画布位置创建

创建链路必须同时保存：

```ts
interface CreationAnchor {
  anchorClient: { x: number; y: number }   // HTML 菜单定位
  anchorFlow: { x: number; y: number }     // node.position 计算基准
}
```

### 4.2 `anchorClient` 的用途
- 直接用于 overlay 菜单的 DOM 定位
- 表达“菜单显示在鼠标附近”的产品语义
- 不参与 node.position 存储

### 4.3 `anchorFlow` 的用途
- 作为新节点的创建基准点
- 通过 React Flow instance 的 `screenToFlowPosition()` 获得
- 自动吸收 viewport 的平移与缩放影响

推荐做法：
```ts
const anchorFlow = screenToFlowPosition({ x: clientX, y: clientY })
```

### 4.4 为什么不建议自己手写变换公式
虽然理论上可写：
```ts
x = (clientX - viewportX) / zoom
y = (clientY - viewportY) / zoom
```

但 titanX 没必要自己维护这层。

原因：
- React Flow 已有官方实例方法
- 自写公式更容易漏掉 pane 边界、DOM offset、后续升级差异
- titanX 当前目标是稳定产品链路，不是重复造 viewport 数学轮子

正式规则：
- v1 坐标换算统一只认 `screenToFlowPosition()`

## 5. 推荐菜单挂载位置

### 5.1 推荐挂在 React Flow 外层 overlay
推荐结构：
- React Flow 负责 canvas / nodes / edges
- 创建菜单挂在 canvas shell 的 overlay 层，或 portal 到同级浮层容器

示意：
```tsx
<div className="canvas-shell">
  <ReactFlow ... />
  {creationMenu.open && <CreationMenuOverlay ... />}
</div>
```

### 5.2 不推荐挂进 viewport 内部
不推荐把菜单渲染到 `.react-flow__viewport` 内。

原因：
- viewport 会跟随 zoom 缩放
- 菜单字号和热区会变形
- 菜单更像“节点的一部分”，不符合动作面板语义

### 5.3 正式结论
- 菜单视觉挂载位置：overlay 层
- 菜单定位依据：`anchorClient`
- 节点落点依据：`anchorFlow`

## 6. 推荐节点落点策略

### 6.1 目标语义
“鼠标附近创建下游节点”不应被误解成：
- 节点左上角严格贴着鼠标点

更准确的产品解释应是：
- 节点整体出现在鼠标附近、且符合从 source 向 downstream 延伸的空间逻辑

### 6.2 推荐基础偏移
对于右侧输出口，推荐默认：
```ts
baseOffset = { x: 220, y: -24 }
```

得到基础落点：
```ts
basePosition = {
  x: anchorFlow.x + 220,
  y: anchorFlow.y - 24,
}
```

原因：
- 保持源节点与下游节点之间有足够视距
- 边不会贴着节点边缘挤在一起
- 更符合 titanX 当前卡片型节点尺寸

### 6.3 推荐稳定 lane 策略
同一输出口多次创建时，推荐按 lane 纵向递增：
```ts
laneKey = `${sourceNodeId}:${sourceHandleId ?? 'default'}`
rowGap = 96
```

第 `laneIndex` 次创建：
```ts
lanePosition = {
  x: basePosition.x,
  y: basePosition.y + laneIndex * 96,
}
```

好处：
- 同一 source 的下游会形成稳定列
- 用户容易建立空间记忆
- 后续连线也更整齐

### 6.4 轻量碰撞避让
在 lane 位置基础上做最多 6 次小范围修正：
```ts
collisionNudge = { x: 36, y: 0 }
maxTries = 6
```

规则：
1. 先尝试 lane 目标位
2. 如果与已有节点相交，则向右微移 `+36`
3. 最多尝试 6 次
4. 若仍冲突，则接受最后一个候选位并交给用户手动微调

这比重型自动布局更适合 v1。

## 7. edge 自动创建时机

正式规则：
- `edge` 只在用户选定菜单项之后创建
- 不能在菜单打开时预创建

推荐时序：
```ts
const targetNode = createNode(...)
createEdge({ source: sourceNodeId, sourceHandle: sourceHandleId, target: targetNode.id })
```

原因：
- 取消菜单无需回滚脏 edge
- 逻辑清晰
- 更适合后续统一 undo/redo

## 8. 边界情况

### 8.1 菜单打开后用户又缩放/平移
如果只保存 `client` 坐标，节点落点会漂。

正式规则：
- 菜单打开瞬间就同时冻结 `anchorClient + anchorFlow`
- 后续即便用户缩放，也不重算本次创建落点

### 8.2 点击输出点但用户取消菜单
正式规则：
- 不创建 node
- 不创建 edge
- 清空 `creationMenu/pendingSource`

### 8.3 拖出输出点但最后落到已有目标 handle
正式规则：
- 如果 `connectionState.isValid === true`，走普通连线链路
- 只有落空白 pane 且无有效连接时，才弹“创建下游节点”菜单

### 8.4 新节点初始位超出可见区或 nodeExtent
如果 titanX 后续启用 `nodeExtent`：
- 创建工厂在最终写入前需做边界裁剪
- 仍保持 lane 规则优先，不做全局乱跳

### 8.5 同一 source 节点有多个输出 handle
正式规则：
- lane 维度应至少到 `sourceNodeId + sourceHandleId`
- 不同 handle 不能共用同一个连续计数，否则会互相打乱布局

### 8.6 节点尺寸尚未完全确定
如果节点宽高后续由内容决定：
- v1 可先按节点族预设尺寸估算碰撞盒
- 不必等真实渲染尺寸后再回填

## 9. 最小可实现方案

### 9.1 状态结构
```ts
interface CreationMenuState {
  open: boolean
  mode: 'from-output-click' | 'from-output-drag'
  sourceNodeId: string
  sourceHandleId?: string | null
  anchorClient: { x: number; y: number }
  anchorFlow: { x: number; y: number }
  allowedTypes: Array<'text' | 'image' | 'video' | 'audio' | 'analysis' | 'storyboard' | 'result'>
}
```

### 9.2 创建命令
```ts
openCreationMenuFromOutput(...)
closeCreationMenu()
createNodeFromOutputMenu(targetType)
```

### 9.3 位置求解
```ts
function resolveNewNodePosition({
  anchorFlow,
  sourceNodeId,
  sourceHandleId,
}) {
  const laneKey = `${sourceNodeId}:${sourceHandleId ?? 'default'}`
  const laneIndex = getAndBumpLaneIndex(laneKey)

  const base = {
    x: anchorFlow.x + 220,
    y: anchorFlow.y - 24,
  }

  let candidate = {
    x: base.x,
    y: base.y + laneIndex * 96,
  }

  return nudgeUntilFree(candidate)
}
```

### 9.4 完整创建链
```ts
function createNodeFromOutputMenu(targetType) {
  const position = resolveNewNodePosition(...)
  const targetNode = createNode({ type: targetType, position })
  createEdge({
    source: creationMenu.sourceNodeId,
    sourceHandle: creationMenu.sourceHandleId,
    target: targetNode.id,
  })
  closeCreationMenu()
}
```

## 10. 对 titanX v1 的正式建议

### 10.1 可直接做
- `screenToFlowPosition()` 做坐标换算
- 点击输出点弹菜单
- 拖出输出点落空白后弹菜单
- 菜单选定后自动创建 `node + edge`

### 10.2 需要包装后做
- `anchorClient + anchorFlow` 双锚点
- overlay 菜单层
- lane 稳定排布
- 轻量碰撞避让

### 10.3 不建议放进 v1
- ELK/Dagre 级别自动布局
- 菜单节点化 / 跟 viewport 缩放
- 预写 edge 再回滚
- 为单一交互重构全局 `nodeOrigin`

---

## 11. 最终冻结结论

titanX 的“输出点附近创建下游节点”在 v1 里应该这样理解：

1. 不是普通 React Flow demo，而是业务化的 downstream creation action。
2. 菜单显示位置与节点创建位置必须分成两套坐标。
3. 节点落点不能只追求“鼠标精确命中”，更要保证 downstream 布局稳定。
4. 最合适的 v1 方案不是重型自动布局，而是：
   - 双锚点
   - overlay 菜单
   - 固定基础偏移
   - lane 递增
   - 轻量碰撞修正
5. 这套方案足够进入 titanX v1，且不会把画布层做虚或做重。
